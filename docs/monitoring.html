Monitoring

Monitoring und Alerting

AdminHelper Monitoring uberwacht Server und Dienste mit Pull- und Push-basierten Checks, speichert Metriken in VictoriaMetrics und alarmiert per Webhook oder E-Mail.

Check-Typen Alerting

1) Architektur

Das Monitoring-System besteht aus vier Komponenten, die zusammenarbeiten, um Server und Dienste zu uberwachen:

Monitoring-Service

FastAPI-Anwendung auf Port 8080. Enthalt die Check-Engine, den Scheduler fur periodische Prufungen und das Alerting-System. Empfangt Push-Daten vom Agent und fuhrt Pull-Checks selbststandig aus.

VictoriaMetrics

Zeitreihen-Datenbank auf Port 8428. Speichert alle Metriken mit einer Standard-Retention von 90 Tagen. Kompatibel mit PromQL-Abfragen.

AdminHelper Agent

Push-basierte Metrikerfassung auf den uberwachten Servern. Sammelt CPU, RAM, Disks, Services und Temperaturen und sendet diese per HTTPS an den Monitoring-Service.

AdminHelper Server

Stellt einen Proxy bereit, uber den die Web-UI auf die Monitoring-API zugreifen kann. Der Monitoring-Service ist nicht direkt von aussen erreichbar.

Datenfluss: Agent → Monitoring-Service → VictoriaMetrics + Alerting. Pull-Checks werden direkt vom Monitoring-Service ausgefuhrt.

2) Check-Typen

AdminHelper unterstutzt 10 verschiedene Check-Typen, aufgeteilt in Pull-basierte (vom Monitoring-Service ausgefuhrt) und Push-basierte (vom Agent gemeldet) Checks.

Pull-basierte Checks

ping — ICMP Ping

Pruft die Erreichbarkeit eines Hosts per ICMP Ping.

Config: target (Hostname/IP), timeout (Sekunden, Standard: 5)

Metrik: monitor_ping_rtt_ms

tcp — TCP Port Check

Pruft ob ein TCP-Port erreichbar ist und misst die Verbindungszeit.

Config: host, port, timeout (Sekunden, Standard: 5)

Metrik: monitor_tcp_connect_ms

http — HTTP/HTTPS Check

Fuhrt einen HTTP-Request aus und pruft Status-Code und Antwortzeit.

Config: url, method (GET/POST), timeout, status_code (erwarteter Code)

Metriken: monitor_http_status_code, monitor_http_response_ms

agent_ping — Agent-Erreichbarkeit

Pruft ob der Agent in den letzten 5 Minuten Daten gesendet hat. Benotigt keine besonderen Konfigurationsparameter.

Config: keine besonderen Parameter

Push-basierte Checks

agent_resources — CPU, Memory, Disk

Uberwacht Systemressourcen mit konfigurierbaren Schwellwerten. Per-Disk und Per-Sensor Schwellwerte sind moglich.

Config: cpu_warning (%), cpu_critical (%), memory_warning (%), memory_critical (%)

Metriken: monitor_cpu_percent, monitor_memory_percent, monitor_disk_percent

service_process — Dienst-Uberwachung

Pruft den Status konfigurierter Dienste. Unterstutzt systemd-Units (Linux) und Windows-Services.

Config: services (Array von Dienstnamen)

Status: running, stopped, failed

docker_health — Docker Container

Pruft den Health-Status laufender Docker-Container. Container werden automatisch vom Agent erkannt.

Config: automatische Erkennung

proxmox_backup — Proxmox Backup Server

Pruft den letzten Backup-Zeitpunkt auf einem Proxmox Backup Server.

Config: max_backup_age_hours, exclude_vmids, exclude_stopped

zfs_health — ZFS Pool Status

Pruft ZFS-Pool-Kapazitat und Health-Status.

Config: Pool-Namen, warning_percent, critical_percent

smart_health — Festplatten S.M.A.R.T.

Uberwacht S.M.A.R.T.-Werte aller Festplatten (HDD, SATA-SSD, NVMe). Automatische Erkennung durch den Agent via smartctl. Geraete werden anhand rotation_rate + Protokoll in HDD / SATA-SSD / NVMe klassifiziert; Temperatur-Grenzwerte sind pro Kind konfigurierbar.

Prueft: SMART-Status, Reallocated / Pending / Uncorrectable Sectors, UDMA CRC Errors, Temperatur (pro Kind), NVMe Available Spare & Percentage Used, NVMe critical_warning Bit-Feld (Spare / Temp / Reliability / Read-Only / Volatile Memory Backup / PMR) sowie den smartctl-Exit-Code (Prefail-Bit 0x10 = kritisch, 0x20/0x40/0x80 = Warnung).

Hinweis: Hardware-RAID-Controller werden nicht unterstuetzt — smartctl -d <controller> muesste separat gepflegt werden.

3) Check-Konfiguration

Jeder Check kann individuell konfiguriert werden. Die wichtigsten Parameter sind:

Parameter Beschreibung
Intervall Vordefiniert: 1m, 5m, 15m, 30m, 1h, 6h, 12h, 24h oder Cron-Ausdrucke (5 Felder)
Severity warning oder critical
Consecutive Fails Anzahl aufeinanderfolgender Fehler bevor der Status wechselt (z.B. 3 = erst nach 3 Fehlern wird alarmiert)
Enabled Checks konnen per Flag aktiviert oder deaktiviert werden

Beispiel: Check per API erstellen

POST /api/monitoring/checks
Content-Type: application/json

{
  "name": "Webserver erreichbar",
  "check_type": "http",
  "server_id": "srv-web-prod-01",
  "interval": "5m",
  "severity": "critical",
  "consecutive_fails": 3,
  "enabled": true,
  "config": {
    "url": "https://example.com/health",
    "method": "GET",
    "timeout": 10,
    "status_code": 200
  }
}
Tipp: Mit consecutive_fails lassen sich kurzzeitige Ausfalle (Flapping) filtern. Ein Wert von 3 bei einem 5-Minuten-Intervall bedeutet, dass erst nach 15 Minuten anhaltendem Fehler alarmiert wird.

4) Alerting

Das Alerting-System benachrichtigt bei Statusanderungen uber konfigurierbare Kanale.

Alert-Regeln

Parameter Beschreibung
match_severity Nur bei bestimmtem Severity alarmieren (warning, critical)
match_server_id Nur fur bestimmten Server (optional, leer = alle Server)
cooldown_minutes Mindestabstand zwischen Alarmen (z.B. 30 = max. alle 30 Minuten)
Recovery-Alerts Automatisch wenn Status zu "ok" zuruckkehrt (ignoriert Cooldown)

Channel: Webhook

Sendet einen JSON-Payload an eine konfigurierbare URL (Timeout: 10 Sekunden). 

// Konfiguration
{
  "url": "https://hooks.slack.com/services/T00/B00/xxxx"
}

// Payload (wird automatisch gesendet)
{
  "alert_rule": "critical-alerts",
  "check_name": "Webserver erreichbar",
  "server_id": "srv-web-prod-01",
  "severity": "critical",
  "old_status": "ok",
  "new_status": "critical",
  "is_recovery": false,
  "subject": "[AdminHelper Monitor] CRITICAL: Webserver erreichbar",
  "text": "Check 'Webserver erreichbar' auf srv-web-prod-01 ist CRITICAL."
}

Channel: E-Mail (SMTP)

Sendet E-Mail-Benachrichtigungen per SMTP. Port 587 aktiviert STARTTLS automatisch. 

// Alert-Channel Konfiguration
{
  "recipients": ["admin@example.com", "ops@example.com"],
  "smtp_host": "smtp.example.com",
  "smtp_port": 587,
  "smtp_user": "alerts@example.com",
  "smtp_password": "geheim"
}
Hinweis: SMTP-Einstellungen konnen alternativ global uber Umgebungsvariablen gesetzt werden. Die Channel-Konfiguration uberschreibt die globalen Werte. Betreff-Format: [AdminHelper Monitor] CRITICAL: Checkname

5) Umgebungsvariablen

Der Monitoring-Service wird uber folgende Umgebungsvariablen konfiguriert:

Variable Standard Beschreibung
VICTORIA_METRICS_URL http://victoria:8428 VictoriaMetrics API URL
MONITOR_API_KEY (leer) Interner API-Key (geteilt mit Server)
DATA_DIR /app/data Verzeichnis fur SQLite-Datenbank
SMTP_HOST (leer) SMTP-Server fur E-Mail-Alerts
SMTP_PORT 587 SMTP-Port (587 = STARTTLS)
SMTP_USER (leer) SMTP-Benutzername
SMTP_PASSWORD (leer) SMTP-Passwort
SMTP_FROM adminhelper@localhost Absender-Adresse
TZ Europe/Berlin Zeitzone

6) Templates

Monitoring-Templates sind wiederverwendbare Sammlungen von Check- und Alert-Definitionen, die auf mehrere Server angewendet werden konnen.

Template-Inhalt

Ein Template enthalt zwei Arrays: check_definitions (Check-Vorlagen) und alert_definitions (Alert-Regel-Vorlagen).

Zuweisung und Sync

Bei Zuweisung an einen Server werden Checks und Alert-Regeln automatisch erstellt. Anderungen am Template konnen auf zugewiesene Server synchronisiert werden. 

// Beispiel: Template-Struktur
{
  "name": "Standard Linux Server",
  "check_definitions": [
    {
      "check_type": "agent_resources",
      "interval": "5m",
      "severity": "critical",
      "config": {
        "cpu_warning": 80,
        "cpu_critical": 95,
        "memory_warning": 85,
        "memory_critical": 95
      }
    },
    {
      "check_type": "agent_ping",
      "interval": "5m",
      "severity": "critical",
      "consecutive_fails": 3
    }
  ],
  "alert_definitions": [
    {
      "channel_type": "webhook",
      "match_severity": "critical",
      "cooldown_minutes": 30,
      "config": {
        "url": "https://hooks.slack.com/services/..."
      }
    }
  ]
}

7) Agent-Keys

Jeder uberwachte Server benotigt einen eigenen Agent-API-Key zur Authentifizierung gegenuber dem Monitoring-Service.

Eigenschaft Details
Generierung Uber Web-UI oder POST /admin/agent-keys
Anzeige Key wird bei Generierung einmalig im Klartext angezeigt
Speicherung In der Datenbank als SHA-256 Hash gespeichert
Verwendung Agent sendet Key im X-API-Key Header bei jedem Push
Wichtig: Der API-Key wird nur einmal im Klartext angezeigt. Notieren Sie ihn sofort oder konfigurieren Sie den Agent direkt nach der Generierung. Ein verlorener Key kann nicht wiederhergestellt werden — es muss ein neuer generiert werden.

8) VictoriaMetrics

VictoriaMetrics dient als Zeitreihen-Datenbank fur alle Monitoring-Metriken. Daten werden uber das InfluxDB Line Protocol geschrieben.

Eigenschaft Wert
Protokoll InfluxDB Line Protocol
Retention 90 Tage (Standard)
Abfrage-Sprache PromQL

Fur jeden Check werden mindestens zwei Metriken geschrieben:

  • monitor_check_status — 0 = ok, 1 = warning, 2 = critical, 3 = unknown
  • monitor_check_duration_ms — Ausfuhrungsdauer des Checks in Millisekunden

PromQL-Abfragen sind uber den Endpunkt /api/monitoring/checks/{id}/metrics moglich. 

# Beispiel PromQL: Durchschnittliche Check-Dauer der letzten Stunde
avg_over_time(monitor_check_duration_ms{check_id="42"}[1h])

# Alle kritischen Checks
monitor_check_status == 2

9) Docker Compose Setup

Die Monitoring-relevanten Teile der docker-compose.yml: 

services:
  monitoring:
    image: adminhelper-monitoring:latest
    ports:
      - "8480:8080"
    environment:
      - VICTORIA_METRICS_URL=http://victoria:8428
      - MONITOR_API_KEY=${MONITOR_API_KEY}
      - DATA_DIR=/app/data
      - SMTP_HOST=${SMTP_HOST:-}
      - SMTP_PORT=${SMTP_PORT:-587}
      - SMTP_USER=${SMTP_USER:-}
      - SMTP_PASSWORD=${SMTP_PASSWORD:-}
      - SMTP_FROM=${SMTP_FROM:-adminhelper@localhost}
      - TZ=Europe/Berlin
    volumes:
      - monitoring-data:/app/data
    depends_on:
      - victoria
    restart: unless-stopped

  victoria:
    image: victoriametrics/victoria-metrics:v1.101.0
    command:
      - "-retentionPeriod=90d"
      - "-storageDataPath=/victoria-data"
    volumes:
      - victoria-data:/victoria-data
    restart: unless-stopped

volumes:
  monitoring-data:
  victoria-data:
Hinweis: Der Monitoring-Service ist intern auf Port 8080 erreichbar und wird extern auf Port 8480 gemappt. Der AdminHelper-Server leitet Anfragen als Proxy an den Monitoring-Service weiter.
