Operations Guide

Installation, Betrieb und Sicherheit

Diese Seite ist für Systemadministration und Betriebsteams gedacht. Sie enthält den vollständigen Ablauf von Erstinstallation bis Wartung, inklusive Sicherheits-Härtung und Runbooks.

Installation starten Hardening-Checkliste

1) Deployment-Optionen

Empfohlen: Docker Compose

Schneller Produktivstart mit TLS-Unterstützung, persistenten Daten und Image-Pull aus der Registry.

Alternative: lokaler Python-Start

Für interne Tests/Entwicklung ohne Container. TLS und Port-Mapping müssen dann separat geregelt werden.

Docker Erstinstallation

cp .env.example .env
# Pflicht: SECRET_KEY, ADMIN_PASSWORD, ggf. SERVER_IMAGE setzen

docker compose pull
docker compose up -d

Mit der Standard-Compose-Datei ist der Server unter https://localhost erreichbar (Host-Port 443 auf Container-Port 8443).

Wichtig: Falls kein Zertifikat in ./certs liegt, erzeugt der Container beim Start automatisch ein selbstsigniertes TLS-Zertifikat.

2) Konfiguration (.env / Environment)

Variable Beispiel Bedeutung
SERVER_IMAGE docker.nevondo.com/.../server:latest Container-Image für den Server-Dienst.
DOMAIN localhost CN/SAN beim autogenerierten Zertifikat.
SECRET_KEY openssl rand -hex 32 Signierschlüssel für JWT.
ADMIN_PASSWORD starkes Initialpasswort Wird für den ersten Admin-Account verwendet.
ALLOWED_IPS 192.168.1.0/24,10.0.0.5 Whitelist für Client-IP/CIDR, leer = kein Filter.
TRUST_PROXY_HEADERS false Wenn true, werden Forwarded-Header global vertraut.
TRUSTED_PROXIES 172.17.0.1,10.0.0.0/8 Empfohlene sichere Variante hinter Reverse Proxy.
MONITOR_API_KEY openssl rand -hex 32 Shared Secret zwischen Server und Monitoring-Service.
MONITOR_AGENT_API_KEYS key1,key2 Kommagetrennte API-Keys für Agent-Authentifizierung.
MONITOR_AGENT_PORT 8480 Externer Port für den Monitoring-Agent-Endpoint.
FRP_BIND_PORT 7000 FRP-Protokoll-Port (Client-Verbindungen).
FRP_HTTPS_PORT 7443 FRP HTTPS-vhost-Port für Web-Tunnel.

Persistente Daten

  • ./data/: SQLite DB und connections.json
  • ./certs/: TLS Zertifikat/Key

3) Ersteinrichtung im Web-UI

  1. Anmeldung mit admin und ADMIN_PASSWORD.
  2. Mindestens einen weiteren Admin-Benutzer anlegen.
  3. Read-only API-Key für Client-Sync erzeugen.
  4. Optional: Nutzer ohne Admin-Rechte für reine Lesesicht anlegen.
Hinweis: API-Keys werden nur bei Erstellung im Klartext angezeigt. Danach ist nur der Hash in der DB vorhanden.

4) Laufender Betrieb

Verbindungen verwalten

Create/Update/Delete über die Seite Verbindungen. Import/Export steht nur Admins zur Verfügung.

Nutzer und Rollen

Nutzerverwaltung über Seite Benutzer. Eigenen Account kann man nicht löschen (Absicherung im Backend).

API-Keys

read für Sync/Extension, read_write nur für trusted Integrationen.

Hooks

Drei Typen: Webhook (URL-Trigger), Event-Hook (intern bei Verbindungs-/User-Änderungen), Scheduled Hook (Zeitplan, z. B. jede Stunde).

5) Hooks im Betrieb

Hooks sind Python-Scripts, die auf drei verschiedene Arten ausgelöst werden können. Alle Hook-Typen werden im Admin-UI unter Hooks verwaltet.

Webhook

Per HTTP-POST mit Token auslösen.
POST /api/hooks/trigger/<token>
Kontext im Script: payload, headers, params

Event-Hook

Automatisch bei Server-Ereignissen ausgelöst (keine externe URL nötig).
Kontext: event_type, event_data

Scheduled Hook

Nach Zeitplan: 5m, 15m, 30m, 1h, 6h, 12h, 24h oder Cron-Ausdruck.
Kontext: triggered_at, last_run

Verfügbare Events (Event-Hook)

EventWann
connection.createdVerbindung angelegt
connection.updatedVerbindung geändert
connection.deletedVerbindung gelöscht
connections.importedBulk-Import
user.createdBenutzer angelegt
user.deletedBenutzer gelöscht
server.startupServer gestartet

Webhook-Aufruf-Beispiel

Referenz-Scripts: docs/examples/webhook_add_connection.py, docs/examples/webhook_http_import.py 

curl -X POST "https://<server>/api/hooks/trigger/<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "connection": {
      "name": "Prod Web",
      "kind": "web",
      "url": "https://prod.example.com",
      "tags": ["prod", "web"]
    }
  }'

Event-Hook Script-Beispiel

# Wird z. B. bei connection.created oder connection.updated ausgelöst
import requests

hook_url = "https://chat.example.com/webhooks/notify"
msg = f"[SRM] {event_type}: {event_data.get('name', '?')} ({event_data.get('kind', '?')})"
requests.post(hook_url, json={"text": msg}, timeout=5)
result["notified"] = True

Scheduled Hook Script-Beispiel

# Läuft z. B. täglich (24h) – bereinigt Duplikate
connections = load_connections()
seen, unique = set(), []
for c in connections:
    key = (c.get("kind"), c.get("host"), c.get("port"))
    if key not in seen:
        seen.add(key)
        unique.append(c)
removed = len(connections) - len(unique)
if removed:
    save_connections(unique)
result["removed_duplicates"] = removed
log(f"Bereinigt am {triggered_at}: {removed} Duplikat(e)")

Empfohlener Betriebsablauf

  1. Hook im Admin-UI anlegen (Hooks → + Hook), Typ wählen.
  2. Script hinterlegen und mit Ausführen-Button testen.
  3. Webhook-Token nur an den aufrufenden Dienst geben und regelmäßig rotieren.
  4. Scripts versioniert im Repository führen.
  5. Nicht mehr benötigte Hooks deaktivieren statt löschen.
Sicherheit: Webhook-Tokens funktionieren ohne JWT — wie Passwörter behandeln und bei Verdacht sofort rotieren. Da Scripts vollen Interpreter-Zugriff haben, dürfen Hooks nur von Admins angelegt werden. Event- und Scheduled Hooks laufen im Hintergrund — Fehler im Script brechen keine Server-Operationen ab.

6) Client-Sync und Extension-Anbindung

Desktop-Client Sync URL

https://<server>/api/connections?api_key=<read_only_key>
  • Client akzeptiert nur HTTPS-URLs.
  • Bei selbstsigniertem Zertifikat im Client Self-signed Zertifikate erlauben aktivieren.

Chrome Extension

  • Verwendet ebenfalls /api/connections?api_key=....
  • Lädt Web-Verbindungen (kind: web) und aktualisiert Badge automatisch.
  • Refresh-Zyklus im Background Service Worker: 5 Minuten.

7) Monitoring

Der Monitoring-Service läuft als separater Container und überwacht registrierte Server über einen leichtgewichtigen Go-Agent. Metriken werden in VictoriaMetrics gespeichert (90 Tage Retention).

Deployment

Monitoring und VictoriaMetrics werden automatisch mit docker compose up -d gestartet. Konfiguration über .env: MONITOR_API_KEY (Shared Secret) und MONITOR_AGENT_API_KEYS (Agent-Keys) setzen.

Agent installieren

Der Unified Go Agent (srm-agent) vereint FRP-Sync und Monitoring. Verfügbar als DEB- und RPM-Paket für Linux, sowie als Windows-Binary. 

# DEB installieren:
apt install ./srm-agent_0.8.0_amd64.deb

# Monitoring einrichten:
sudo srm-agent monitor init \
  --url https://<server>/api/monitoring \
  --api-key <MONITOR_AGENT_API_KEY> \
  --server-id <SERVER-ID>

# FRP-Sync einrichten (optional):
sudo srm-agent frpc init \
  --url https://<server> \
  --token <PROVISION-TOKEN> \
  --server-id <SERVER-ID>

# Als systemd-Service installieren:
sudo srm-agent service install

Features

Check-Typen

Ping, TCP, HTTP, SNMP und Agent-basierte Ressourcen-Checks (CPU, RAM, Disk, Services).

Agent-Plugins

Docker, ZFS, Proxmox/PBS werden automatisch erkannt. Systemd- und Windows-Services überwachbar.

Templates

Monitoring-Konfigurationen als Templates definieren und an Server zuweisen.

Alerting

Webhook- und E-Mail-Benachrichtigungen mit Cooldown. Recovery-Alerts bei Wiederherstellung.

8) Ansible Playbooks

Die integrierte Ansible-Verwaltung ermöglicht zentrale Playbook-Verwaltung über den Server und lokale Ausführung über den Desktop-Client.

Playbooks verwalten (Web-UI)

Im Admin-Bereich unter Ansible: Playbooks erstellen, bearbeiten und löschen. YAML-Inhalt wird beim Speichern validiert. Tag-basierte Filterung und Suche sind verfügbar.

Playbooks ausführen (Desktop-Client)

  1. Im Desktop-Client (Server-Modus): Seite Playbooks öffnen.
  2. Playbook auswählen.
  3. Ziel-Server auswählen (einzeln oder nach Tags).
  4. Playbook ausführen klicken — öffnet ein Terminal mit ansible-playbook.
Voraussetzung: ansible-playbook muss auf dem Desktop-Rechner installiert sein. Der Client generiert automatisch ein INI-Inventory aus den ausgewählten Servern.

9) Backup und Restore

Backup (Dateibasiert)

# im Projektroot
mkdir -p backups

tar -czf backups/srm-backup-$(date +%F-%H%M).tar.gz \
  data certs .env

Restore

docker compose down
# Backup zurückspielen (data/, certs/, .env)
docker compose up -d
Achtung: Bei Restore auf eine neue Instanz müssen TLS, DNS und IP-Filterwerte zur Zielumgebung passen.

10) Updates und Rollback

Standard-Update

docker compose pull
docker compose up -d

Rollback-Prinzip

  1. Auf vorherige Image-Tag/Digest zurückstellen (SERVER_IMAGE).
  2. docker compose up -d ausführen.
  3. Falls nötig, Daten aus letztem Backup wiederherstellen.

11) Security-Härtung Checkliste

  • SECRET_KEY mit starker Entropie setzen und geheim halten.
  • ADMIN_PASSWORD sofort ändern und separaten Admin-User anlegen.
  • ALLOWED_IPS und TRUSTED_PROXIES produktiv setzen.
  • Nur notwendige Ports veröffentlichen (idealerweise über Reverse Proxy/WAF).
  • Read-write API-Keys nur für klar definierte Integrationen ausgeben.
  • Webhook-Tokens regelmäßig rotieren und Hook-Scripts reviewen.
  • Regelmäßige Backups automatisieren und Restore testen.

12) Incident-Runbook (Kurzfassung)

Login schlägt fehl

Prüfen, ob richtige Instanz angesprochen wird und JWT-Secret konsistent ist. Bei Neuinitialisierung ggf. Admin neu anlegen.

403 auf alle Requests

IP-Filter prüfen: ALLOWED_IPS / Proxy-Header-Konfiguration.

Client-Sync Fehler

HTTPS-URL, API-Key-Recht (read) und Zertifikatsvertrauen prüfen.

Verlust von API-Key/Token

Key löschen/rotieren und neue Secrets an alle Clients verteilen.
