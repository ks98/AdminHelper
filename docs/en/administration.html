Operations Guide

Installation, runtime operations, and security hardening

This page is intended for platform administrators and operations teams. It covers deployment, configuration, backups, update strategy, webhook operations, and incident runbooks.

Start installation Hardening checklist

1) Deployment Options

Recommended: Docker Compose

Fast production setup with TLS runtime, persistent data, and image pull from registry.

Alternative: local Python process

Useful for local diagnostics/dev. TLS and public exposure must be handled separately.

First-time Docker setup

cp .env.example .env
# set SECRET_KEY and ADMIN_PASSWORD at minimum

docker compose pull
docker compose up -d

Default compose publishes host port 443 to container port 8443.

Important: if no certs are mounted in ./certs, the container entrypoint generates a self-signed certificate.

2) Configuration (.env / Environment)

Variable Example Purpose
SERVER_IMAGE docker.nevondo.com/.../server:latest Server image reference used by compose.
DOMAIN localhost Certificate CN/SAN default for auto-generated certs.
SECRET_KEY openssl rand -hex 32 JWT signing key.
ADMIN_PASSWORD strong initial password Used for first admin bootstrap.
ALLOWED_IPS 192.168.1.0/24,10.0.0.5 IP/CIDR allowlist; empty means allow all.
TRUST_PROXY_HEADERS false Global trust of forwarded headers (less secure).
TRUSTED_PROXIES 172.17.0.1,10.0.0.0/8 Preferred secure proxy trust configuration.
MONITOR_API_KEY openssl rand -hex 32 Shared secret between server and monitoring service.
MONITOR_AGENT_API_KEYS key1,key2 Comma-separated API keys for agent authentication.
MONITOR_AGENT_PORT 8480 External port for the monitoring agent endpoint.
FRP_BIND_PORT 7000 FRP protocol port (client connections).
FRP_HTTPS_PORT 7443 FRP HTTPS vhost port for web tunnels.

Persistent state

  • ./data/: SQLite DB + connections.json
  • ./certs/: TLS cert and key

3) First Login and Baseline Setup

  1. Login with user admin and value from ADMIN_PASSWORD.
  2. Create at least one additional admin user.
  3. Create read-only API key for client sync.
  4. Create non-admin read users where needed.
Note: API keys are shown in clear text only once at creation time.

4) Day-to-Day Operations

Connections

Create/update/delete from the web UI. Import/export is admin-only.

Users and roles

User management is available in the Users page. Self-delete is blocked by backend logic.

API keys

Prefer read keys. Use read_write only for trusted integrations.

Hooks

Three types: Webhook (URL trigger), Event Hook (internal events), Scheduled Hook (time-based, e.g. every hour).

5) Hooks in Operations

Hooks are Python scripts triggered in three different ways. All hook types are managed under Hooks in the admin UI.

Webhook

Triggered via HTTP POST with a token.
POST /api/hooks/trigger/<token>
Script context: payload, headers, params

Event Hook

Triggered automatically on server events (no external URL required).
Script context: event_type, event_data

Scheduled Hook

Runs on a schedule: 5m, 15m, 30m, 1h, 6h, 12h, 24h, or a cron expression.
Script context: triggered_at, last_run

Available events (Event Hook)

EventWhen
connection.createdConnection added
connection.updatedConnection changed
connection.deletedConnection removed
connections.importedBulk import
user.createdUser created
user.deletedUser deleted
server.startupServer started

Webhook trigger example

Reference scripts: docs/examples/webhook_add_connection.py, docs/examples/webhook_http_import.py 

curl -X POST "https://<server>/api/hooks/trigger/<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "connection": {
      "name": "Prod Web",
      "kind": "web",
      "url": "https://prod.example.com",
      "tags": ["prod", "web"]
    }
  }'

Event Hook script example

# Triggered e.g. on connection.created or connection.updated
import requests

hook_url = "https://chat.example.com/webhooks/notify"
msg = f"[SRM] {event_type}: {event_data.get('name', '?')} ({event_data.get('kind', '?')})"
requests.post(hook_url, json={"text": msg}, timeout=5)
result["notified"] = True

Scheduled Hook script example

# Runs e.g. daily (24h) — remove duplicate connections
connections = load_connections()
seen, unique = set(), []
for c in connections:
    key = (c.get("kind"), c.get("host"), c.get("port"))
    if key not in seen:
        seen.add(key)
        unique.append(c)
removed = len(connections) - len(unique)
if removed:
    save_connections(unique)
result["removed_duplicates"] = removed
log(f"Cleaned up at {triggered_at}: removed {removed} duplicate(s)")

Recommended operational flow

  1. Create hook in admin UI (Hooks → + Hook), select type.
  2. Write script and validate using the Run button.
  3. Webhook tokens: distribute only to the calling service and rotate regularly.
  4. Version scripts in a repository for auditability.
  5. Disable unused hooks instead of deleting them to preserve history.
Security: webhook tokens are bearer secrets — treat them like passwords and rotate on any suspected exposure. Scripts have full interpreter access; only admins may create hooks. Event and scheduled hooks run in the background — script errors do not affect server operations.

7) Monitoring

The monitoring service runs as a separate container and monitors registered servers via a lightweight Go agent. Metrics are stored in VictoriaMetrics (90-day retention).

Deployment

Monitoring and VictoriaMetrics are started automatically with docker compose up -d. Configure via .env: set MONITOR_API_KEY (shared secret) and MONITOR_AGENT_API_KEYS (agent keys).

Agent Installation

The unified Go agent (srm-agent) combines FRP sync and monitoring. Available as DEB and RPM packages for Linux, and as a Windows binary. 

# Install DEB package:
apt install ./srm-agent_0.8.0_amd64.deb

# Set up monitoring:
sudo srm-agent monitor init \
  --url https://<server>/api/monitoring \
  --api-key <MONITOR_AGENT_API_KEY> \
  --server-id <SERVER-ID>

# Set up FRP sync (optional):
sudo srm-agent frpc init \
  --url https://<server> \
  --token <PROVISION-TOKEN> \
  --server-id <SERVER-ID>

# Install as systemd service:
sudo srm-agent service install

Features

Check Types

Ping, TCP, HTTP, SNMP, and agent-based resource checks (CPU, RAM, Disk, Services).

Agent Plugins

Docker, ZFS, Proxmox/PBS are auto-detected. Systemd and Windows services can be monitored.

Templates

Define monitoring configurations as templates and assign them to servers.

Alerting

Webhook and email notifications with cooldown. Recovery alerts when checks return to OK.

8) Ansible Playbooks

The integrated Ansible management enables central playbook management via the server and local execution from the desktop client.

Manage Playbooks (Web UI)

In the admin area under Ansible: create, edit, and delete playbooks. YAML content is validated on save. Tag-based filtering and search are available.

Execute Playbooks (Desktop Client)

  1. In the desktop client (server mode): open the Playbooks page.
  2. Select a playbook.
  3. Choose target servers (individually or by tags).
  4. Click Run playbook — opens a terminal with ansible-playbook.
Prerequisite: ansible-playbook must be installed on the desktop machine. The client automatically generates an INI inventory from the selected servers.

9) Backup and Restore

Backup

mkdir -p backups

tar -czf backups/srm-backup-$(date +%F-%H%M).tar.gz \
  data certs .env

Restore

docker compose down
# restore data/, certs/, .env
docker compose up -d

10) Updates and Rollback

Normal update

docker compose pull
docker compose up -d

Rollback pattern

  1. Pin SERVER_IMAGE to a previous known-good tag/digest.
  2. Redeploy with docker compose up -d.
  3. If necessary, restore data from the latest verified backup.

11) Security Hardening Checklist

  • Set strong SECRET_KEY and protect it as secret.
  • Change bootstrap admin password immediately.
  • Set ALLOWED_IPS and TRUSTED_PROXIES for production.
  • Expose only required ports and prefer reverse-proxy controls.
  • Issue read-write API keys only when absolutely necessary.
  • Rotate webhook tokens and review hook scripts regularly.
  • Automate backups and test restores periodically.

12) Incident Runbook (Short)

Login failures

Verify target instance, credential source, and JWT secret consistency.

Global 403 responses

Check IP filtering and trusted-proxy settings first.

Sync failures

Validate HTTPS URL, API key permission, and certificate trust path.

Leaked token/key

Rotate/delete immediately and redeploy new credential to all consumers.
