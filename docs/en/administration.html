Operations Guide

Installation, runtime operations, and security hardening

This page is intended for platform administrators and operations teams. It covers deployment, configuration, backups, update strategy, webhook operations, and incident runbooks.

Start installation Hardening checklist

1) Deployment Options

Recommended: Docker Compose

Fast production setup with TLS runtime, persistent data, and image pull from registry.

Alternative: local Python process

Useful for local diagnostics/dev. TLS and public exposure must be handled separately.

First-time Docker setup

cp .env.example .env
# set SECRET_KEY and ADMIN_PASSWORD at minimum

docker compose pull
docker compose up -d

Default compose publishes host port 443 to container port 8443.

Important: if no certs are mounted in ./certs, the container entrypoint generates a self-signed certificate.

2) Configuration (.env / Environment)

Variable Example Purpose
SERVER_IMAGE docker.nevondo.com/.../server:latest Server image reference used by compose.
DOMAIN localhost Certificate CN/SAN default for auto-generated certs.
SECRET_KEY openssl rand -hex 32 JWT signing key.
ADMIN_PASSWORD strong initial password Used for first admin bootstrap.
ALLOWED_IPS 192.168.1.0/24,10.0.0.5 IP/CIDR allowlist; empty means allow all.
TRUST_PROXY_HEADERS false Global trust of forwarded headers (less secure).
TRUSTED_PROXIES 172.17.0.1,10.0.0.0/8 Preferred secure proxy trust configuration.

Persistent state

  • ./data/: SQLite DB + connections.json
  • ./certs/: TLS cert and key

3) First Login and Baseline Setup

  1. Login with user admin and value from ADMIN_PASSWORD.
  2. Create at least one additional admin user.
  3. Create read-only API key for client sync.
  4. Create non-admin read users where needed.
Note: API keys are shown in clear text only once at creation time.

4) Day-to-Day Operations

Connections

Create/update/delete from the web UI. Import/export is admin-only.

Users and roles

User management is available in the Users page. Self-delete is blocked by backend logic.

API keys

Prefer read keys. Use read_write only for trusted integrations.

Webhooks

Webhook tokens can trigger scripts without JWT. Rotate tokens regularly.

5) Webhook Scripts in Operations

Webhooks allow controlled automation of connection data. Trigger endpoint: POST /api/webhooks/trigger/<token>.

Recommended operational flow

  1. Create webhook in admin UI (Webhooks -> + Webhook).
  2. Store script in version control, then paste/deploy to server UI.
  3. Validate with test payload before production usage.
  4. Rotate token on schedule and on any potential leak.

Trigger example

Reference scripts: docs/examples/webhook_add_connection.py, docs/examples/webhook_http_import.py 

curl -X POST "https://<server>/api/webhooks/trigger/<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "connection": {
      "name": "Prod Web",
      "kind": "web",
      "url": "https://prod.example.com",
      "tags": ["prod", "web"]
    }
  }'

Python imports in scripts

Scripts have full Python interpreter access — import is unrestricted. Example of an HTTP call from a script: 

import requests

r = requests.get("https://cmdb.example.com/api/servers", timeout=10)
connections = load_connections()
for srv in r.json():
    connections.append({"id": uuid4(), "name": srv["hostname"], "host": srv["ip"], "kind": "ssh"})
save_connections(connections)
result["imported"] = len(r.json())
Security: webhook tokens are bearer secrets — treat them like passwords. Scripts have full interpreter access, so only admins can create or modify webhooks. Rotate tokens immediately on any suspected exposure.

6) Client Sync and Extension Integration

Desktop client sync URL

https://<server>/api/connections?api_key=<read_only_key>
  • Desktop sync requires HTTPS URL.
  • For self-signed certs, enable the corresponding client option.

Chrome extension source URL

Uses the same endpoint and API key model; only kind: web entries are shown.

7) Backup and Restore

Backup

mkdir -p backups

tar -czf backups/srm-backup-$(date +%F-%H%M).tar.gz \
  data certs .env

Restore

docker compose down
# restore data/, certs/, .env
docker compose up -d

8) Updates and Rollback

Normal update

docker compose pull
docker compose up -d

Rollback pattern

  1. Pin SERVER_IMAGE to a previous known-good tag/digest.
  2. Redeploy with docker compose up -d.
  3. If necessary, restore data from the latest verified backup.

9) Security Hardening Checklist

  • Set strong SECRET_KEY and protect it as secret.
  • Change bootstrap admin password immediately.
  • Set ALLOWED_IPS and TRUSTED_PROXIES for production.
  • Expose only required ports and prefer reverse-proxy controls.
  • Issue read-write API keys only when absolutely necessary.
  • Rotate webhook tokens and review scripts regularly.
  • Automate backups and test restores periodically.

10) Incident Runbook (Short)

Login failures

Verify target instance, credential source, and JWT secret consistency.

Global 403 responses

Check IP filtering and trusted-proxy settings first.

Sync failures

Validate HTTPS URL, API key permission, and certificate trust path.

Leaked token/key

Rotate/delete immediately and redeploy new credential to all consumers.
