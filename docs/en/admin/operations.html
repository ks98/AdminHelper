Operations & Config

Where data lives, how logs are named, how to take consistent backups – the operational knobs you need to keep the system healthy over time.

Data Directories & Postgres Volume

State splits across two locations: DATA_DIR for file-based artifacts, the postgres-data Docker volume for the actual databases.

PathPurpose
postgres-data (volume)PostgreSQL cluster with two DBs: adminhelper (users, connections, servers, hooks) and adminhelper_monitor (checks, alerts, templates, agent keys).
/data/.bootstrap_tokenOne-time setup-token hash for the first admin (see installation guide).
/data/.secret_keyAuto-generated 256-bit secret; never delete without re-issuing all tokens.
frp-config (volume)Only the generated frps.toml (no cert material anymore). The FRP TLS material is provided by the ca-issuer (volume frps-certs, recoverable). The old server-run FRP master PKI (volume frp-pki) has been removed — see ca-pki below.
ca-pki (volume)PKI hierarchy of the ca-issuer: root (passphrase-encrypted) + intermediates (tunnel/access/internal). The central backup target of the internal CA — losing it means every client must re-enroll. Issuer-private, never mounted into the gateway/frps.
gateway-certs (volume)The gateway's TLS material (access-signed leaf + key + CA trust bundle), provisioned by the ca-issuer. Recoverable — the issuer re-provisions it.
victoria-data (volume)VictoriaMetrics storage – 90 d retention (separate, not in this scope).

Backups

scripts/backup.sh bundles the whole state — including the CA crown jewel — into one tarball, with the stack running:

bash
./scripts/backup.sh                 # -> ./backups/adminhelper-backup-<ts>.tar.gz
./scripts/backup.sh --with-victoria # also the (large) metric history

Included: ca-pki (Root + intermediates), pg_dump of both DBs, monitoring-data, optionally victoria-data, and .env. Not included — and not needed — are gateway-certs/frps-certs: the ca-issuer re-provisions them from ca-pki on boot.

Keep CA_ROOT_PASSPHRASE separately

The passphrase is intentionally not written into the backup — it encrypts the Root key, which IS in the backup; storing both together would defeat the encryption (ADR 0001 §5/D7). Keep it in a password manager. A restored stack runs without it; it is only needed to rotate intermediates.

For automatic daily DB backups, uncomment the pg-backup service in docker-compose.yml (DBs only, 7-day retention) — it does not replace the full backup.sh incl. the CA.

Restore

bash
./scripts/restore.sh ./backups/adminhelper-backup-<ts>.tar.gz

Stops the stack, restores the volumes + DBs and brings it back up. The restored Root is identical — already-enrolled agents stay trusted, no re-enrollment. The backed-up .env is written to .env.restored (it never overwrites an existing .env); on a fresh host, add the separately kept CA_ROOT_PASSPHRASE and move it to .env.

Logging

The server container uses named loggers prefixed adminhelper.*; some modules log under their module path app.*. Useful logger names:

  • adminhelper.auth / adminhelper.auth_router – logins, refresh rotations, reuse detection
  • adminhelper.rate_limit – blocked requests
  • adminhelper.provisioning – server provisioning
  • adminhelper.servers / adminhelper.ansible – server and Ansible actions
  • app.modules.frp.pki / app.modules.frp.docker_manager – PKI actions and the FRP container
  • app.modules.hooks.scheduler – hook executions incl. subprocess exit codes

The monitoring container logs under the monitor.* prefix (e.g. monitor.engine, monitor.scheduler, monitor.alerter, monitor.victoria). Both containers write plain-text lines to stdout, not JSON.

Multi-Worker Notes

The default compose stack runs a single Uvicorn worker – APScheduler assumes one scheduling instance, and the in-memory rate-limit fallback is per-process. For horizontal scaling:

  • Scale only the HTTP layer (add replicas behind a load balancer with sticky sessions for refresh cookies).
  • Keep exactly one worker with ENABLE_SCHEDULER=true; others run with scheduler disabled.
  • Redis is required – the in-memory rate-limit fallback breaks across workers.
Update flow

docker compose pull && docker compose up -d. The server runs alembic upgrade head on startup; downtime is typically below 10 seconds.

Timezone

TZ is passed to all containers. Default Europe/Berlin. It affects audit-log timestamps and cron-like hook jobs.

Renewing certificates

The gateway's TLS certificate is issued by the ca-issuer (access-signed, 365 days). It is provisioned into the gateway-certs volume on first start; a redeploy/restart of the ca-issuer renews it when needed. The FRP client certificates expire after 12 months – see FRP Tunnel for rotation.

Enforcing mTLS (CERT_REQUIRED)

Since 0.29.0 the data plane (:443) runs enforced by default (MTLS_ENFORCE=true): the gateway requires a valid client certificate at the TLS handshake (CERT_REQUIRED) and the server rejects protected routes without a valid scope with 403. A fresh install is usable enforced via scripts/install.sh (it fetches the first admin's cert out-of-band). Permissive (MTLS_ENFORCE=false) lets certless clients through and only logs — needed only for a manual first bootstrap. Both are driven by one variable: MTLS_ENFORCE.

Enroll every client first, then flip

Only enforce once every client – agent, desktop and your own browser – holds a certificate, otherwise you lock yourself out. Agent and desktop enroll automatically; prepare the browser via the desktop P12 export (Users & Access → Browser access). Test on real hardware first: Windows desktop enrollment and the browser import.

Enforce:

bash
# in your .env:
MTLS_ENFORCE=true

# restart gateway + server (they read the variable at startup):
docker compose up -d gateway server

The gateway log confirms the mode at startup: mTLS ENFORCED (CERT_REQUIRED) or mTLS permissive.

Rollback (any time, lossless): set MTLS_ENFORCE=false and docker compose up -d gateway server – the data plane is permissive again immediately.

New clients after enforcing

The enrollment plane (:8444) always stays certless and token-gated – new agents and desktops fetch their first certificate there regardless of enforcement. A new user: an admin mints them an enrollment token (POST /api/enrollment/token/for) and hands it over out-of-band; in the desktop client the newcomer picks "First time? Set up this device with a token", enters the server URL + token and enrolls – without a prior login. They then sign in normally (now with a certificate). No permissive window is needed for this (the decoupled enrollment door, ADR 0003).

Only the very first admin (there is no admin yet who could issue a token) is the special case — easiest via scripts/install.sh, which creates the admin + a first token out-of-band and then arms enforcement. Manually without the script: a one-time permissive setup window (MTLS_ENFORCE=false, enroll the first admin + desktop, then back to true).
