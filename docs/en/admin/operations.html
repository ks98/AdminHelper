Operations & Config

Where data lives, how logs are named, how to take consistent backups – the operational knobs you need to keep the system healthy over time.

Data Directories & Postgres Volume

State splits across two locations: DATA_DIR for file-based artifacts, the postgres-data Docker volume for the actual databases.

PathPurpose
postgres-data (volume)PostgreSQL cluster with two DBs: adminhelper (users, connections, servers, hooks) and adminhelper_monitor (checks, alerts, templates, agent keys).
/data/.bootstrap_tokenOne-time setup-token hash for the first admin (see installation guide).
/data/.secret_keyAuto-generated 256-bit secret; never delete without re-issuing all tokens.
/data/frp-config/pki/FRP root CA (10 y) + server cert + per-client certs.
/data/certs/Self-signed TLS for the FastAPI port 8443.
victoria-data (volume)VictoriaMetrics storage – 90 d retention (separate, not in this scope).

Backups

Two backup paths — combine both for a complete restore:

1. PostgreSQL dump (databases)

bash
# Manual dump of both DBs (no stop required):
docker compose exec postgres pg_dump -U adminhelper -d adminhelper -Fc -f /tmp/adminhelper.dump
docker compose exec postgres pg_dump -U adminhelper -d adminhelper_monitor -Fc -f /tmp/adminhelper_monitor.dump
docker compose cp postgres:/tmp/adminhelper.dump ./backups/
docker compose cp postgres:/tmp/adminhelper_monitor.dump ./backups/

# Or via the bundled script:
docker compose exec postgres /usr/local/bin/pg-backup.sh    # if scripts/ is mounted

For automatic daily backups, uncomment the pg-backup service in docker-compose.yml — writes to ./backups/, 7-day rolling retention.

2. File data (PKI, bootstrap token, certificates)

bash
tar czf adminhelper-files-$(date +%F).tar.gz data/

Restore

bash
# Restore DB:
docker compose cp ./backups/adminhelper-2026-05-02.dump postgres:/tmp/restore.dump
docker compose exec postgres pg_restore -U adminhelper -d adminhelper --clean --if-exists --no-owner /tmp/restore.dump

# Files:
tar xzf adminhelper-files-2026-05-02.tar.gz

Logging

Four logger names emit structured JSON by default:

LoggerWhat it records
appHTTP requests, auth events, CRUD actions
monitoringCheck execution, ingest results, alert state changes
frpTunnel lifecycle, cert issuance/rotation
schedulerAPScheduler jobs (queued / ran / failed / retried)

Override per-logger level via env, e.g. LOG_LEVEL_FRP=DEBUG. Systemd/journald picks up the JSON automatically; tools like jq work out of the box:

bash
docker compose logs server | jq 'select(.logger == "frp" and .level == "ERROR")'

Multi-Worker Notes

The default compose stack runs a single Uvicorn worker – APScheduler assumes one scheduling instance, and the in-memory rate-limit fallback is per-process. For horizontal scaling:

  • Scale only the HTTP layer (add replicas behind a load balancer with sticky sessions for refresh cookies).
  • Keep exactly one worker with ENABLE_SCHEDULER=true; others run with scheduler disabled.
  • Redis is required – the in-memory rate-limit fallback breaks across workers.
Update flow

docker compose pull && docker compose up -d. The server runs alembic upgrade head on startup; downtime is typically below 10 seconds.
