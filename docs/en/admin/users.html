Users & Access

User accounts, roles, API keys, and the rate-limit/audit layers that keep the access surface tight.

Roles

RolePermissions
adminFull access incl. user management, hooks, FRP PKI, settings
operatorConnection & server CRUD, monitoring, audit log read
viewerRead-only on connections and monitoring dashboard

JWT Refresh Rotation

Access tokens are HS256, 8 h lifetime, passed in Authorization: Bearer. Refresh tokens are 7 d and live in an httpOnly cookie. On every refresh:

  1. The server verifies signature + expiry.
  2. The old jti is written to the refresh blacklist.
  3. A brand-new access + refresh pair is issued.
  4. If a blacklisted jti appears again → reuse detection: all sessions of that user are invalidated immediately.
Rate limit

Login is limited to 5 failed attempts per 15 min per IP, keyed in Redis (ratelimit:login:<ip>). The in-memory fallback works for single-worker setups only.

API Keys

API keys are issued under Settings → API Keys with a scope list. The server stores only the SHA-256 hash – the clear-text key is shown once at creation time. Scopes include:

  • connections_read / connections_write
  • monitor_ingest (agent push)
  • monitor_read (dashboards, extension)
  • web_only (browser extension – strictly read + filter kind=web)
  • hooks_admin (dangerous – allows RCE via hook scripts)

Audit Log

Every state-changing action writes to the AuditEntry table: user_id, action, target, ip, timestamp, details (JSON). Query via Settings → Audit or the /api/audit endpoint.

MFA

TOTP-based second factor, enabled per user in their profile. Backup codes are one-time, SHA-256-hashed at rest. Admins can force MFA in Settings → Security.
