Users & Access

AdminHelper distinguishes three kinds of access: interactive users (web and desktop login), API keys (agents and scripts), and FRP provision tokens (single use). This page explains how to manage all three.

User roles

RoleCan
adminEverything: manage users, create API keys, register servers, define hooks, global settings.
userManage own connections, read servers and monitoring. No admin actions.

Creating a user

  1. Open Users → New in the web panel
  2. Assign username and password; the password is hashed with bcrypt
  3. Choose the role – admins see and edit everything

Admins can reset other users' passwords at any time. Regular users can change their own password in the settings.

Login flow

The login flow uses JWT (HS256) with refresh-token rotation:

  • Access token: valid for 8 h, sent on every request in the Authorization header.
  • Refresh token: valid for 7 days, rotated on every use (the old one is blacklisted immediately).
  • Reuse detection: if a blacklisted refresh token is submitted again, the request is rejected and the event is logged as a possible token theft.
Login rate limit

5 failed login attempts per IP within 60 seconds → further attempts are rejected with 429. A successful login resets the counter. Implemented via Redis (fixed-window), with an in-memory fallback for single-worker setups.

API keys

API keys authenticate non-interactive clients (Go agent, scripts, CI jobs). Differences from user credentials:

  • No expiry, but revocable with one click
  • One of two permissions: read (read only) or read_write (read and write)
  • The server stores only the SHA-256 hash; the clear text is gone after it is shown once

Creating API keys:

  1. Web panel → API Keys → New
  2. Assign a name and choose the permission (read or read_write)
  3. Copy the clear-text key immediately – it is never shown again
  4. Send it as the HTTP header X-API-Key: <key> (fallback: query parameter ?api_key=… for the browser extension and sync URLs)

IP restriction

With ALLOWED_IPS all access can be limited to a whitelist. Example:

.env
ALLOWED_IPS=10.0.0.0/8,192.168.1.0/24,203.0.113.42

If the list is left empty, access is open (for reverse-proxy- or VPN-protected setups).
