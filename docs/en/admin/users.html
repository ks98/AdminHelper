Users & Access

AdminHelper distinguishes three kinds of access: interactive users (web and desktop login), API keys (agents and scripts), and FRP provision tokens (single use). This page explains how to manage all three.

User roles

RoleCan
adminEverything: in the web panel manage users, create API keys, define hooks, the FRP server configuration and global settings; in the desktop client (server mode) register servers and maintain connections, tunnels, monitoring, and Ansible.
userManage own connections, read servers and monitoring. No admin actions.

Creating a user

User management is part of the web admin panel – Users is also the default page after login.

  1. Open Users → New in the web panel
  2. Assign username and password; the password is hashed with bcrypt
  3. Choose the role – admins see and edit everything

Admins can reset other users' passwords at any time. Regular users can change their own password in the settings.

The assignment of servers to a user stays in the web panel: it is maintained directly in the user dialog (it controls which servers the user sees). The server entries themselves are created and edited in the desktop client (server mode, Infrastructure).

Login flow

The login flow uses JWT (HS256) with refresh-token rotation:

  • Access token: valid for 8 h, sent on every request in the Authorization header.
  • Refresh token: valid for 7 days, rotated on every use (the old one is blacklisted immediately).
  • Reuse detection: if a blacklisted refresh token is submitted again, the request is rejected and the event is logged as a possible token theft.
Login rate limit

5 failed login attempts per IP within 60 seconds → further attempts are rejected with 429. A successful login resets the counter. Implemented via Redis (fixed-window), with an in-memory fallback for single-worker setups.

Browser access (client certificate)

Once the data plane is switched to CERT_REQUIRED, every browser opening the web panel needs a valid client certificate from the internal CA. Unlike the agent and the desktop, a browser cannot fetch a certificate itself — you export one once from the desktop client and import it into the browser's certificate store.

  1. Export: Desktop client → Settings (in server mode, signed in) → Export browser certificate. Choose a password (at least 8 characters) and click Export. The client enrolls a long-lived access certificate, packages it as a password-protected .p12 and asks via a save dialog where to write it (default name adminhelper-browser.p12).
  2. Import – Chrome/Edge: Settings → Privacy and security → Manage certificates → Your certificates → Import, pick the .p12, enter the password. (On Windows/macOS the certificate goes into the OS store; the exact menu wording varies by version.)
  3. Import – Firefox: Settings → Privacy & Security → Certificates → View Certificates → Your Certificates → Import, pick the .p12, enter the password.
  4. Connect: reload the web panel. On the first request the browser asks which client certificate to present — choose the imported one.
Browser cert under enforced mTLS

Since 0.29.0 the data plane runs enforced by default (CERT_REQUIRED) — a browser without an imported certificate is rejected at the TLS handshake and the web panel won't load. So hand out the .p12 to everyone who uses the panel in a browser. The browser certificate is long-lived and does not auto-renew — just export and import a fresh one before it expires.

Onboarding a new user under enforced mTLS

Once the data plane is enforced (CERT_REQUIRED, see Operations & Config), a brand-new client without a certificate can't get through the :443 handshake — not even to log in. Onboarding therefore goes through the decoupled enrollment door (ADR 0003): cert first via a token, login second.

  1. Create the user (admin) – as under "Creating a user" above. The certificate CN will be exactly this username.
  2. Issue an enrollment token (admin, who already holds a cert): POST /api/enrollment/token/for with { "username": "<new-user>" }. The response carries a one-time token, valid for 10 minutes.
  3. Hand the token over out-of-band – via a secure channel to the new user (copy-paste). It is single-use and short-lived.
  4. Enroll the device (new user): in the desktop client pick "First time? Set up this device with a token", enter the server URL + token → Enroll device. The client generates a key on-device, fetches its certificate at :8444 and stores it in the OS keyring. No login needed.
  5. Sign in – then the normal login (username/password); the client now presents its certificate at the handshake.
  6. Optional browser – for the web panel the user exports a .p12 from the desktop and imports it (see "Browser access" above).
Exception: the very first admin

For the first admin there is nobody yet who could issue a token. Easiest is scripts/install.sh (it creates the admin + a first token out-of-band and arms enforcement). Manually without the script: a one-time permissive window (MTLS_ENFORCE=false, create the first admin + enroll the desktop, then MTLS_ENFORCE=true). From then on every further onboarding goes through the token door — no more loosening.

API keys

API keys authenticate non-interactive clients (Go agent, scripts, CI jobs). Differences from user credentials:

  • No expiry, but revocable with one click
  • One of two permissions: read (read only) or read_write (read and write)
  • The server stores only the SHA-256 hash; the clear text is gone after it is shown once

Creating API keys:

  1. Web panel → API Keys → New
  2. Assign a name and choose the permission (read or read_write)
  3. Copy the clear-text key immediately – it is never shown again
  4. Send it as the HTTP header X-API-Key: <key> (fallback: query parameter ?api_key=… for sync URLs)

IP restriction

With ALLOWED_IPS all access can be limited to a whitelist. Example:

.env
ALLOWED_IPS=10.0.0.0/8,192.168.1.0/24,203.0.113.42

If the list is left empty, access is open (for reverse-proxy- or VPN-protected setups).
