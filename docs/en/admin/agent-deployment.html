Agent Deployment

One Go binary, two packages (DEB + RPM), shipped with matching service units. The provision flow redeems a one-time token and installs a persistent client certificate.

Debian / Ubuntu

The DEB and RPM packages are attached as assets to every GitHub release. Download the file and install it locally:

bash
VERSION=0.26.0
curl -fsSLO "https://github.com/ks98/AdminHelper/releases/download/v${VERSION}/adminhelper-agent_${VERSION}_amd64.deb"

# Installs deps and supersedes the old srm-* packages (Replaces/Conflicts):
sudo apt install "./adminhelper-agent_${VERSION}_amd64.deb"

RHEL / Rocky / Fedora

Grab the matching .rpm asset from the releases page, then install it locally:

bash
sudo dnf install ./adminhelper-agent-*.x86_64.rpm

From the AdminHelper package repo (apt upgrade / dnf upgrade)

The AdminHelper server itself hosts a GPG-signed apt/rpm repo on the gateway's repo plane (port REPO_PORT, default 8445, certless). Agent hosts then pull new versions through the normal package manager instead of a manual download — meant mainly for ongoing updates across many servers. Prerequisite: the server runs with a configured repo signing key (see CI/CD & Release); otherwise /repo just returns 404.

The trust root is the GPG fingerprint, not the transport: package authenticity rests on the signature (like any distro mirror). Check the fingerprint of the fetched key against the published value (65490B6F130D967CD6C82411934E128434DB8E0B) before you continue.

Debian/Ubuntu (apt)

bash
SERVER=srm.example.com                 # your AdminHelper server
REPO="https://${SERVER}:8445/repo"

# 1) Fetch the signing key + verify the fingerprint (trust root!)
sudo install -d -m 0755 /usr/share/keyrings
curl -fsSL "${REPO}/apt/adminhelper-archive-keyring.gpg" \
  | sudo tee /usr/share/keyrings/adminhelper-archive-keyring.gpg >/dev/null
gpg --show-keys --with-fingerprint /usr/share/keyrings/adminhelper-archive-keyring.gpg

# 2) Repo source (deb822); Signed-By binds the repo to exactly this key
sudo tee /etc/apt/sources.list.d/adminhelper.sources >/dev/null <<EOF
Types: deb
URIs: ${REPO}/apt
Suites: stable
Components: main
Signed-By: /usr/share/keyrings/adminhelper-archive-keyring.gpg
EOF

# 3) TLS against the internal server cert. Option A (hardened, host already
#    enrolled -> the internal CA sits at /etc/adminhelper/identity/ca.crt):
echo 'Acquire::https::'"${SERVER}"'::CAInfo "/etc/adminhelper/identity/ca.crt";' \
  | sudo tee /etc/apt/apt.conf.d/99adminhelper-ca >/dev/null
#    Option B (simplest bootstrap, no CA file) — the GPG signature still carries
#    authenticity:
# echo 'Acquire::https::'"${SERVER}"'::Verify-Peer "false";' \
#   | sudo tee /etc/apt/apt.conf.d/99adminhelper-ca >/dev/null

sudo apt update
sudo apt install adminhelper-agent     # later updates: apt upgrade

Rocky/Alma/RHEL (dnf)

bash
SERVER=srm.example.com
REPO="https://${SERVER}:8445/repo"
sudo tee /etc/yum.repos.d/adminhelper.repo >/dev/null <<EOF
[adminhelper]
name=AdminHelper Agent
baseurl=${REPO}/rpm
enabled=1
gpgcheck=1
repo_gpgcheck=1
gpgkey=${REPO}/rpm/RPM-GPG-KEY-adminhelper
# Option A (hardened): sslcacert=/etc/adminhelper/identity/ca.crt
# Option B (simple, no CA file):
sslverify=0
EOF
sudo dnf install adminhelper-agent     # later updates: dnf upgrade
CA pinning vs. first install

Option A requires the host to be provisioned/enrolled already (then the internal CA sits at /etc/adminhelper/identity/ca.crt). For the very first install of a not-yet-enrolled host use Option B or the local .deb/.rpm download above; after provisioning you can switch to CA pinning. Package integrity is covered by the GPG signature either way.

Windows

Grab the latest MSI from the GitHub releases page (the Windows MSI is built best-effort and may lag behind the Linux packages). Install with elevated privileges – the installer registers the Windows service.

Provisioning

In the desktop client (server mode): Infrastructure → Add server, then open the Provisioning tab of that server entry. The token is valid for 24 h and redeemable exactly once. You also need the server ID shown for that server entry.

bash
# Linux
sudo adminhelper-agent provision \
  --url       https://srm.example.com \
  --token     TOKEN_VALUE \
  --server-id SERVER_ID
powershell
# Windows (PowerShell, as Admin)
& 'C:\Program Files\AdminHelper\adminhelper-agent.exe' provision `
    --url       https://srm.example.com `
    --token     TOKEN_VALUE `
    --server-id SERVER_ID

After provisioning the agent holds a client certificate signed by the FRP CA and enables the adminhelper-agent.timer systemd unit (Linux) or the Windows service (auto-start). Metrics are then pushed tunnel-free to the AdminHelper server (port 443) at POST /api/monitoring/agent/{server_id}/report using the server-specific X-API-Key; the server forwards them internally to the monitoring service. No separate monitoring port needs to be reachable. Upgrade note: when moving to the internal-only monitoring (server proxy instead of the host port 8480), existing agents must be re-provisioned once so their stored MONITOR_URL switches to /api/monitoring.

mTLS enrollment (automatic): during provisioning the agent also fetches its own client certificate from the ca-issuer — an ECDSA key generated on-device (never leaves the host), CSR sent to the gateway's enroll plane on port 8444. Cert + key are stored under /etc/adminhelper/identity/ (key 0600) and the internal root CA is pinned. The agent then presents this cert on all server pushes and auto-renews it at ~50 % of its lifetime. Best-effort: if enrollment fails (e.g. port 8444 unreachable) the agent keeps working with the API key for now.

Subcommand reference

All subcommands of the agent binary. Required flags are marked with *.

CommandFunctionImportant flags
versionPrint the version—
runRun FRPC sync + monitor push (without flag: continuous, every 5 minutes)--once
provisionProvision the server against AdminHelper (server API key + optional monitor + FRP)--url*, --token*, --server-id*, --cacert, --insecure
frpc syncOne-off config sync—
monitor initFirst-time setup of the monitor agent--url*, --api-key*, --server-id*, --services, --cacert, --insecure
monitor pushOne-off metrics push—
service installRegister and start the OS service—
service uninstallDeregister the OS service—

--insecure applies only to the one-off provision call: the agent captures the certificate the server presents and pins it (trust-on-first-use), so the recurring sync/push loop verifies again — instead of disabling TLS verification permanently. With --cacert nothing is skipped at all.

Verification

bash
systemctl status adminhelper-agent.timer
journalctl -u adminhelper-agent.service -n 50
adminhelper-agent run --once

Logs

The agent also writes its logs to a rotating file — independent of the service manager, handy for a bug report:

  • Linux: /var/log/adminhelper/agent.log
  • Windows: %ProgramData%\AdminHelper\logs\agent.log

Same format as stdout/journal: time=… level=INFO msg="…" component=frpc (component: agent, frpc or monitor). Rotates at 10 MB, 5 compressed archives. Verbosity is set via LOG_LEVEL (env) or --log-level debug|info|warn|error.

For a bug report, adminhelper-agent diagnostics writes a redacted report (version, OS, config without secrets, log excerpt) to /tmp/ — review it and attach it to a GitHub issue.

Migration from legacy agents

The DEB/RPM packages carry Replaces + Conflicts for srm-frpc-client, srm-monitor-agent, and srm-agent. A single apt install adminhelper-agent removes the old trio and preserves the config.
