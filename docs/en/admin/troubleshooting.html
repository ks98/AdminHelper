Troubleshooting

Quick diagnostic paths for the five most common failure classes: login, TLS, agents, monitoring, and FRP.

Install aborts

  • password authentication failed for user "adminhelper" (install waits 240s, then aborts): a stale postgres-data volume from an earlier (aborted) attempt. Postgres only sets the password on the volume's first init, so a freshly generated .env password (e.g. after deleting the folder) no longer matches. Wipe the volumes — safe as long as there is no real data yet: 
    cd adminhelper
docker compose down -v
# or directly on install:
./scripts/install.sh … --reset
    Gotcha: the compose project name is the directory name, so two installs in different paths sharing the basename adminhelper reuse the same adminhelper_postgres-data volume — a "fresh" folder can still hit an old one.

Login fails

  1. Rate-limited? Check docker compose logs server | grep ratelimit – 5 bad attempts / 15 min blocks the IP.
  2. Password forgotten? Set ADMIN_PASSWORD=… in .env, restart, log in, clear it out again.
  3. Refresh cookie rejected? Check the clock skew between the browser and server (TOTP also relies on this).

TLS / reverse proxy

bash
# Cert valid?
openssl s_client -connect srm.example.com:443 -servername srm.example.com < /dev/null 2>/dev/null | openssl x509 -noout -dates

# X-Forwarded-For trusted?
docker compose logs server | grep 'untrusted proxy'

Agent doesn't check in

bash
adminhelper-agent run --verbose   # one-shot debug run
systemctl status adminhelper-agent.timer
journalctl -u adminhelper-agent.service -n 100 --no-pager

Common causes: clock skew > 5 min, expired client cert, DNS failure, firewall blocking 7443/tcp.

Desktop client, “Allow self-signed certificates”: the first connection pins the server certificate's SHA-256 fingerprint (trust-on-first-use, like SSH known_hosts) in the OS keyring; afterwards only that exact certificate is accepted and any change is rejected as a possible MITM. If a legitimate certificate rotation triggers AdminHelper TOFU: … changed, click Reset pinned certificate in the desktop settings and reconnect (re-pins on next connect).

Desktop client, already-enrolled device (mTLS): once the device is enrolled, the client verifies the server certificate against the CA pinned at enrollment (no longer the leaf pin). After a server reinstall or re-created PKI the old CA no longer matches and the connection fails with AdminHelper: Das Server-Zertifikat wird nicht mehr von der … gepinnten CA gedeckt (mögliche MITM-Attacke). If the change is expected, click Reset device identity in the desktop settings (mode Server) — this clears the mTLS device certificate and the certificate pin — then sign in and re-enroll the device.

Monitoring ingest dropping samples

bash
# Check auth
docker compose logs monitoring | grep 'invalid api key'

# Check VictoriaMetrics health
docker compose exec monitoring curl -s http://victoria:8428/health

# Disk full?
docker compose exec monitoring df -h /data/victoria

FRP visitor connect hangs

  • Visitor-port pool exhausted? Scale VISITOR_PORT_END.
  • Client cert expired? Check adminhelper-agent frpc rotate.
  • Wrong sk on STCP tunnels – both sides must match.

Diagnostics bundle

Collect a redacted diagnostics bundle for a bug report, on the server host:

bash
./scripts/diagnostics.sh               # writes adminhelper-diagnostics-<ts>.tar.gz
./scripts/diagnostics.sh --tail 1000   # more log lines per service

The bundle contains report.txt (versions, OS, docker, service status), compose.yml, an env.sanitized (secret values masked) and the last log lines per service. Then attach it to a GitHub issue.

For the desktop client: Settings → Diagnostics → "Create diagnostics report". For an agent on a managed server: adminhelper-agent diagnostics.

Skim before sending

Secrets (SECRET_KEY, passwords, API keys, JWT/Bearer tokens) are masked automatically, but redaction is best-effort — please glance over the bundle before uploading.
