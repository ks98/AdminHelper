Monitoring

Define checks, set thresholds, wire alerts into hooks. The actual evaluation happens in the separate monitoring service, backed by VictoriaMetrics for metric storage with 90 days retention.

Check Types

KindRuns whereTypical use
httpMonitoring serviceAPI/website reachability, status codes, latency
tcpMonitoring servicePort reachability (e.g. DB, SMTP)
pingMonitoring serviceICMP echo, RTT, packet loss
sshMonitoring serviceSSH banner check – does the daemon respond?
cpuAgentCPU load / utilization
ramAgentMemory & swap usage
diskAgentFill level per mountpoint
smartAgentSMART attributes, NVMe Critical-Warning bits
dockerAgentContainer health / restart count
zfsAgentPool state, scrub age, resilver progress

Thresholds & Alerts

Each check has a threshold block with three severities: warn, crit, recover. Transitions fire events that can trigger hooks (Slack, email, webhook).

yaml
kind: http
config:
  url: https://api.example.com/health
  expected_codes: [200]
  timeout_sec: 5.0
threshold:
  warn:  { latency_ms: 1000 }
  crit:  { latency_ms: 3000, status: unreachable }
interval_sec: 60

NVMe Critical Warning

The smart collector decodes the NVMe Critical-Warning byte:

BitMeaning
0Available spare below threshold
1Temperature above critical
2NVM subsystem reliability degraded
3Media placed in read-only mode
4Volatile memory backup device failed
5Persistent memory region unreliable
Agent-based checks need an agent

CPU/RAM/Disk/SMART/Docker/ZFS require the Go agent on the target. Server-based probes (http, tcp, ping, ssh) run from the monitoring container and don't need an agent.

Charts

Charts on the Server details page query VictoriaMetrics via the AdminHelper proxy – the allowlist permits only /api/v1/query and /api/v1/query_range, so raw-host SSRF is impossible.
