Monitoring

Define checks, set thresholds, wire alerts into hooks. Since v0.32.0 checks, alert rules, and templates are maintained in the desktop client (server mode) – on the fleet-wide Monitoring page and per server in the Monitoring tab under Infrastructure. The actual evaluation happens in the separate monitoring service, backed by VictoriaMetrics for metric storage with 90 days retention.

Check Types

The type identifier (check_type) is exactly the value the desktop client shows in its dropdown. Server-side probes (ping, tcp, http) run inside the monitoring container; all agent_* and plugin checks evaluate data pushed by the agent.

KindRuns whereTypical use
pingMonitoring serviceICMP reachability + round-trip time
tcpMonitoring servicePort reachability + connect time
httpMonitoring serviceHTTP status code, latency, optional body text match
agent_pingAgent (push)Did the agent report within stale_minutes?
agent_resourcesAgent (push)CPU, RAM, disk mountpoints and temperature sensors vs. thresholds
service_processAgent (push)systemd units: auto mode (failed + enabled-but-inactive) or list mode (explicit service names)
proxmox_backupAgent (push)Backup age of all VMs/CTs vs. max_backup_age_hours
zfs_healthAgent (push)Pool health (DEGRADED/FAULTED/…) + capacity vs. capacity_warn/crit
docker_healthAgent (push)Container state + unhealthy detection, optional ignore_containers
smart_healthAgent (push)SMART attributes + temperature for ATA (HDD/SSD) and NVMe
Agent plugins are auto-detected

The plugin checks proxmox_backup, zfs_health and docker_health only evaluate data when the agent finds the respective subsystem (Proxmox, ZFS, Docker) on the host. If the subsystem is absent, the check simply has no data – that is not an error.

Thresholds & Alerts

Each check has a threshold block with three severities: warn, crit, recover. Transitions fire events that can trigger hooks (Slack, email, webhook).

yaml
kind: http
config:
  url: https://api.example.com/health
  expected_codes: [200]
  timeout_sec: 5.0
threshold:
  warn:  { latency_ms: 1000 }
  crit:  { latency_ms: 3000, status: unreachable }
interval_sec: 60

NVMe Critical Warning

The smart collector decodes the NVMe Critical-Warning byte:

BitMeaning
0Available spare below threshold
1Temperature above critical
2NVM subsystem reliability degraded
3Media placed in read-only mode
4Volatile memory backup device failed
5Persistent memory region unreliable
Agent-based checks need an agent

The agent_* and plugin checks (agent_resources, service_process, smart_health, docker_health, zfs_health, proxmox_backup) require the Go agent on the target. Server-based probes (ping, tcp, http) run from the monitoring container and don't need an agent.

Charts & overview (desktop client)

The Monitoring page of the desktop client combines uPlot line charts with a checks overview, the alert rules, and the alert log. Charts query VictoriaMetrics via the AdminHelper proxy – the allowlist permits only /api/v1/query and /api/v1/query_range, so raw-host SSRF is impossible. The view refreshes automatically.

Templates

A monitoring template bundles several check and alert definitions and is assigned to a server instead of creating each check by hand. On assignment the definitions are materialized into real checks and alert rules for that server.

  1. Desktop client → Monitoring → Templates → create a template, add check and alert definitions
  2. Assign the template to a server – AdminHelper creates all contained checks and alert rules
  3. When the template is edited later, a live sync runs: every assigned server is updated automatically (checks created, updated or deleted). Each definition carries a stable def_id the sync uses to match.

Definitions may use placeholders that are substituted per server at assignment time:

PlaceholderReplaced with
{{hostname}}Server hostname
{{server_name}}Server display name
{{server_id}}Internal server ID

Removing an assignment deletes the checks and alerts it created. Deleting a template removes them on all assigned servers.

Notifications & Alerting

Alert rules fire when a check changes status (e.g. ok → critical). Each rule optionally filters by severity and server and dispatches over one channel.

ChannelConfiguration
webhookurl – AdminHelper sends a JSON POST with check name, type, server, severity, old/new status and a plain-text message (10 s timeout, no redirects, success on HTTP < 300). For security (SSRF), targets that resolve to private/reserved addresses (e.g. 127.0.0.0/8, 10/8, 169.254.0.0/16) are rejected — reach an internal webhook receiver via a publicly resolvable name.
emailrecipients (or to, also as CSV). SMTP defaults from the environment variables SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD, SMTP_FROM; overridable per rule. Port 465 uses implicit TLS (SMTPS), port 587 uses STARTTLS.

Cooldown: After a successfully sent alert, cooldown_minutes (default 30) suppresses further alerts from the same rule for the same check.

Recovery: When a check returns to ok, a recovery notification is sent – it deliberately bypasses the cooldown so the all-clear always gets through. Every dispatch is recorded in the alert log with success/error. The alert log is pruned automatically every day (90-day retention).

Personal notifications (notification hub)

Independently of the rule-based alert channels above, the notification hub fans events out per user: each user decides what they are notified about and how. Where the webhook/email rules above send to fixed destinations (e.g. a team webhook), the hub targets the people assigned to a server.

What is delivered

  • Bell in the desktop client (header): a personal feed with an unread counter that refreshes in real time (server push, while a server session is active) — a new notification appears immediately, without waiting for the next poll. Opening it marks the entries as read.
  • Desktop notifications (native OS toasts): optional, enabled under Settings → Notifications (asks for the system permission once).
  • E-mail: additionally, when enabled in a rule and a contact e-mail is set.

Configure your own rules

Under Settings → Notifications in the desktop client every user sets a contact e-mail and any number of rules. Each rule consists of:

FieldMeaning
ScopeAll servers (all you may see), Tag (every server with that tag) or Single server.
Minimum severityFrom which level on you are notified: info, warning or critical.
ChannelThe bell receives every match; e-mail can be added optionally. (Telegram is prepared in the data model but not yet exposed in the UI.)

A user's rules are merged – one event produces exactly one feed entry, and the channels are the union of all matching rules.

What triggers a notification

  • Monitoring: every status transition of a check (e.g. ok → critical and the all-clear back). A transition's severity is the worse of the two states, so the recovery also reaches subscribers with a "warning" threshold.
  • Security: a new admin user was created.
  • Lifecycle: a server was removed from the inventory, an FRP tunnel was created.
Only what you may see

The hub only delivers a user events for servers they may see – all of them as an admin, otherwise only those assigned to them (see Users & access). "All servers" therefore means "all servers visible to me", not the whole inventory.

E-mail delivery: outbound e-mails are sent via an external SMTP relay (SMTP_HOST/SMTP_PORT/SMTP_USER/SMTP_PASSWORD/SMTP_FROM, see Operations & configuration). Delivery runs asynchronously through an outbox with automatic retry (linear backoff up to NOTIFICATION_MAX_ATTEMPTS, default 5); if no SMTP relay is configured, the bell still works fully.
