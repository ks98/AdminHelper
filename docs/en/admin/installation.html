Admin Guide

Installation

AdminHelper ships as a Docker Compose stack: a TLS gateway (nginx), server, ca-issuer, Postgres, Redis, FRP server (frps), monitoring service and VictoriaMetrics. The gateway is the single public TLS edge — no external reverse proxy is required.

Prerequisites

  • Linux host (Debian 12+, Ubuntu 22.04+, Rocky 9)
  • Docker Engine + Docker Compose v2
  • Public IPv4/IPv6 with DNS A records for srm.example.com (panel) and frp.example.com (tunnel endpoint)
  • Open ports: 443/tcp (web/API), 8444/tcp (enrollment), 7000/tcp (FRP control), 7443/tcp (FRP mTLS), 6000-6999/tcp (visitor ports)

Quick install (one command)

The installer fetches only the runtime files (the self-contained docker-compose.yml + a few ops scripts, not the source tree), brings up the stack, creates the first admin with a one-time enrollment token, and leaves mTLS enforced by default:

bash
curl -fsSL https://raw.githubusercontent.com/ks98/AdminHelper/v0.34.0/scripts/install.sh \
  | bash -s -- --domain srm.example.com
# installs the latest release; pin a specific one with --ref vX.Y.Z.
# prompts for the admin password. Non-interactive: --admin-password … --yes.
# --permissive leaves the stack permissive (no enforced mTLS).
# --reset wipes volumes from a failed attempt (docker compose down -v).

The script prints the credentials: URL, admin login, and an enrollment token. Redeem the token in the desktop client under "Enroll with token" (server URL + token) — the client generates its mTLS cert on-device — then log in normally. Export the browser .p12 afterwards from the desktop's export button. Updates: ./scripts/update.sh inside the created adminhelper/ directory.

Why the token? Under enforced mTLS the data plane :443 (including login) rejects certless clients at the TLS handshake. The very first admin therefore cannot fetch a cert via login; the script mints the token out-of-band over the internal network. The admin onboards follow-up users afterwards (ADR 0003).

Start the Stack (manual)

The official docker-compose.yml lives in the repository root. For a quick test this is enough:

bash
git clone https://github.com/ks98/AdminHelper.git
cd AdminHelper

cp .env.example .env

# Generate secure random secrets (SECRET_KEY, MONITOR_API_KEY,
# POSTGRES_PASSWORD, CA_ROOT_PASSPHRASE).
# Idempotent: leaves any value already set untouched.
./scripts/init-secrets.sh

# Then edit the non-secret values in .env:
#   DOMAIN, ADMIN_PASSWORD, TZ, TRUSTED_PROXIES, ALLOWED_IPS

docker compose pull
docker compose up -d
MONITOR_API_KEY without the init script = dead monitoring UI

Server and monitoring containers share MONITOR_API_KEY as an internal shared secret. If the value in .env stays empty, the monitoring container generates its own random — but the server sends an empty key, so every /api/monitoring/* call ends in a 401. ./scripts/init-secrets.sh prevents this.

Just as important: SECRET_KEY. If it stays at change-me-in-production or empty, the server generates a random value into ./data/.secret_key on first start — it works, but it is not portable between hosts. The init script writes a fixed value into .env so the stack stays reproducibly migratable.

TLS gateway (built-in)

The stack contains its own gateway (nginx) that acts as the single public TLS edge on 443 and forwards internally to the server (plain-HTTP, no host port). server and ca-issuer are reachable only on the compose network — this is what makes the client identity (mTLS header) the gateway sets unforgeable from the outside.

The gateway's TLS certificate is issued by the internal ca-issuer on first start (signed by the internal CA's access intermediate, SAN = DOMAIN). Since this CA is not publicly trusted, browsers show a certificate warning at first — expected; the root CA is pinned in the clients' trust model (see Troubleshooting). An external reverse proxy is not required; if you still want one in front, terminate its own TLS and forward to the gateway on 443 (and add its IP to TRUSTED_PROXIES).

Client certificates (mTLS). The agent and the desktop fetch their client certificate from the internal CA automatically (enrollment, see the developer overview "Unified PKI & mTLS"). For the browser, the desktop client exports a PKCS12 (.p12, password-protected) that you import into the browser's certificate store (step by step under Users & Access → Browser access). The data plane currently runs permissive — a client cert is not yet required; enforcing mTLS (CERT_REQUIRED) is a separate operator step via MTLS_ENFORCE (see Operations & Config).

First Login – Bootstrap Token (manual)

Note: this manual path goes to :443 — under the enforced default (since 0.29.0) it is rejected certless at the handshake. Use scripts/install.sh (Quick install above) or set MTLS_ENFORCE=false for the manual bootstrap, create the admin + enroll, then arm again.

There is no default login anymore. With an empty ADMIN_PASSWORD, the server generates a one-time setup token at data/.bootstrap_token and prints it to the WARNING log:

bash
docker compose logs server | grep -A2 'Setup-Token'

Use the token to create the first admin:

bash
curl -k -X POST https://srm.example.com/api/auth/bootstrap \
  -H 'Content-Type: application/json' \
  -d '{"token":"<TOKEN>","username":"admin","password":"<your-password>"}'

The response contains access_token and refresh_token immediately. The token file is deleted after a successful bootstrap and the endpoint can no longer be called once a user exists (returns 409).

CI / power users: set ADMIN_PASSWORD directly

Setting any value other than admin or empty in .env (e.g. ADMIN_PASSWORD=initialpw) creates an admin user with that password on first start – no bootstrap step needed. Convenient for CI pipelines.

Health check

The server exposes a short JSON status at /api/health. Use this endpoint for monitoring systems or Docker health checks.

Updates

Updates are release-bound: every published release ships a verified runtime bundle (adminhelper-runtime-vX.Y.Z.tar.gz — the docker-compose.yml, all ops scripts, a MANIFEST.sha256 and a VERSION marker). scripts/update.sh resolves a target version, downloads that one asset, verifies it against the release's SHA256SUMS, and swaps the runtime files into place atomically. So whatever a release changes (new/changed/removed scripts, a new compose) lands on the host — "which files belong to release X" lives in the release, not in the script.

bash
# Default: move to the latest published release (prereleases excluded),
# only if newer — no silent downgrade:
./scripts/update.sh

# Pin to a specific version (re-apply / down- or upgrade explicitly):
./scripts/update.sh --ref v0.34.0

# Dry run: print installed vs. available, change nothing:
./scripts/update.sh --check

# Self-heal: no version jump, just re-pull the pinned images:
./scripts/update.sh --redeploy

# More flags: --skip-backup  --with-victoria  --no-rollback  --yes

The installer pins the images in .env to the installed version (e.g. :0.34.0) — so an install stays reproducible and never silently jumps. update.sh lives locally in the adminhelper/ directory after install.

Built to be safe — backup, verification, automatic rollback

Before the swap, update.sh backs up first (data incl. the CA crown jewel) and snapshots the current runtime files (backups/runtime-prev-*.tar.gz). The bundle is checked against SHA256SUMS — which also catches truncated downloads that a plain curl --fail misses. When release signing is armed, update.sh first verifies the minisign signature of SHA256SUMS against a public key pinned in the script (authenticity, not just transport) and aborts on an invalid/missing signature. If the health check fails after deploy, the script auto-rolls-back both the runtime files and the image pins (--no-rollback keeps the broken state for debugging; a data restore stays deliberately manual via restore.sh).

The updater updates itself: if a release ships a newer update.sh, it re-execs into the new version before performing the swap — so future updater improvements reach every host. One-time migration for installs before 0.34.0: their old update.sh doesn't know about the bundle yet; re-fetch it once via the install one-liner (or run a --ref v0.34.0 update), after which it is self-updating.

Uninstall

scripts/uninstall.sh removes a server install completely. By default it is interactive: it asks per category, and only deletes once every question has been answered. Run it from the install directory (where docker-compose.yml lives):

bash
# Interactive — asks per category (stack, volumes, ./data+./certs, .env,
# backups, images):
./scripts/uninstall.sh

# Non-interactive (e.g. inside a script): stack + volumes + ./data/./certs
# + .env go, backups and images are kept:
./scripts/uninstall.sh --yes

# "Really wipe everything" — images and backups too (--rmi/--purge-backups also
# pre-set the matching question to YES):
./scripts/uninstall.sh --rmi --purge-backups --yes

Asked/removed individually: every container of the compose project and the network, all named volumes (including ca-pki with the root CA, postgres-data and victoria-data), the host bind mounts ./data and ./certs, the secrets file .env, and optionally ./backups/ and the Docker images. Since ./data/./certs are owned by the in-container user (uid 10001), the script deletes them as root in a throwaway container — without requiring sudo.

Irreversible — and the root CA goes with it

Dropping ca-pki destroys the internal root CA: already-enrolled agents/clients are no longer trusted afterwards. If you intend to rebuild on the same host and keep that trust, run ./scripts/backup.sh first and restore with restore.sh.

Protected by default: ./backups/ (it holds the CA + DB dumps) is kept and the images stay — only --purge-backups / --rmi remove those too. The runtime files themselves (compose + scripts/) are not deleted; the script prints the exact rm -rf command for the rest at the end.
