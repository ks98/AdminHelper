Admin Guide

Installation

AdminHelper ships as a Docker Compose stack: server, Postgres, Redis, FRP server (frps), monitoring service and VictoriaMetrics. This guide walks through a production setup behind a TLS-terminating reverse proxy.

Prerequisites

  • Linux host (Debian 12+, Ubuntu 22.04+, Rocky 9)
  • Docker Engine + Docker Compose v2
  • Public IPv4/IPv6 with DNS A records for srm.example.com (panel) and frp.example.com (tunnel endpoint)
  • Open ports: 443/tcp (web), 7000/tcp (FRP control), 7443/tcp (FRP mTLS), 6000-6999/tcp (visitor ports)

Environment Variables

VariableDefaultPurpose
DOMAINlocalhostMain FQDN of the server. Used in self-signed certificates and links.
SECRET_KEYautoJWT signing key. Set it manually or let it auto-generate (persisted to data/.secret_key).
ADMIN_PASSWORDemptyOptional. Empty/'admin' triggers the bootstrap-token flow (see above). A non-default value creates an admin directly (CI/dev).
POSTGRES_PASSWORDautoGenerated by ./scripts/init-secrets.sh. Postgres refuses to start if empty.
DATABASE_URLfrom composepostgresql+psycopg://adminhelper:$POSTGRES_PASSWORD@postgres:5432/adminhelper – set automatically by the compose service.
DATA_DIR/app/dataData directory for bootstrap token, .secret_key, FRP config, certificates (the DB lives in Postgres).
REDIS_URLemptyRedis URL for rate-limiting. Without Redis an in-memory fallback is used (only safe for single-worker). Compose sets it to redis://redis:6379/0.
TRUSTED_PROXIESemptyComma-separated CIDRs of the trusted reverse proxies. Only these IPs may send accepted X-Forwarded-For headers.
TRUST_PROXY_HEADERSfalseLegacy switch, trusts all proxy headers. Only set it when TRUSTED_PROXIES is not usable.
ALLOWED_IPSemptyWhitelist of IPs/CIDRs allowed to access the web panel. Empty = no filter.
MONITOR_SERVICE_URLhttp://monitoring:8080URL of the monitoring service (container-internal).
MONITOR_API_KEYemptyShared secret between server and monitoring. Must be identical in both containers.
VISITOR_PORT_START / _END6000 / 6999Port range for STCP visitor tunnels in the desktop client.
FRP_BIND_PORT7000TCP port for the FRP bind (control channel).
FRP_HTTPS_PORT7443HTTPS port for the FRP tunnel.

Start the Stack

The official docker-compose.yml lives in the repository root. For a quick test this is enough:

bash
git clone https://github.com/ks98/AdminHelper.git
cd AdminHelper

cp .env.example .env

# Generate secure random secrets (SECRET_KEY + MONITOR_API_KEY).
# Idempotent: leaves any value already set untouched.
./scripts/init-secrets.sh

# Then edit the non-secret values in .env:
#   DOMAIN, ADMIN_PASSWORD, TZ, TRUSTED_PROXIES, ALLOWED_IPS

docker compose pull
docker compose up -d
MONITOR_API_KEY without the init script = dead monitoring UI

Server and monitoring containers share MONITOR_API_KEY as an internal shared secret. If the value in .env stays empty, the monitoring container generates its own random — but the server sends an empty key, so every /api/monitoring/* call ends in a 401. ./scripts/init-secrets.sh prevents this.

Just as important: SECRET_KEY. If it stays at change-me-in-production or empty, the server generates a random value into ./data/.secret_key on first start — it works, but it is not portable between hosts. The init script writes a fixed value into .env so the stack stays reproducibly migratable.

Reverse Proxy (nginx)

nginx
server {
    listen 443 ssl http2;
    server_name srm.example.com;

    ssl_certificate     /etc/letsencrypt/live/srm.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/srm.example.com/privkey.pem;

    client_max_body_size 100M;

    location / {
        proxy_pass         http://127.0.0.1:8443;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade           $http_upgrade;
        proxy_set_header   Connection        "upgrade";
    }
}

First Login – Bootstrap Token

There is no default login anymore. With an empty ADMIN_PASSWORD, the server generates a one-time setup token at data/.bootstrap_token and prints it to the WARNING log:

bash
docker compose logs server | grep -A2 'Setup-Token'

Use the token to create the first admin:

bash
curl -k -X POST https://srm.example.com/api/auth/bootstrap \
  -H 'Content-Type: application/json' \
  -d '{"token":"<TOKEN>","username":"admin","password":"<your-password>"}'

The response contains access_token and refresh_token immediately. The token file is deleted after a successful bootstrap and the endpoint can no longer be called once a user exists (returns 409).

CI / power users: set ADMIN_PASSWORD directly

Setting any value other than admin or empty in .env (e.g. ADMIN_PASSWORD=initialpw) creates an admin user with that password on first start – no bootstrap step needed. Convenient for CI pipelines.

Health check

The server exposes a short JSON status at /api/health. Use this endpoint for monitoring systems or Docker health checks.
