Admin Guide

Installation

AdminHelper ships as a Docker Compose stack with four services: server, redis, frps, monitoring + victoria. This guide walks through a production setup behind a TLS-terminating reverse proxy.

Prerequisites

  • Linux host (Debian 12+, Ubuntu 22.04+, Rocky 9)
  • Docker Engine + Docker Compose v2
  • Public IPv4/IPv6 with DNS A records for srm.example.com (panel) and frp.example.com (tunnel endpoint)
  • Open ports: 443/tcp (web), 7000/tcp (FRP control), 7443/tcp (FRP mTLS), 6000-6999/tcp (visitor ports)

Environment Variables

VariableDefaultPurpose
SECRET_KEYautoJWT + cookie signing. Auto-generates on first start and persists to /data/.secret_key.
ADMIN_PASSWORDemptyOptional. Empty/'admin' triggers the bootstrap-token flow. A non-default value creates an admin user directly (CI/dev).
POSTGRES_PASSWORDautoGenerated by ./scripts/init-secrets.sh. Postgres refuses to start if empty.
DATABASE_URLfrom composepostgresql+psycopg://adminhelper:$POSTGRES_PASSWORD@postgres:5432/adminhelper – set automatically by the compose service.
REDIS_URLredis://redis:6379/0Rate-limit storage.
TRUSTED_PROXIES127.0.0.1Reverse-proxy IPs whose X-Forwarded-For is trusted.
ALLOWED_IPSemptyOptional whitelist for admin endpoints.
MONITOR_API_KEYautoShared secret between server and monitoring service.
VISITOR_PORT_START6000First dynamically assigned FRP visitor port.
VISITOR_PORT_END6999Last visitor port in the pool.
FRP_DOMAIN—Public hostname for HTTP tunnels.
LOG_LEVELINFOApplies to all four loggers (app, monitoring, frp, scheduler).
CORS_ORIGINS*Comma-separated list, default allows any origin.

Generate .env

bash
git clone https://git.nevondo.com/ks98/adminhelper.git /srv/adminhelper
cd /srv/adminhelper

cp .env.example .env

# Generate secure random secrets (SECRET_KEY + MONITOR_API_KEY).
# Idempotent: leaves any value already set untouched.
./scripts/init-secrets.sh

# Then edit .env for the non-secret values:
#   DOMAIN, ADMIN_PASSWORD, TZ, TRUSTED_PROXIES, ALLOWED_IPS, FRP_DOMAIN
Why init-secrets.sh matters

Server and monitoring containers share MONITOR_API_KEY as an internal secret. If the value in .env stays empty, the monitoring container generates its own random — but the server sends an empty key, so every /api/monitoring/* call returns 401 and the monitoring page looks broken. The init script fixes this.

Reverse Proxy (nginx)

nginx
server {
    listen 443 ssl http2;
    server_name srm.example.com;

    ssl_certificate     /etc/letsencrypt/live/srm.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/srm.example.com/privkey.pem;

    client_max_body_size 100M;

    location / {
        proxy_pass         http://127.0.0.1:8443;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade           $http_upgrade;
        proxy_set_header   Connection        "upgrade";
    }
}

First Login – Bootstrap Token

There is no default login anymore. With an empty ADMIN_PASSWORD, the server generates a one-time setup token at data/.bootstrap_token and prints it to the WARNING log:

bash
docker compose logs server | grep -A2 'Setup-Token'

Use the token to create the first admin:

bash
curl -k -X POST https://srm.example.com/api/auth/bootstrap \
  -H 'Content-Type: application/json' \
  -d '{"token":"<TOKEN>","username":"admin","password":"<your-password>"}'

The response contains access_token and refresh_token immediately. The token file is deleted after a successful bootstrap and the endpoint can no longer be called once a user exists (returns 409).

CI / power users: set ADMIN_PASSWORD directly

Setting any value other than admin or empty in .env (e.g. ADMIN_PASSWORD=initialpw) creates an admin user with that password on first start – no bootstrap step needed. Convenient for CI pipelines.
