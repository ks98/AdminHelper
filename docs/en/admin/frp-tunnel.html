FRP Tunnel

AdminHelper runs a managed FRP server with its own mTLS PKI. Agents authenticate with client certificates; RDP/SSH/VNC tunnels get dynamically assigned visitor ports.

PKI

  • Root CA: /data/frp-ca/ca.crt + ca.key, 10-year lifetime.
  • Client certificates: 1-year lifetime, auto-rotated by the agent when less than 30 days remain.
  • CA key is never transmitted – signing happens inside the server container.

Tunnel Types

TypeDescriptionUse case
tcpFRP-assigned remote port on the server, visible from the public network.RDP/SSH/VNC over untrusted networks – treat this as public-facing.
httpHTTP-reverse-proxy tunnel on FRP_DOMAIN, subdomain-routed.Web UIs behind NAT (e.g. IPMI, home-automation).
stcpSecret-key gated, only another visitor with matching sk can connect.Private tunnels – zero public port exposure.

Visitor Ports

The desktop client uses stcp whenever possible. When the tunnel is opened, the server assigns a free port from the VISITOR_PORT_START..END pool (default 6000–6999). The port is released as soon as the client disconnects.

Certificate Rotation

The agent checks its cert daily:

  • If valid-to is more than 30 days in the future → no-op.
  • Less than 30 days → request a new cert via POST /api/frp/client-cert/rotate.
  • Expired → agent falls back to provisioning flow; re-provisioning required.
Backup the CA

Losing /data/frp-ca/ means every agent must re-provision. Back it up encrypted, off-site. The DB alone is useless without this directory.
