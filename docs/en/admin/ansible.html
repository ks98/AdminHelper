Ansible

Ansible management separates central maintenance from local execution: playbooks are stored and versioned in the web panel, but they run via the desktop client using a locally installed ansible-playbook. The server itself runs no playbooks.

Playbook CRUD in the Web Panel

Playbooks are managed via /api/ansible/playbooks — all write operations are admin-only. Each playbook has metadata in the database (table ansible_playbooks) and a YAML file on disk at DATA_DIR/ansible/playbooks/<id>/<filename>.

FieldMeaning
nameDisplay name in the panel and desktop client
filenameName of the YAML file — must end in .yml or .yaml, no path separators
descriptionOptional description
tagsList of tags for filtering/search
contentThe YAML content — written to disk, not stored in the DB

On create (POST) and on edit (PUT, if content is included) the server validates the YAML via yaml.safe_load; invalid YAML returns 422. If the filename changes, the server renames the file on disk or deletes the old one. On delete (DELETE) the entire <id>/ directory is removed.

HTTP
GET    /api/ansible/playbooks              # List all playbooks
POST   /api/ansible/playbooks              # Create playbook (admin)
GET    /api/ansible/playbooks/{id}         # Playbook metadata
GET    /api/ansible/playbooks/{id}/content # Fetch YAML content
PUT    /api/ansible/playbooks/{id}         # Update playbook (admin)
DELETE /api/ansible/playbooks/{id}         # Delete playbook (admin)

CRUD operations fire hooks: playbook.created, playbook.updated and playbook.deleted (payload: id + name). For hook configuration details: Hooks & Events.

3-Step Workflow (Desktop)

The desktop client guides you through a three-step wizard. Step 2 only becomes active once a playbook is selected; step 3 only once at least one target server is also selected.

  1. Select playbook — list of all playbooks maintained on the server, with name, description, filename and tags.
  2. Select target servers — two modes: Servers (individual checkbox selection) or Tags (one chip per tag selects all servers carrying that tag).
  3. Run — summary (playbook + server count), then launch via ansible-playbook in a native terminal.

On run, the client fetches the YAML content fresh from the server (/{id}/content), writes it to a temporary file, generates the inventory and starts the run. Servers and playbooks are sourced from the server web interface — nothing is maintained locally.

Inventory Generation

The desktop client automatically builds an INI-format inventory file from the selected servers in a temporary directory. All selected hosts go under [all]; in addition, one group per server tag is created (sorted alphabetically) listing the matching hosts.

INI
[all]
web01.example.com
db01.example.com

[database]
db01.example.com

[web]
web01.example.com

Groups are derived directly from the servers' tags — the same tags set on the server in the web panel that drive selection in tag mode. Hostnames are the servers' hostname fields.

Requirements

  • ansible-playbook must be installed on the desktop machine and on the PATH. If missing, the run aborts with an error message.
  • Target servers must be created in the server web interface under Servers — only hosts maintained there appear in the desktop wizard.
  • Playbook maintenance (create/edit/delete) requires an admin session.
SSH reachability

Execution runs on the desktop machine against the hostnames from the inventory. SSH access, keys and optionally ansible_user are handled as usual via your Ansible/SSH configuration — AdminHelper only generates the inventory and playbook file.
