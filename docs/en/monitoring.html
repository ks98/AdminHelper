Monitoring

Monitoring and Alerting

AdminHelper Monitoring watches servers and services using pull- and push-based checks, stores metrics in VictoriaMetrics, and alerts via webhook or e-mail.

Check Types Alerting

1) Architecture

The monitoring system consists of four components that work together to monitor servers and services:

Monitoring Service

FastAPI application on Port 8080. Contains the check engine, the scheduler for periodic checks, and the alerting system. Receives push data from the agent and runs pull checks independently.

VictoriaMetrics

Time-series database on Port 8428. Stores all metrics with a default retention of 90 days. Compatible with PromQL queries.

AdminHelper Agent

Push-based metric collection on monitored servers. Collects CPU, RAM, disks, services, and temperatures, then sends them via HTTPS to the monitoring service.

AdminHelper Server

Provides a proxy through which the web UI can access the monitoring API. The monitoring service is not directly reachable from the outside.

Data flow: Agent → Monitoring Service → VictoriaMetrics + Alerting. Pull checks are executed directly by the monitoring service.

2) Check Types

AdminHelper supports 10 different check types, divided into pull-based (executed by the monitoring service) and push-based (reported by the agent) checks.

Pull-based Checks

ping — ICMP Ping

Checks the reachability of a host via ICMP ping.

Config: target (hostname/IP), timeout (seconds, default: 5)

Metric: monitor_ping_rtt_ms

tcp — TCP Port Check

Checks whether a TCP port is reachable and measures the connection time.

Config: host, port, timeout (seconds, default: 5)

Metric: monitor_tcp_connect_ms

http — HTTP/HTTPS Check

Performs an HTTP request and checks the status code and response time.

Config: url, method (GET/POST), timeout, status_code (expected code)

Metrics: monitor_http_status_code, monitor_http_response_ms

agent_ping — Agent Reachability

Checks whether the agent has sent data within the last 5 minutes. Does not require any special configuration parameters.

Config: no special parameters

Push-based Checks

agent_resources — CPU, Memory, Disk

Monitors system resources with configurable thresholds. Per-disk and per-sensor thresholds are supported.

Config: cpu_warning (%), cpu_critical (%), memory_warning (%), memory_critical (%)

Metrics: monitor_cpu_percent, monitor_memory_percent, monitor_disk_percent

service_process — Service Monitoring

Checks the status of configured services. Supports systemd units (Linux) and Windows services.

Config: services (array of service names)

Status: running, stopped, failed

docker_health — Docker Container

Checks the health status of running Docker containers. Containers are automatically detected by the agent.

Config: automatic detection

proxmox_backup — Proxmox Backup Server

Checks the last backup timestamp on a Proxmox Backup Server.

Config: max_backup_age_hours, exclude_vmids, exclude_stopped

zfs_health — ZFS Pool Status

Checks ZFS pool capacity and health status.

Config: pool names, warning_percent, critical_percent

smart_health — Disk S.M.A.R.T.

Monitors S.M.A.R.T. values of hard drives. Automatic detection by the agent.

Checks: Reallocated Sectors, Pending Sectors, Temperature

3) Check Configuration

Each check can be configured individually. The most important parameters are:

Parameter Description
Interval Predefined: 1m, 5m, 15m, 30m, 1h, 6h, 12h, 24h or cron expressions (5 fields)
Severity warning or critical
Consecutive Fails Number of consecutive failures before the status changes (e.g. 3 = alert is triggered only after 3 failures)
Enabled Checks can be enabled or disabled via a flag

Example: Creating a check via API

POST /api/monitoring/checks
Content-Type: application/json

{
  "name": "Webserver reachable",
  "check_type": "http",
  "server_id": "srv-web-prod-01",
  "interval": "5m",
  "severity": "critical",
  "consecutive_fails": 3,
  "enabled": true,
  "config": {
    "url": "https://example.com/health",
    "method": "GET",
    "timeout": 10,
    "status_code": 200
  }
}
Tip: Use consecutive_fails to filter out short-lived outages (flapping). A value of 3 with a 5-minute interval means an alert is only triggered after 15 minutes of sustained failure.

4) Alerting

The alerting system sends notifications on status changes through configurable channels.

Alert Rules

Parameter Description
match_severity Only alert on a specific severity (warning, critical)
match_server_id Only for a specific server (optional, empty = all servers)
cooldown_minutes Minimum interval between alerts (e.g. 30 = at most every 30 minutes)
Recovery Alerts Sent automatically when status returns to "ok" (ignores cooldown)

Channel: Webhook

Sends a JSON payload to a configurable URL (timeout: 10 seconds). 

// Configuration
{
  "url": "https://hooks.slack.com/services/T00/B00/xxxx"
}

// Payload (sent automatically)
{
  "alert_rule": "critical-alerts",
  "check_name": "Webserver reachable",
  "server_id": "srv-web-prod-01",
  "severity": "critical",
  "old_status": "ok",
  "new_status": "critical",
  "is_recovery": false,
  "subject": "[AdminHelper Monitor] CRITICAL: Webserver reachable",
  "text": "Check 'Webserver reachable' on srv-web-prod-01 is CRITICAL."
}

Channel: E-Mail (SMTP)

Sends e-mail notifications via SMTP. Port 587 enables STARTTLS automatically. 

// Alert channel configuration
{
  "recipients": ["admin@example.com", "ops@example.com"],
  "smtp_host": "smtp.example.com",
  "smtp_port": 587,
  "smtp_user": "alerts@example.com",
  "smtp_password": "secret"
}
Note: SMTP settings can alternatively be set globally via environment variables. The channel configuration overrides the global values. Subject format: [AdminHelper Monitor] CRITICAL: Checkname

5) Environment Variables

The monitoring service is configured using the following environment variables:

Variable Default Description
VICTORIA_METRICS_URL http://victoria:8428 VictoriaMetrics API URL
MONITOR_API_KEY (empty) Internal API key (shared with server)
DATA_DIR /app/data Directory for SQLite database
SMTP_HOST (empty) SMTP server for e-mail alerts
SMTP_PORT 587 SMTP port (587 = STARTTLS)
SMTP_USER (empty) SMTP username
SMTP_PASSWORD (empty) SMTP password
SMTP_FROM adminhelper@localhost Sender address
TZ Europe/Berlin Time zone

6) Templates

Monitoring templates are reusable collections of check and alert definitions that can be applied to multiple servers.

Template Contents

A template contains two arrays: check_definitions (check templates) and alert_definitions (alert rule templates).

Assignment and Sync

When assigned to a server, checks and alert rules are created automatically. Changes to the template can be synchronized to assigned servers. 

// Example: Template structure
{
  "name": "Standard Linux Server",
  "check_definitions": [
    {
      "check_type": "agent_resources",
      "interval": "5m",
      "severity": "critical",
      "config": {
        "cpu_warning": 80,
        "cpu_critical": 95,
        "memory_warning": 85,
        "memory_critical": 95
      }
    },
    {
      "check_type": "agent_ping",
      "interval": "5m",
      "severity": "critical",
      "consecutive_fails": 3
    }
  ],
  "alert_definitions": [
    {
      "channel_type": "webhook",
      "match_severity": "critical",
      "cooldown_minutes": 30,
      "config": {
        "url": "https://hooks.slack.com/services/..."
      }
    }
  ]
}

7) Agent Keys

Each monitored server requires its own agent API key for authentication against the monitoring service.

Property Details
Generation Via web UI or POST /admin/agent-keys
Display Key is shown in plain text only once upon generation
Storage Stored in the database as a SHA-256 hash
Usage Agent sends the key in the X-API-Key header with every push
Important: The API key is shown in plain text only once. Note it down immediately or configure the agent right after generation. A lost key cannot be recovered — a new one must be generated.

8) VictoriaMetrics

VictoriaMetrics serves as the time-series database for all monitoring metrics. Data is written via the InfluxDB Line Protocol.

Property Value
Protocol InfluxDB Line Protocol
Retention 90 days (default)
Query Language PromQL

For each check, at least two metrics are written:

  • monitor_check_status — 0 = ok, 1 = warning, 2 = critical, 3 = unknown
  • monitor_check_duration_ms — Execution duration of the check in milliseconds

PromQL queries are available via the endpoint /api/monitoring/checks/{id}/metrics. 

# Example PromQL: Average check duration over the last hour
avg_over_time(monitor_check_duration_ms{check_id="42"}[1h])

# All critical checks
monitor_check_status == 2

9) Docker Compose Setup

The monitoring-relevant parts of the docker-compose.yml: 

services:
  monitoring:
    image: adminhelper-monitoring:latest
    ports:
      - "8480:8080"
    environment:
      - VICTORIA_METRICS_URL=http://victoria:8428
      - MONITOR_API_KEY=${MONITOR_API_KEY}
      - DATA_DIR=/app/data
      - SMTP_HOST=${SMTP_HOST:-}
      - SMTP_PORT=${SMTP_PORT:-587}
      - SMTP_USER=${SMTP_USER:-}
      - SMTP_PASSWORD=${SMTP_PASSWORD:-}
      - SMTP_FROM=${SMTP_FROM:-adminhelper@localhost}
      - TZ=Europe/Berlin
    volumes:
      - monitoring-data:/app/data
    depends_on:
      - victoria
    restart: unless-stopped

  victoria:
    image: victoriametrics/victoria-metrics:v1.101.0
    command:
      - "-retentionPeriod=90d"
      - "-storageDataPath=/victoria-data"
    volumes:
      - victoria-data:/victoria-data
    restart: unless-stopped

volumes:
  monitoring-data:
  victoria-data:
Note: The monitoring service listens internally on port 8080 and is mapped externally to port 8480. The AdminHelper server forwards requests as a proxy to the monitoring service.
