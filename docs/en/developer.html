Developer Guide

Architecture, implementation, and extension workflow

This page documents the engineering side of the project: local setup, source layout, API behavior, webhook scripting, security controls, and CI/CD.

Start with setup Jump to API

1) Development Setup

Area Required Notes
Client (Tauri) Rust stable, cargo tauri Linux needs WebView/GTK packages and xfreerdp3 or xfreerdp for RDP.
Server Python 3.12+, pip FastAPI + SQLAlchemy + SQLite.
Extension Chrome/Chromium Manifest V3 extension, loaded unpacked for local testing.
Agent Go 1.24+, make Unified Go agent for Linux + Windows (FRP sync + monitoring).
Monitoring Python 3.12+, pip FastAPI monitoring service, started via Docker Compose.

Install Tauri CLI

cargo install tauri-cli --locked --version '^2'

Run server locally (without Docker)

cd server
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8080

4) Client Architecture (Tauri)

Module File Responsibility
Command bridge commands.rs Exposes load/save/open/sync/password commands to frontend.
Connection openers connection/* SSH/RDP/Web launch behavior per OS.
Local storage storage.rs Stores connections.json and settings.json in app data folder.
Sync sync.rs Validates and fetches sync source; HTTPS only.
Password handling password.rs RDP only; keyring/Credential Manager integration.
Ansible ansible.rs Generate INI inventory from servers, store playbook, launch ansible-playbook in terminal.
Important: local CRUD is blocked in Sync mode by the frontend.

5) Server Architecture (FastAPI)

Entry point: server/app/main.py. At startup, DB tables are created and an initial admin user is created if the DB is empty.

Modular structure under app/core/ (config, auth, DB, middleware, events) and app/modules/:

Module Path Details
Auth core/auth.py, modules/users/auth_router.py JWT bearer auth, bcrypt with SHA-256 pre-hash.
Users modules/users/ User management, login/logout, token refresh.
Connections modules/connections/ Connection CRUD (SQLite). Auto-migration from JSON to SQL at startup.
Servers modules/servers/ Server registry with tags, host, port.
FRP modules/frp/ Hierarchical: config, tunnel, generate, status, PKI, provision (6 sub-routers).
Hooks modules/hooks/ Webhook, event, schedule (APScheduler). Script runner with full Python access (admin-only).
API Keys modules/api_keys/ API key management, SHA-256 hashed in DB.
Ansible modules/ansible/ Playbook CRUD with YAML validation. Files stored under DATA_DIR/ansible/playbooks/.
Monitoring Proxy modules/monitoring_proxy.py Reverse proxy to the monitoring service for the web UI.
IP Filtering core/middleware.py IP/CIDR allowlist with optional trusted proxy support.

6) Extension Architecture

  • background.js: refresh alarm every 5 minutes, fetch + cache + badge update.
  • popup.js: setup flow, search, list/tag views, open tab actions.
  • options.js: settings + connection test.
  • chrome.storage.sync for config and chrome.storage.local for cached connection data.

7) API and Authentication Behavior

Two auth modes exist: JWT bearer for users and API key via X-API-Key header or api_key query parameter.

Endpoint Access Notes
POST /api/auth/login public Returns access token.
GET /api/connections JWT or API key (read/read_write) Returns all connections.
POST/PUT /api/connections Admin JWT or API key read_write Create/update; UUID is generated on create.
DELETE /api/connections/{id} Admin JWT only Delete is stricter than write key mode.
/api/users, /api/api-keys, /api/hooks Admin JWT only System administration domain.

8) Hook Scripting

All hooks use the same script runner (server/app/script_runner.py) with full Python builtins including import (admin-only). The context injected into the script depends on the hook type.

Common variables (all types)

ObjectTypeUse
load_connections() function Read all connections from JSON storage.
save_connections(list) function Persist modified connection list.
uuid4() function Create a new UUID string.
result, logs, log(msg) dict/list/function Return value and runtime logging.
import ... statement All Python modules available: requests, json, re, …

Type-specific context

TypeExtra variablesTrigger
webhook payload, headers, params POST /api/hooks/trigger/<token>
event event_type (str), event_data (dict) Internal: connection.*/user.*/server.startup
schedule triggered_at (ISO str), last_run (ISO str | None) APScheduler background thread

Example: Webhook – create or update a connection

Repository file: docs/examples/webhook_add_connection.py 

connections = load_connections()
incoming = payload.get("connection")
# normalize + validate + upsert
save_connections(connections)
result["mode"] = mode
result["connectionId"] = connection_id

Example: Event Hook – notify on changes

import requests

msg = f"[SRM] {event_type}: {event_data.get('name', '?')} ({event_data.get('kind', '?')})"
requests.post("https://chat.example.com/webhooks/notify", json={"text": msg}, timeout=5)
result["notified"] = True

Example: Scheduled Hook – daily cleanup

# interval: 24h
connections = load_connections()
seen, unique = set(), []
for c in connections:
    key = (c.get("kind"), c.get("host"), c.get("port"))
    if key not in seen:
        seen.add(key)
        unique.append(c)
removed = len(connections) - len(unique)
if removed:
    save_connections(unique)
result["removed"] = removed
log(f"Cleaned up at {triggered_at}: removed {removed} duplicate(s)")
Execution model: Webhook scripts run synchronously in the request thread — always set timeouts on HTTP calls. Event and scheduled hooks run in daemon threads; exceptions do not affect server operations.

  "trustCert": false,
  "lastUsed": "2026-01-27T10:39:21.574Z"
}

Client settings

  • mode: local | sync
  • url: sync source URL (HTTPS)
  • intervalMinutes: 1..1440
  • storePasswords: local RDP keychain usage
  • allowSelfSignedCerts: optional TLS cert bypass for sync requests

10) Security Controls in Code

  • Passwords are not stored inside connection JSON; RDP credentials are optional and local-only.
  • API keys are stored as SHA-256 hashes on server side.
  • IP filtering supports CIDR and trusted proxy chains.
  • Client sync accepts HTTPS URLs only.
  • Hook scripts run with full Python builtins (including import) inside an isolated namespace with only injected storage functions exposed.
Recommendation: in proxy setups, set TRUSTED_PROXIES instead of enabling global TRUST_PROXY_HEADERS=true.

11) CI/CD Pipeline (.gitlab-ci.yml)

  • docker: builds and pushes server image to registry.
  • download: fetches frpc binary (v0.61.1).
  • build: creates in parallel:
    • Desktop: DEB + RPM + Linux binary (Debian Trixie)
    • Go Agent: Linux + Windows binaries, DEB + RPM packages
  • publish: DEB publishing via Aptly API script (on tags).

Publishing script: desktop/scripts/ci_publish_via_api.sh (requires mTLS and Aptly environment variables).

12) Updating

Update each component in a controlled, repeatable sequence. Server updates are image-based; client updates are build/distribution based.

Update server

docker compose pull
docker compose up -d

Create a backup of data/, certs/, and .env first.

Update client

cd desktop/src-tauri
cargo tauri build

Distribute the new build and recheck runtime dependencies (for example FreeRDP on Linux).

Recommendation: verify sync, auth, and webhook behavior after every version update.

13) Typical Extension Workflow

  1. Update JS model in desktop/src/connectionModel.js or settingsModel.js.
  2. Update Rust structs in desktop/src-tauri/src/models.rs.
  3. Expose command in commands.rs and register in main.rs.
  4. Adjust server schema/routes if API contract changes.
  5. Validate local mode and sync mode behavior end-to-end.

14) Troubleshooting

Sync fails

Client requires HTTPS. For self-signed certs, enable the matching client option.

RDP launch fails on Linux

Check xfreerdp3 / xfreerdp availability in PATH.

Server login issues

Initial ADMIN_PASSWORD only applies when DB is empty.

Unexpected 403

Review ALLOWED_IPS and TRUSTED_PROXIES.
