Go Agent

Unified Go agent under apps/agent/. Replaces the legacy shell + Python agents and merges monitoring, FRP-client setup, and service installer into a single binary that runs on Linux (systemd) and Windows (service manager) alike.

Project Layout

tree
apps/agent/
├── cmd/adminhelper-agent/
│   ├── main.go             Cobra root with five subcommand groups
│   ├── handlers.go         Subcommand implementations
│   └── run.go              standalone mode without systemd
├── internal/
│   ├── config/             paths_linux.go / paths_windows.go, key=value parser
│   ├── provision/          activate via /api/servers/{id}/provision/activate, orchestrates frpc.Apply + monitor.Init

├── deb/                    Debian control files
├── rpm/                    Spec file

├── build-rpm.sh

Cobra Subcommands

Six top-level commands, three of them with sub-subcommands:

CommandPurpose
versionEmit version + git commit (set via -ldflags).
runStandalone mode: run one full cycle (monitor push + FRP sync).
provisionInitial setup: redeem token against /api/servers/{id}/provision/activate, install server API key + optional monitor key + optional FRP bundle in a single call.
frpc syncPeriodic config sync via hash compare; only writes when the server-side hash changed.
monitor initCreate the monitor config (URL, API key, services list).
monitor pushOne metric collection + push run.
service install / uninstallInstall/remove the systemd unit (Linux) or Windows service (SCM).

Configuration

Two config files in key=value format (quotes allowed). OS paths:

  • Linux: /etc/frp/frp.conf and /etc/adminhelper/monitor.conf
  • Windows: under %ProgramData%\adminhelper\

FrpcConfig fields: ADMINHELPER_URL, API_KEY, SERVER_ID, CACERT, INSECURE. MonitorConfig additionally carries MONITOR_URL and SERVICES (comma-separated).

Monitor Collector

Central collection in internal/monitor/collector.go. Every collector returns map[string]any, which is JSON-serialised on push:

  • collectCPU() – load and per-core utilization
  • collectMemory() – total / used / free / swap
  • collectLoad() – load_1m, load_5m, load_15m
  • collectDisks() – per mountpoint with FS type, usage, inodes
  • collectUptime()
  • collectTemperatures() – lm-sensors / Windows WMI
  • collectServiceHealth() + collectWatchedServices(names)
  • collectDocker() – container list via the Docker socket
  • collectZFS() – zpool status -p
  • collectProxmox() – pveversion + API (PVE hosts only)
  • collectSmart() – smartctl --scan-open + --all --json=cv; ATA and NVMe paths parsed separately

All data is shipped under the resources JSON key in POST /agent/{server_id}/report at the monitoring service.

Platform Splits: Linux vs. Windows

Collector, service activation, and paths are separated via Go build tags (//go:build linux / //go:build windows). The table below lists the concrete per-function behaviour on each platform.

FunctionLinuxWindows
collectServiceHealth()
services_*.go		 systemctl list-units + list-unit-files + list-units --state=failed; populates failed, enabled_inactive and all_services (with active_state / enabled_state). sc query state= all; Windows state is mapped to the systemd format (RUNNING→active, STOPPED/PAUSED→inactive, *_PENDING→activating). failed stays empty, enabled_state is always unknown.
collectWatchedServices(names)
services_*.go		 systemctl is-active; PID via systemctl show -p MainPID. sc query (search for RUNNING); PID via sc queryex.
collectTemperatures()
temperatures.go (shared)		 A single shared file with no build tag: reads gopsutil/v4/sensors.SensorsTemperatures(). Identical code on both platforms; gopsutil reads hwmon/lm-sensors on Linux and WMI on Windows. Sensors with temperature ≤ 0 are dropped; with no sensors it returns nil (not an error).
collectLoad()
collector.go (shared)		 load.Avg() yields load_1m/load_5m/load_15m. Returns nil (no load average) — guarded by runtime.GOOS == "windows".
collectDisks()
collector.go (shared)		 Shared code via gopsutil/disk. The filter lists skipFSTypes (squashfs, tmpfs, overlay, proc…) and skipMountPrefixes (/sys, /proc, /dev/, /snap…) are Linux-specific; on Windows they never match and filter nothing.
enableMonitorService()
platform_*.go		 systemctl enable --now adminhelper-agent.timer. No-op: prints a hint to run adminhelper-agent service install (SCM integration left as a TODO).
service.Install()
service_*.go		 Writes .service (Type=oneshot, ExecStart=<exe> run) + .timer (OnBootSec=60, OnUnitActiveSec=300, Persistent=true) to /etc/systemd/system, then daemon-reload + enable --now on the timer. Unit name adminhelper-agent. Copies the binary to %ProgramFiles%\AdminHelper\, registers via sc create (binPath=<exe> run, start= auto, obj= LocalSystem), sets recovery via sc failure (restart after 60 s) and starts with sc start. Service name AdminHelper-Agent — a long-running service, not a timer.
service.Uninstall()
service_*.go		 systemctl stop/disable for timer and service, removes the unit files, daemon-reload. sc stop + sc delete on AdminHelper-Agent.
FrpDir()
paths_*.go		 /etc/frp %ProgramData%\AdminHelper\frp
MonitorDir()
paths_*.go		 /etc/adminhelper %ProgramData%\AdminHelper

systemd Timer with Jitter

Instead of cron, the agent uses systemd with OnBootSec=60, OnUnitActiveSec=300 (5 min interval) and RandomizedDelaySec=30. The jitter spreads load across the 5-minute window and avoids thundering-herd requests at the server.

Build

bash
cd apps/agent
make build-linux       # bin/adminhelper-agent       (ELF amd64)
make build-windows     # bin/adminhelper-agent.exe   (PE amd64)

VERSION=0.24.0 bash build-deb.sh
VERSION=0.24.0 bash build-rpm.sh

The DEB package installs /usr/local/bin/adminhelper-agent plus /usr/bin/frpc (downloaded in CI) and registers the systemd unit + timer.

DEB/RPM Replaces

The DEB/RPM packages carry Replaces + Conflicts entries for the legacy packages srm-frpc-client, srm-monitor-agent and srm-agent. A single apt install adminhelper-agent migrates clean installs automatically.
