Desktop Client

The Desktop Client is a Tauri 2 wrapper around a Svelte 5 SPA. The Rust backend (apps/desktop/src-tauri/) exposes typed commands; the frontend (apps/desktop/ui/) consumes them through a 1:1 bridge. Since v0.32.0, in server mode the client is also the operational cockpit: the Infrastructure hub manages the server inventory and, per server, connections, FRP tunnels, monitoring checks, and provisioning; plus the fleet-wide Monitoring and Ansible pages. Operational writes go through the api_proxy to the AdminHelper server's REST API.

Pages & Infrastructure hub (server mode)

In server mode the client is the operational cockpit. Five pages:

PageContent
DashboardOverview/status after login.
InfrastructureServer inventory (create/edit/delete) and, per server, tabs: Overview, Connections (ssh/rdp/vnc/web/custom, server-side CRUD), Tunnel (FRP tunnels per server; the FRP server config is picked but stays a web-admin concern), Monitoring (this server's checks), Provisioning (one-time token + agent command).
ConnectionsQuick launcher for starting sessions; in server mode also server-side editing.
MonitoringFleet-wide: checks overview, alert rules (CRUD), templates (CRUD), alert log.
AnsibleCreate/edit/delete AND run playbooks (3-step wizard).

These operational pages exist only in server mode; in standalone/local mode the client stays a pure connection launcher. Writes go through api_proxy against the server's REST API (admin session required).

RDP Race Guard

Clicking Connect multiple times in quick succession would, without protection, spawn multiple RDP processes. The Desktop Client assigns a monotonic UUID per click. If the FRP tunnel setup is still running when a newer connect ID comes in, the older one is discarded. Implementation in apps/desktop/src-tauri/src/rdp/race_guard.rs.

Keyring Secrets

Passwords are stored in the OS keyring, never in the local connection file:

  • Linux: secret-service (GNOME Keyring / KWallet)
  • macOS: Keychain
  • Windows: Credential Manager

Service prefix adminhelper:, account = <connection-id>.

TLS certificate pinning (TOFU)

When the user enables “Allow self-signed certificates”, TLS verification is not turned off (the old danger_accept_invalid_certs(true) disabled chain and hostname checks with no pinning – open to MITM). Instead tofu.rs pins the server certificate using the trust-on-first-use model (like SSH known_hosts):

  • Single choke point: on the self-signed path auth::build_client builds a reqwest client with a custom rustls ServerCertVerifier (use_preconfigured_tls, ring provider). This covers every credential-carrying path – login, refresh, api_proxy, the tunnel/connection fetches and sync.
  • Pin = SHA-256 of the leaf DER. Captured on first connect and stored in the OS keyring (key tofu|cert|<host:port>, service com.adminhelper.app) with an in-memory cache so the enforcement path does not hit the keyring per request. Afterwards: accept only on an exact fingerprint match, otherwise reject (possible MITM). Hostname/expiry are intentionally irrelevant – the pin is the identity.
  • The handshake signature is still verified against the presented public key (delegated to the ring provider), so a captured certificate cannot be replayed without its private key.
  • Recovery after a legitimate certificate rotation: reset_server_cert_pin clears the pin (keyring + cache) – UI button “Reset pinned certificate” in settings.
  • Enrolled state (mTLS): once the device is enrolled (A5), every request goes through enrolled_client – the CA pin in enrollment.rs (hostname-agnostic), not the leaf pin. If the server certificate no longer chains to the CA pinned at enrollment (server reinstall / re-created PKI → BadSignature or UnknownIssuer), the CaPinVerifier returns an explicit danger message instead of the raw rustls error. Recovery: reset_device_identity clears the mTLS identity (enroll|key/enroll|cert/enroll|ca) and the TOFU pin in one step – UI button “Reset device identity” (server mode, only shown when enrolled; with a danger confirmation). Sign-in / re-enrollment is required afterwards.
  • Verification: a real-handshake test in tofu.rs (tokio-rustls server, two distinct certificates) proves first-use pinning, reconnect with the same cert, and rejection on a cert change.
  • Accompanying hardening: api_proxy only sends the JWT to the logged-in server URL (token-destination pin, auth::stored_server_url); launch_ansible only accepts paths under the app temp dir (adminhelper_ansible prefix, canonicalized – blocks ../symlink escape); the webview CSP connect-src is narrowed to 'self' ipc: http://ipc.localhost (no https: exfiltration channel, since all server traffic goes through api_proxy).

FRP Sidecar

Tauri capabilities are scoped: frpc can only be spawned from the app binaries directory. At build time the matching target binary (frpc-x86_64-unknown-linux-gnu, frpc-aarch64-apple-darwin, …) must be dropped into apps/desktop/src-tauri/binaries/.

Server Mode Flow

In Server mode (setting mode = 'server') the client loads connections from the AdminHelper server via JWT and brings up the FRP tunnel itself. The steps wire together across auth.rs, sync.rs, frpc.rs and tunnel.rs:

  1. Login – login posts to /api/auth/login, then fetches /api/auth/me and stores the access token, refresh token and server URL in the OS keyring (service com.adminhelper.app, keys auth|jwt, auth|refresh, auth|server_url). Implemented in auth.rs.
  2. Login on every start – For security the client no longer restores a stored session on startup: server mode always requires a fresh password login (server URL and username are pre-filled, persisted in settings.json as serverUrl/lastUsername – never the password). During a running session authenticated_get still transparently refreshes the access token at /api/auth/refresh on a 401. The opportunistic mTLS cert renewal (maybe_renew) now hangs off a successful login (previously off the removed check_session).
  3. Load connections – fetch_connections_jwt retrieves the list via authenticated_get from /api/connections (with transparent 401 refresh retry) and writes it to the server/sync cache (connections.json). Local mode uses a separate store (connections.local.json) that a server/sync fetch never overwrites – so local connections survive switching into server mode and back. Passwords are never part of the sync data.
  4. Start tunnel – start_tunnel fetches the visitor bundle (TOML + PKI map) from /api/frp/generate/visitor-bundle, writes visitor.toml and the PKI files into the app data directory (relative pki/ paths are rewritten to absolute, .key files with 0600) and then spawns frpc -c <config> as a sidecar.
  5. Status & events – tunnel_status returns running, visitor_name and connected_since. The sidecar process emits the Tauri events frpc-terminated and frpc-error; the tunnel store (stores/tunnel.ts) listens for them and feeds the status indicator (TunnelIndicator.svelte) with the UI states connecting / connected / disconnected.
  6. Tunnel resolution – fetch_tunnels retrieves the mappings from /api/frp/tunnels, resolve_connection rewrites matching connections: stcp → host = 127.0.0.1, port = visitor_port (web also http://127.0.0.1:<port>), https → url = https://<custom-domain>.

Sidecar auto-start happens only in Server mode: startIfServerMode() in the tunnel store bails out when there is no session or settings.mode !== 'server'. On app exit, RunEvent::Exit in main.rs tears down the frpc process via stop_frpc.

Connect Flow

  1. User clicks Connect → connectFlow store sets state = 'connecting'
  2. Bridge calls connect_rdp (with race-guard UUID)
  3. Rust backend resolves the FRP tunnel (local visitor port)
  4. Password is fetched from the keyring or prompted via the PasswordPrompt modal
  5. RDP/SSH/VNC client is launched with a temporary config
  6. After exit the tunnel is torn down
