Desktop Client

The Desktop Client is a Tauri 2 wrapper around a Svelte 5 SPA. The Rust backend (apps/desktop/src-tauri/) exposes 23 typed commands; the frontend (apps/desktop/ui/) consumes them through a 1:1 bridge.

Tauri Commands

The 23 commands are registered in src/main.rs via invoke_handler and cover auth, connection persistence, connect flow, FRP tunnel, keyring, API proxy and Ansible. Each command is typed – the bridge in apps/desktop/ui/src/lib/bridge/ mirrors them 1:1.

RDP Race Guard

Clicking Connect multiple times in quick succession would, without protection, spawn multiple RDP processes. The Desktop Client assigns a monotonic UUID per click. If the FRP tunnel setup is still running when a newer connect ID comes in, the older one is discarded. Implementation in apps/desktop/src-tauri/src/rdp/race_guard.rs.

Keyring Secrets

Passwords are stored in the OS keyring, never in the local connection file:

  • Linux: secret-service (GNOME Keyring / KWallet)
  • macOS: Keychain
  • Windows: Credential Manager

Service prefix adminhelper:, account = <connection-id>.

FRP Sidecar

Tauri capabilities are scoped: frpc can only be spawned from the app binaries directory. At build time the matching target binary (frpc-x86_64-unknown-linux-gnu, frpc-aarch64-apple-darwin, …) must be dropped into apps/desktop/src-tauri/binaries/.

Server Mode Flow

In Server mode (setting mode = 'server') the client loads connections from the AdminHelper server via JWT and brings up the FRP tunnel itself. The steps wire together across auth.rs, sync.rs, frpc.rs and tunnel.rs:

  1. Login – login posts to /api/auth/login, then fetches /api/auth/me and stores the access token, refresh token and server URL in the OS keyring (service com.adminhelper.app, keys auth|jwt, auth|refresh, auth|server_url). Implemented in auth.rs.
  2. Session restore – check_session validates the stored access token against /api/auth/me; on 401 it exchanges the refresh token at /api/auth/refresh for a fresh token pair and re-saves it to the keyring. No re-login after restart.
  3. Load connections – fetch_connections_jwt retrieves the list via authenticated_get from /api/connections (with transparent 401 refresh retry) and writes it locally. Passwords are never part of the sync data.
  4. Start tunnel – start_tunnel fetches the visitor bundle (TOML + PKI map) from /api/frp/generate/visitor-bundle, writes visitor.toml and the PKI files into the app data directory (relative pki/ paths are rewritten to absolute, .key files with 0600) and then spawns frpc -c <config> as a sidecar.
  5. Status & events – tunnel_status returns running, visitor_name and connected_since. The sidecar process emits the Tauri events frpc-terminated and frpc-error; the tunnel store (stores/tunnel.ts) listens for them and feeds the status indicator (TunnelIndicator.svelte) with the UI states connecting / connected / disconnected.
  6. Tunnel resolution – fetch_tunnels retrieves the mappings from /api/frp/tunnels, resolve_connection rewrites matching connections: stcp → host = 127.0.0.1, port = visitor_port (web also http://127.0.0.1:<port>), https → url = https://<custom-domain>.

Sidecar auto-start happens only in Server mode: startIfServerMode() in the tunnel store bails out when there is no session or settings.mode !== 'server'. On app exit, RunEvent::Exit in main.rs tears down the frpc process via stop_frpc.

Connect Flow

  1. User clicks Connect → connectFlow store sets state = 'connecting'
  2. Bridge calls connect_rdp (with race-guard UUID)
  3. Rust backend resolves the FRP tunnel (local visitor port)
  4. Password is fetched from the keyring or prompted via the PasswordPrompt modal
  5. RDP/SSH/VNC client is launched with a temporary config
  6. After exit the tunnel is torn down
