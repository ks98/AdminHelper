Hooks & Events

The hook system is the primary outbound integration surface. Every relevant server event fires an event, registered scripts run in a sanitised subprocess with a hard timeout.

Architecture

tree
server/app/modules/hooks/
├── router.py          /api/hooks CRUD
├── models.py          Hook, HookExecution
├── schemas.py         Pydantic schemas
├── scheduler.py       Event dispatch + retry queue
├── script_runner.py   Async subprocess spawn
└── script_worker.py   Child process: stdin=JSON, stdout/stderr capture

Event Catalogue

EventPayload (excerpt)
server.provisionedserver_id, hostname, ip
server.online / server.offlineserver_id, last_seen
connection.created / updated / deletedConnection object
connection.launchedconnection_id, user, client (desktop / web)
alert.triggered / resolved / acknowledgedcheck_id, severity, value
user.login / login_faileduser_id, ip, reason?
hook.failedMeta-event on hook failure (CANNOT trigger more hooks – loop guard)

Hook Definition

json
{
  "name": "Slack alert",
  "event": "alert.triggered",
  "script": "#!/usr/bin/env bash\nset -euo pipefail\nEVENT=$(cat)\ncurl -sS -X POST $SLACK_URL -d \"payload={\\\"text\\\": \\\"Alert: $(echo $EVENT | jq .check_name)\\\"}\"",
  "timeout_sec": 10,
  "enabled": true
}

The event JSON is passed via stdin, not via argument or env var. This avoids shell-escape issues and ARG_MAX limits.

Sandbox Model

Hook scripts run in a fork with tightened defaults:

  • Minimal env: only PATH, HOME, LANG, plus explicitly configured secrets (SLACK_URL, etc.). No leakage of SECRET_KEY, ADMIN_PASSWORD, etc.
  • CWD is a tempfile.mkdtemp() directory, cleaned up afterwards.
  • Timeout via asyncio.wait_for + process.kill(), default 10 s, max 300 s.
  • Output limit: 64 KiB each for stdout and stderr; the rest is truncated and marked in the log ([truncated]).
  • User switch: runs as nobody when the server is deployed with root capabilities (Docker: --user 65534).
Not a full sandbox

The hook isolation is defense-in-depth, not a seccomp/namespace jail. Admins with hook-CRUD privileges can run arbitrary code on the server host. Grant hook access only to trusted roles.

Retry Strategy

If a hook fails (exit_code != 0 or timeout), it is retried with exponential backoff: 30 s, 2 min, 10 min, 1 h. After the fourth failure HookExecution.state = failed is set and the meta-event hook.failed is emitted (loop guard prevents cascades).

Script Runner Implementation

python
async def run_hook(hook: Hook, event: dict) -> HookExecution:
    payload = json.dumps(event).encode()
    env = _sanitised_env(hook)  # only PATH, HOME, LANG, + explicit secrets

    with tempfile.TemporaryDirectory() as cwd:
        proc = await asyncio.create_subprocess_exec(
            "/bin/sh", "-c", hook.script,
            stdin=asyncio.subprocess.PIPE,
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.PIPE,
            cwd=cwd, env=env,
        )
        try:
            out, err = await asyncio.wait_for(
                proc.communicate(payload),
                timeout=min(hook.timeout_sec, MAX_TIMEOUT),
            )
        except asyncio.TimeoutError:
            proc.kill()
            return HookExecution(state="timeout", ...)
        return HookExecution(state="ok" if proc.returncode == 0 else "failed",
                             stdout=out[:65_536], stderr=err[:65_536],
                             exit_code=proc.returncode)

Firing Custom Events

Internal code can emit events at any time:

python
from app.modules.hooks import dispatch_event

await dispatch_event("connection.launched", {
    "connection_id": conn.id,
    "user": current_user.username,
    "client": "desktop",
})

dispatch_event is non-blocking: the hook scheduler queues the run and returns immediately. The HTTP response of the emitter does not wait on the hook.

Test Execution

Every hook can be tested from the web panel with a sample event: POST /api/hooks/{id}/test. The execution log shows stdout/stderr/exit_code in real time – handy for script debugging.
