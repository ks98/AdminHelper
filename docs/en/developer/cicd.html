CI/CD & Release

GitHub Actions builds and tests every component, pushes the server and monitoring images to the GitHub Container Registry (ghcr.io), and creates a draft release with the desktop and agent artifacts on a version tag.

Workflows

WorkflowPurpose
ci.ymlLint & tests for every component: server pytest (Postgres service, Python 3.12/3.13), monitoring pytest (Postgres service for the migration smoke test), Go agent (gofmt/vet/test), desktop UI & web panel (svelte-check/eslint/vitest, Playwright E2E), Rust/Tauri (fmt/clippy/test). Additional PR gates against “green but broken”: cargo check on Windows, a cargo tauri build smoke on Linux (debug, deb only), Docker builds of both images without push, and an FRP pin consistency check (FRP_VERSION + SHA-256 across ci.yml/release.yml/docker-compose.yml). Coverage runs report-only in the test jobs (no gate)
docker.ymlBuild & push the server + monitoring images to ghcr.io/ks98/adminhelper/{server,monitoring}
release.ymlDesktop DEB/RPM/binary, Go-agent DEB/RPM + Linux/Windows binaries, Windows MSI (best-effort), SHA256SUMS, draft release
audit.ymlWeekly dependency audit (Monday morning, also manually via workflow_dispatch): pip-audit (server + monitoring locks), cargo audit (desktop), govulncheck (agent), npm audit (web + desktop UI)

Trigger Matrix

EventWhat runs
Pull requestci.yml (must pass)
Push to mainci.yml + docker.yml (image tags :main + :sha-…)
Tag v*docker.yml (tags :0.26.0 + :latest + :sha-…) + release.yml (draft release with artifacts)
Schedule (weekly)audit.yml (dependency audit)

Container Images (ghcr.io)

Tagging is driven by docker/metadata-action. :latest is produced only on a (non-prerelease) semver tag – not on every main push:

text
ghcr.io/ks98/adminhelper/server:0.24.0      # release tag
ghcr.io/ks98/adminhelper/server:latest      # latest semver release
ghcr.io/ks98/adminhelper/server:main        # latest main build
ghcr.io/ks98/adminhelper/server:sha-abc1234 # per commit
# likewise: ghcr.io/ks98/adminhelper/monitoring:…
Package visibility

Freshly created ghcr packages are private. After the first successful docker.yml run, server and monitoring must be switched to public once in the GitHub package settings – otherwise docker compose pull fails for external users with unauthorized/manifest-unknown.

Release Artifacts

release.yml runs on a v* tag and attaches the built artifacts to a draft release (not published until a human clicks “Publish”):

  • Desktop client: .deb + .rpm (Tauri) + ELF binary
  • Go agent: .deb + .rpm + linux/windows binaries
  • Desktop client Windows .msi (best-effort – does not block the release if the Windows build fails)
  • SHA256SUMS over all assets

The agent version is derived from the tag (${GITHUB_REF_NAME#v}) and injected into the binaries via -ldflags "-X main.version=$VERSION". The tauri-cli is pinned to an exact version in CI and release (cargo install tauri-cli --locked --version …) so a new CLI release cannot break a tag build.

Version Locations

A release needs the new version set in multiple places at once – the tag alone isn't enough:

  • apps/desktop/src-tauri/tauri.conf.json + Cargo.toml + Cargo.lock – Tauri bundle
  • apps/desktop/ui/package.json + apps/web/package.json (+ each package-lock.json) – Vite builds
  • apps/desktop/ui/src/components/AppShell.svelte – sidebar badge
  • apps/agent/build-deb.sh + build-rpm.sh – default VERSION (CI overrides it from the tag)
  • docs/**/*.html – footer <span>v0.24.0</span> + landing “What's new” + code samples
  • README.md – agent DEB install command
  • CHANGELOG.md – new section at the top + link ref

The server itself does not carry its own version – the backend is identified by Docker image tag.

Release Checklist

  1. Bump the versions in the files listed above to the target value.
  2. Update the footer versions in docs/**/*.html (footer span + landing “What's new” + code samples).
  3. Update CHANGELOG.md (Keep a Changelog: new section + link ref).
  4. Commit & tag: git commit -am "release: v0.24.0" && git tag v0.24.0.
  5. git push --follow-tags.
  6. Watch docker.yml + release.yml; after the first image push, switch the ghcr packages to public (once).
  7. In the GitHub draft release, review the notes, verify the mandatory assets, and click “Publish”.
Draft instead of auto-publish

The release is deliberately created as a draft, and the release job is success-gated on the mandatory Linux builds. This lets you verify before publishing that all mandatory artifacts (Linux desktop + agent) are actually attached – the Windows MSI build is best-effort and may be missing.
