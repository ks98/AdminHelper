CI/CD & Release

The GitLab CI pipeline builds the Docker image, desktop DEBs, and Go-agent packages. Release artifacts are shipped to a signed Debian repo via the Aptly REST API – not GitLab Packages.

Pipeline Stages

StageJobs
checkpytest, svelte-check, cargo check, go vet
dockerBuild & push server image to docker.nevondo.com
downloadfrpc binary from the official release
buildDesktop DEB (Trixie), Go-agent DEB/RPM for linux-amd64 + windows-amd64
publishUpload DEBs via Aptly API, create & publish repo snapshots

Trigger Matrix

EventWhat runs
Merge requestcheck (must pass)
Branch pushcheck + docker + build, no publish
Tag v*.*.*Full pipeline including publish to the prod repo
Tag agent-v*.*.*Agent build + agent publish only (server & desktop stay untouched)

Version Locations

A release needs the new version set in multiple places at once – the tag alone isn't enough. As of v0.23.1:

  • frontend-src/package.json – Vite build of the web panel
  • desktop-src/package.json – Vite build of the desktop frontend
  • desktop/src-tauri/tauri.conf.json + desktop/src-tauri/Cargo.toml – Tauri bundle version
  • extension/manifest.json – browser extension
  • .gitlab-ci.yml – AGENT_VERSION variable in agent build/publish jobs (multiple occurrences)
  • docs/**/*.html – footer <span>v0.23.1</span> across 50 HTML files (DE + EN)
  • CHANGELOG.md – new section at the top

The server itself does not carry its own version – the backend is identified by Docker image tag. In the Go agent the version is injected via -ldflags "-X main.version=$VERSION" at CI build time.

Aptly Publish

The publish CI job uploads DEBs to an Aptly staging repo via the REST API (mutual TLS). After a successful upload a snapshot is created and published to stable:

bash
# desktop/scripts/ci_publish_via_api.sh (shortened)
curl --cert "$APTLY_CLIENT_CERT" --key "$APTLY_CLIENT_KEY" \
     -F file=@"$DEB" \
     "$APTLY_URL/api/files/srm-staging"

curl --cert "$APTLY_CLIENT_CERT" --key "$APTLY_CLIENT_KEY" \
     -X POST "$APTLY_URL/api/repos/srm/file/srm-staging"

curl --cert "$APTLY_CLIENT_CERT" --key "$APTLY_CLIENT_KEY" \
     -X POST -H 'Content-Type: application/json' \
     -d "{\"Name\":\"srm-$VERSION\",\"SourceKind\":\"local\",\"SourceRepo\":\"srm\"}" \
     "$APTLY_URL/api/snapshots/srm"

curl --cert "$APTLY_CLIENT_CERT" --key "$APTLY_CLIENT_KEY" \
     -X PUT -H 'Content-Type: application/json' \
     -d "{\"Snapshots\":[{\"Component\":\"main\",\"Name\":\"srm-$VERSION\"}]}" \
     "$APTLY_URL/api/publish/stable/trixie"
No GitLab Packages

DEBs are published exclusively via Aptly. GitLab Packages is not used – don't upload manually, don't use it as a fallback. The public APT source pulls exclusively from the Aptly repo.

Docker Image

bash
# Multi-stage: frontend build → Python runtime
docker build \
  --target runtime \
  -t docker.nevondo.com/ks98/adminhelper/server:$CI_COMMIT_TAG \
  -t docker.nevondo.com/ks98/adminhelper/server:latest \
  .
docker push docker.nevondo.com/ks98/adminhelper/server:$CI_COMMIT_TAG
docker push docker.nevondo.com/ks98/adminhelper/server:latest

The latest tag is only pushed on prod releases – dev branches get only the commit-SHA tag, preventing deployment confusion.

Tauri Build (Trixie)

The desktop-client build runs in a debian:trixie Docker runner. The toolchain is installed per job, but the Cargo cache + target/ directory are reused via GitLab cache – saving around 6 min per build:

yaml
cache:
  key:
    files: [desktop/src-tauri/Cargo.lock]
  paths:
    - .cargo/bin
    - .cargo/registry/index
    - .cargo/registry/cache
    - .cargo/git/db
    - .rustup/toolchains
    - desktop/src-tauri/target

Go-Agent Build

Cross-compile for Linux + Windows in a single job. Version is injected via -ldflags so that adminhelper-agent version is correct:

bash
VERSION=${CI_COMMIT_TAG#v}
GOOS=linux   GOARCH=amd64 go build \
  -ldflags "-X main.Version=$VERSION -X main.Commit=$CI_COMMIT_SHORT_SHA" \
  -o bin/adminhelper-agent ./cmd/adminhelper-agent

GOOS=windows GOARCH=amd64 go build \
  -ldflags "-X main.Version=$VERSION -X main.Commit=$CI_COMMIT_SHORT_SHA" \
  -o bin/adminhelper-agent.exe ./cmd/adminhelper-agent

VERSION=$VERSION bash build-deb.sh
VERSION=$VERSION bash build-rpm.sh

Release Checklist

  1. Bump the versions in the files listed above (e.g. with sed across frontend-src/package.json, desktop-src/package.json, desktop/src-tauri/tauri.conf.json, desktop/src-tauri/Cargo.toml, extension/manifest.json, .gitlab-ci.yml).
  2. Update the footer versions in docs/**/*.html: find docs -name '*.html' -exec sed -i 's/v0\.OLD/v0.NEW/g' {} \;
  3. Update CHANGELOG.md (manual, Conventional-Commits style).
  4. Commit & tag: git commit -am "release: v0.23.1" && git tag v0.23.1.
  5. git push --follow-tags.
  6. Watch the pipeline – once publish is green, the new version is live on deb.ks98.de.
  7. Add release notes in the GitLab UI (Markdown from CHANGELOG.md).
Agent-only release

For Go-agent hotfixes a tag agent-v0.20.1 is enough. The pipeline skips server + desktop jobs and only publishes the agent DEB/RPM. Saves 10+ min of CI time.
