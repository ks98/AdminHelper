Developer Guide

Server (Backend)

A FastAPI monolith: HTTP routers, SQLAlchemy models, APScheduler jobs, and FRP PKI are one code monolith — separation by module, not by deployment. For scaling, the web workers and the scheduler run as separate processes of the same image (see Event Bus & Scheduler).

Entry Points

apps/server/app/core/config.pyEnv-var resolution, JWT settings, secret-key persistence (.secret_key file).
apps/server/app/core/auth.pyJWT encode/decode, bcrypt hashing with SHA-256 prehash, refresh rotation, token blacklist, API-key auth.

apps/server/app/core/events.pyAsync event bus for hooks (ThreadPoolExecutor with 4 workers, fire-and-forget).

No dedicated alerts/audit/settings router

Alerts and check status live in the monitoring service and are exposed via monitoring_proxy. There is no dedicated audit-log router as of v0.23.2; logging happens through Python logging and hook events.

Auth Architecture

JWT access token (HS256, 8 h) and refresh token (7 d) are returned in the JSON body; the refresh token is additionally set as an HttpOnly; Secure; SameSite=Strict cookie (path /api/auth) that browser clients use instead of localStorage (desktop/CLI keep using the body). Token claims: sub (username), type (access/refresh), exp, iat (issued-at), jti (UUID).

  1. Login signs both tokens and returns them in TokenResponse.
  2. On POST /api/auth/refresh the server checks signature, expiry and blacklist.
  3. The old jti is added to token_blacklist immediately, then new tokens are issued (rotation).
  4. If an already-revoked refresh is presented again → reuse detection: all sessions of the affected user chain are invalidated (WARNING log with IP).
  5. Password reset revokes sessions: changing a password sets users.tokens_valid_after; any token whose iat predates it (or that has no iat) is rejected — without needing to know individual jtis.

Password hashing uses SHA-256 prehash → bcrypt, because bcrypt only consumes 72 bytes.

API keys are accepted via the X-API-Key header; the ApiKeyOrUser dependency allows JWT or API key on the same endpoint.

Rate limit & IP filter

Login limit (5 per 60 s/IP) and webhook limit (20 per 60 s/IP) are Redis-backed with in-memory fallback. IP filter via ALLOWED_IPS (CIDR list); proxy trust via TRUSTED_PROXIES (safe) instead of TRUST_PROXY_HEADERS (legacy).

Server Provisioning

New servers are onboarded via one-time tokens with a 24-hour TTL — since v0.23.2 FRP-independent inside the server module:

  1. Admin calls POST /api/servers/{server_id}/provision/token and receives {token, expiresAt, serverId, serverName}.
  2. The agent redeems it with POST /api/servers/{server_id}/provision/activate (header X-Provision-Token) and gets back: a fresh server API key (always), monitor agent key + URL (when the monitor service is reachable), frpc.toml (when the server has FRP tunnels) and a one-time enrollment token. Fields are null when the respective component is not configured — an unreachable monitor service does not block provisioning. A per-client PKI bundle is no longer shipped: the agent enrolls its own mTLS/tunnel cert at the ca-issuer (A4/A7).
  3. Tokens are one-time: used_at is set, further activations fail.
  4. At runtime the agent syncs its frpc.toml via GET /api/frp/provision/{server_id}/config + config-hash (only when FRP is configured).

FRP — TOML Structure

modules/frp/config_generator.py renders the FRP configs straight from the DB models into TOML strings — no template engine, just string lists. Three generators:

FunctionTargetContent
generate_frps_toml(config)frps.toml (server)bindPort, optional vhostHTTPSPort/subDomainHost/maxPortsPerClient, optional dashboard block (webServer.* on 127.0.0.1), auth.method = "token" + token, plus a [transport.tls] server block.
generate_frpc_toml(config, tunnels, frpc_user)frpc.toml (agent)serverAddr/serverPort/user, token auth, client TLS block, then one [[proxies]] block per active tunnel.
generate_visitor_toml(config, tunnels, visitor_user)Visitor frpc.toml (admin PC)Aggregates all active STCP tunnels into one [[visitors]] block each (bindPort = visitor_port).

Only tunnels with enabled = true are rendered. Two tunnel types get special handling:

  • STCP (type = "stcp"): gets a secretKey and allowUsers (default ["ops-admin"], otherwise the passed allow_users). The matching [[visitors]] block sets serverName, serverUser (= agent user, because FRP registers proxies as {user}.{name}) and the same secretKey.
  • HTTPS (type = "https"): renders customDomains from the comma-separated custom_domains field.

extra_config (JSON) is appended per tunnel preserving types: strings quoted, bools as true/false, numbers raw.

frpc.toml (STCP example)
serverAddr = "frp.example.com"
serverPort = 7000
user = "k01-lnx1"

auth.method = "token"
auth.token = "<token>"

[transport.tls]
enable = true
trustedCaFile = "/etc/adminhelper/identity/ca.crt"
certFile = "/etc/adminhelper/identity/agent.crt"
keyFile = "/etc/adminhelper/identity/agent.key"

[[proxies]]
name = "ssh"
type = "stcp"

FRP PKI & TLS

As of A7 the frps server cert and the agent tunnel cert come from the ca-issuer's tunnel intermediate (ECDSA P-256, a single trust root): frps mounts the issuer-provisioned material (frps.crt/frps.key/ca.crt) read-only at /etc/frp-pki, and the agent uses the tunnel cert it enrolled (A4) at /etc/adminhelper/identity — one cert for server pushes and the frp tunnel, auto-renewed at ~50 % of its lifetime.

The server no longer signs FRP certificates (D6, F3). The former server-run FRP CA (modules/frp/pki.py + pki_router.py with the /api/frp/pki/* endpoints + the startup CA generation in _ensure_pki()) was fully removed with the PKI migration, along with the frp-pki volume and the former master/published split. _ensure_frps_config() only writes the non-secret frps.toml at startup.

The desktop STCP visitor now presents its enrolled access identity as its frpc client cert (the desktop exports key/cert/CA from its keyring into files under identity/). For frps to accept it, the ca-issuer adds the access intermediate to frps's ca.crt (ensure_frps_cert(..., extra_trust)); the real per-tunnel authorization stays the STCP secretKey + server-side bundle filtering, not the cert scope.

The TLS wiring in frps.toml/frpc.toml is produced by the config_generator: the server block (force = true) points at the issuer-provisioned material under /etc/frp-pki; the agent block (enable = true) at the enrolled tunnel cert under /etc/adminhelper/identity; the visitor block at the desktop's exported enrolled identity (relative identity/, rewritten to absolute paths at runtime).

SQLAlchemy Models

Tables live in their respective modules (e.g. modules/users/models.py). The schema is managed by Alembic (apps/server/alembic/): docker-entrypoint.sh runs alembic upgrade head before startup. The former SQLite approach (Base.metadata.create_all() at startup plus idempotent _migrate_add_columns() PRAGMA migrations) was dropped with the move to Postgres. The test suite, by contrast, builds the schema directly via Base.metadata.create_all() and checks for drift between models and migrations with test_alembic_consistency.py.

  • User ↔ Server (n:m via user_servers), TokenBlacklist
  • Server (UUID PK, JSON tags, os_type) ↔ Connection (n:1, server_id nullable for standalone)
  • ApiKey with permission ∈ {read, read_write}
  • Hook (hook_type ∈ {webhook, event, schedule}, JSON event_triggers, Python script)
  • FrpServerConfig ↔ FrpTunnel (n:1) ↔ ProvisionToken (n:1, 24 h TTL, one-time)
  • Playbook (Ansible) with disk-backed YAML content

Event Bus & Scheduler

APScheduler runs in a dedicated scheduler process (app.scheduler_main, RUN_MODE=scheduler), not in the web workers — otherwise uvicorn --workers N would run every job N times (duplicate outbox e-mails, duplicate hook runs). A single instance owns all system jobs (cleanups, outbox drain) and the periodic hook reconcile. Trigger types: predefined intervals (5m, 10m, 15m, 30m, 1h, 6h, 12h, 24h) or cron (5 fields).

Since the web workers have no running scheduler, the hook CRUD routers no longer register jobs directly; they only persist the hook row. The scheduler process reconciles schedule hooks from the DB every 30 s (reconcile_scheduled_hooks() — added/changed/removed) and writes next_run back. Multi-worker is enabled via WEB_CONCURRENCY (default 1); Redis is then required for a correct global rate limit, and the DB pool applies per worker (DB_POOL_SIZE/DB_MAX_OVERFLOW).

The event bus in core/events.py distributes events through a ThreadPoolExecutor (4 workers, fire-and-forget). Active event types:

  • server.created, server.updated, server.deleted
  • connection.created, connection.updated, connection.deleted, connections.imported
  • user.created, user.deleted
  • api_key.created, api_key.deleted
  • frp.config.created, frp.config.updated, frp.config.deleted
  • frp.tunnel.created, frp.tunnel.updated, frp.tunnel.deleted
  • playbook.created, playbook.updated, playbook.deleted

Details in Hooks & Events. A curated subset of these events is additionally mirrored into the notification hub via modules/notifications/event_bridge.py (see below).

Notification hub

The server is the hub for user-centric notifications (modules/notifications/). Rationale: only the server knows the user↔server mapping (user_servers) — monitoring stays a pure event source and learns nothing about users. Three tables (Alembic migration a7c3efba142a):

TablePurpose
notification_subscriptionPer-user rules: scope_type (all/tag/server) + scope_ref, min_severity, channel flags. A user may have several.
notificationThe bell feed: one row per recipient, read_at marks it read.
notification_outboxDelivery queue for external channels (e-mail), with attempts/next_attempt_at for retry/backoff.

Data flow:

  1. Sources call service.ingest_event() — the monitoring service via POST /api/internal/events (auth X-Internal-Key = MONITOR_API_KEY, the reverse direction of the monitoring proxy), and the in-process bus via event_bridge.py (curated user.created/server.deleted/frp.tunnel.created).
  2. The resolver (service.resolve_recipients()) maps server_id → user_servers → users × subscriptions — least privilege: a non-admin is only notified about servers assigned to them, admins about all; an event without a server reference goes to admins only. Tag scope matches against the server's JSON tags at runtime (deliberately no tag normalization).
  3. Per recipient one notification row (bell) plus notification_outbox rows for enabled external channels, provided an address is set. A user's several matching subscriptions collapse into one feed entry with the union of channels.
  4. An APScheduler system job (schedule_outbox_drain, every minute, in the dedicated scheduler process — see Event Bus & Scheduler above) drains the outbox via outbox.drain_outbox() + sender.send_email() (smtplib) out of the request path, with linear backoff up to NOTIFICATION_MAX_ATTEMPTS (default 5), then status failed.

Clients fetch the feed via GET /api/notifications (own feed, paginated) plus /unread-count and POST /api/notifications/read; personal settings via GET/PUT /api/users/me/notification-prefs (replace-all).

Push (SSE): GET /api/notifications/stream is a long-lived text/event-stream. After ingest_event() commits, the server publishes a lightweight "refresh" signal (stream_hub.publish()) on the Redis channel notif:events; each web worker holds a Redis subscription and delivers it to its locally connected streams of the affected users. The stream handler is async and DB-free (it only reads from an asyncio.Queue) and authenticates with a short-lived session, so an open stream never holds a DB connection for its whole lifetime. The signal carries no content — the client pulls the rows via REST (no second serialization/authorization path). Without REDIS_URL there is no push; the bell falls back to 30 s polling. The desktop tunnels the stream through the Rust client (src-tauri/src/notifications.rs) because the WebView can't open an EventSource against the self-signed cert; it emits a notification Tauri event per frame.

Persistence

State splits across two locations:

  • PostgreSQL (compose volume postgres-data): two DBs adminhelper (server) and adminhelper_monitor (monitoring).
  • File system under DATA_DIR (default /app/data): bootstrap token, JWT secret, Ansible playbook files.
  • Shared volume frp-config (frps mounts it :ro at /etc/frp): only the non-secret frps.toml (0600). The FRP TLS material is provided by the ca-issuer (volume frps-certs); the PKI crown jewel is its ca-pki volume.

The container starts as root only briefly to chown the mounted paths, then drops via gosu to a non-root user (uid 10001) — uvicorn, Alembic and hook subprocesses all run unprivileged. uvicorn listens plain-HTTP on 8080 (no host port); TLS is terminated by the upstream gateway (nginx), not the server.

tree
/app/data/                         # DATA_DIR (server-only)
├── .bootstrap_token            # one-time setup-token hash (P0-6)
├── .secret_key                 # auto-gen JWT secret
└── ansible/playbooks/{id}/{file}.yaml

/app/frp-config/                   # shared with frps (volume frp-config, :ro for frps)

Tests

Pytest suite under server/tests/. Tests run against a real Postgres 17 via testcontainers (local) or services: postgres:17-alpine (CI). SAVEPOINT pattern in the db_session fixture: outer transaction + begin_nested + after_transaction_end listener — tests can call commit(), everything is rolled back at the end. Plus test_alembic_consistency.py as drift detector between Base.metadata and Alembic migrations.

bash
cd apps/server
uv run pytest -x                 # fail fast
uv run pytest --cov=app          # coverage
