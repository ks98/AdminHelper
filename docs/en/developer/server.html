Developer Guide

Server (Backend)

A FastAPI monolith: HTTP routers, SQLAlchemy models, APScheduler jobs, and FRP PKI all run in the same process. Separation by module, not by deployment.

Entry Points

FilePurpose
server/app/main.pyFastAPI app, lifespan, middleware stack, static mounts for Vite output, auto-migrations, SPA fallback.
server/app/core/config.pyEnv-var resolution, JWT settings, secret-key persistence (.secret_key file).
server/app/core/auth.pyJWT encode/decode, bcrypt hashing with SHA-256 prehash, refresh rotation, token blacklist, API-key auth.
server/app/core/database.pySQLAlchemy engine + SessionLocal; SQLite with check_same_thread=False.
server/app/core/rate_limit.pyRedis-based fixed-window limiter with in-memory fallback.
server/app/core/middleware.pyIP filter (ALLOWED_IPS) and proxy trust (TRUSTED_PROXIES).
server/app/core/events.pyAsync event bus for hooks (ThreadPoolExecutor with 4 workers, fire-and-forget).

Router Overview

Nine top-level routers under server/app/modules/; each module bundles models, schemas and routes.

PrefixModulePurpose
/api/authmodules/users/auth_router.pyLogin, refresh (body, no cookie), logout, /me.
/api/usersmodules/users/User CRUD (admin), /me/servers for self access.
/api/connectionsmodules/connections/Connection CRUD, /touch, /export, /import (replace/merge).
/api/serversmodules/servers/Server inventory with user assignments; cleanup hook to monitoring on delete.
/api/api-keysmodules/api_keys/API keys with read or read_write permission, SHA-256 hashed.
/api/hooksmodules/hooks/Hook CRUD, public /trigger/{token}, /run for manual test, /rotate for webhook token.
/api/ansible/playbooksmodules/ansible/Playbook CRUD with YAML validation, disk storage under /data/ansible/playbooks/{id}/.
/api/frpmodules/frp/Six sub-routers: config, tunnel, generate, status, pki, provision.
/api/monitoringmodules/monitoring_proxy.pyReverse proxy to the monitoring microservice; public agent push endpoint /agent/{id}/report.
No dedicated alerts/audit/settings router

Alerts and check status live in the monitoring service and are exposed via monitoring_proxy. There is no dedicated audit-log router as of v0.20.0; logging happens through Python logging and hook events.

Auth Architecture

JWT access token (HS256, 8 h) and refresh token (7 d) are both returned in the JSON body. Token claims: sub (username), type (access/refresh), exp, jti (UUID).

  1. Login signs both tokens and returns them in TokenResponse.
  2. On POST /api/auth/refresh the server checks signature, expiry and blacklist.
  3. The old jti is added to token_blacklist immediately, then new tokens are issued (rotation).
  4. If an already-revoked refresh is presented again → reuse detection: all sessions of the affected user chain are invalidated (WARNING log with IP).

Password hashing uses SHA-256 prehash → bcrypt, because bcrypt only consumes 72 bytes.

API keys are accepted via the X-API-Key header; the ApiKeyOrUser dependency allows JWT or API key on the same endpoint.

Rate limit & IP filter

Login limit (5 per 60 s/IP) and webhook limit (20 per 60 s/IP) are Redis-backed with in-memory fallback. IP filter via ALLOWED_IPS (CIDR list); proxy trust via TRUSTED_PROXIES (safe) instead of TRUST_PROXY_HEADERS (legacy).

FRP Provisioning

New servers are onboarded via one-time tokens with a 24-hour TTL:

  1. Admin calls POST /api/frp/provision/{server_id}/token and receives {token, expiresAt}.
  2. The agent redeems it with POST /api/frp/provision/{server_id}/activate (header X-Provision-Token) and gets back a fresh API key, frpc.toml, PKI bundle (CA + client cert + key) and the server name.
  3. Tokens are one-time: used_at is set, further activations fail.
  4. At runtime the agent syncs its frpc.toml via GET /provision/{server_id}/config + config-hash.

SQLAlchemy Models

Tables live in their respective modules (e.g. modules/users/models.py) and are created at startup via Base.metadata.create_all(). Idempotent column migrations run through _migrate_add_columns() with PRAGMA table_info.

  • User ↔ Server (n:m via user_servers), TokenBlacklist
  • Server (UUID PK, JSON tags, os_type) ↔ Connection (n:1, server_id nullable for standalone)
  • ApiKey with permission ∈ {read, read_write}
  • Hook (hook_type ∈ {webhook, event, schedule}, JSON event_triggers, Python script)
  • FrpServerConfig ↔ FrpTunnel (n:1) ↔ ProvisionToken (n:1, 24 h TTL, one-time)
  • Playbook (Ansible) with disk-backed YAML content

Event Bus & Scheduler

APScheduler runs in the same event loop as FastAPI. At startup load_all_scheduled_hooks() registers every active schedule hook; trigger types: predefined intervals (5m, 10m, 15m, 30m, 1h, 6h, 12h, 24h) or cron (5 fields).

The event bus in core/events.py distributes events through a ThreadPoolExecutor (4 workers, fire-and-forget). Active event types:

  • server.created, server.updated, server.deleted
  • connection.created, connection.updated, connection.deleted, connections.imported
  • user.created, user.deleted
  • api_key.created, api_key.deleted
  • frp.config.created, frp.config.updated, frp.config.deleted
  • frp.tunnel.created, frp.tunnel.updated, frp.tunnel.deleted
  • playbook.created, playbook.updated, playbook.deleted

Persistence

All stateful data lives under DATA_DIR (default /app/data):

tree
/app/data/
├── db.sqlite3                  # main DB (SQLAlchemy)
├── .secret_key                 # auto-gen JWT secret
├── connections.json.migrated   # legacy migration marker
├── frp-config/
│   ├── frps.toml
│   └── pki/
│       ├── ca.crt, ca.key
│       ├── frps.crt, frps.key
│       └── {client}.crt, {client}.key
└── ansible/playbooks/{id}/{file}.yaml

Tests

Pytest suite under server/tests/. The test DB is isolated via DATA_DIR=/tmp/... in conftest.py. FastAPI's TestClient uses a shared-memory SQLite so migrations are deterministic.

bash
cd server
uv run pytest -x                 # fail fast
uv run pytest --cov=app          # coverage
