Server (Backend)

A FastAPI monolith: HTTP routers, SQLAlchemy models, APScheduler jobs, and FRP PKI all run in the same process. Separation by module, not by deployment.

Entry Points

FilePurpose
server/app/main.pyFastAPI app, lifespan, middleware stack, static mounts for Vite output, auto-migrations, SPA fallback.
server/app/core/config.pyEnv-var resolution, JWT settings, secret-key persistence (.secret_key file).
server/app/core/auth.pyJWT encode/decode, bcrypt hashing with SHA-256 prehash, refresh rotation, token blacklist, API-key auth.
server/app/core/database.pySQLAlchemy engine + SessionLocal; PostgreSQL via psycopg3, pool config (size=10, overflow=20, pre-ping, recycle=3600).
server/app/core/rate_limit.pyRedis-based fixed-window limiter with in-memory fallback.
server/app/core/middleware.pyIP filter (ALLOWED_IPS) and proxy trust (TRUSTED_PROXIES).
server/app/core/events.pyAsync event bus for hooks (ThreadPoolExecutor with 4 workers, fire-and-forget).

Router Overview

Ten top-level routers under server/app/modules/; each module bundles models, schemas and routes.

PrefixModulePurpose
/api/authmodules/users/auth_router.pyLogin, refresh (body, no cookie), logout, /me.
/api/usersmodules/users/User CRUD (admin), /me/servers for self access.
/api/connectionsmodules/connections/Connection CRUD, /touch, /export, /import (replace/merge).
/api/serversmodules/servers/Server inventory with user assignments; cleanup hook to monitoring on delete.
/api/api-keysmodules/api_keys/API keys with read or read_write permission, SHA-256 hashed.
/api/hooksmodules/hooks/Hook CRUD, public /trigger/{token}, /run for manual test, /rotate for webhook token.
/api/ansible/playbooksmodules/ansible/Playbook CRUD with YAML validation, disk storage under /data/ansible/playbooks/{id}/.
/api/servers/{id}/provisionmodules/provisioning/Server-centric provisioning: token lifecycle and activate (FRP-independent). Returns server API key, optional monitor key, optional FRP bundle.
/api/frpmodules/frp/Five sub-routers: config, tunnel, generate, status, pki, plus provision/{id}/config + config-hash for the running FRPC sync.

No dedicated alerts/audit/settings router

Alerts and check status live in the monitoring service and are exposed via monitoring_proxy. There is no dedicated audit-log router as of v0.23.2; logging happens through Python logging and hook events.

Auth Architecture

JWT access token (HS256, 8 h) and refresh token (7 d) are both returned in the JSON body. Token claims: sub (username), type (access/refresh), exp, iat (issued-at), jti (UUID).

  1. Login signs both tokens and returns them in TokenResponse.
  2. On POST /api/auth/refresh the server checks signature, expiry and blacklist.
  3. The old jti is added to token_blacklist immediately, then new tokens are issued (rotation).
  4. If an already-revoked refresh is presented again → reuse detection: all sessions of the affected user chain are invalidated (WARNING log with IP).
  5. Password reset revokes sessions: changing a password sets users.tokens_valid_after; any token whose iat predates it (or that has no iat) is rejected — without needing to know individual jtis.

Password hashing uses SHA-256 prehash → bcrypt, because bcrypt only consumes 72 bytes.

API keys are accepted via the X-API-Key header; the ApiKeyOrUser dependency allows JWT or API key on the same endpoint.

Rate limit & IP filter

Login limit (5 per 60 s/IP) and webhook limit (20 per 60 s/IP) are Redis-backed with in-memory fallback. IP filter via ALLOWED_IPS (CIDR list); proxy trust via TRUSTED_PROXIES (safe) instead of TRUST_PROXY_HEADERS (legacy).

Server Provisioning

New servers are onboarded via one-time tokens with a 24-hour TTL — since v0.23.2 FRP-independent inside the server module:

  1. Admin calls POST /api/servers/{server_id}/provision/token and receives {token, expiresAt, serverId, serverName}.
  2. The agent redeems it with POST /api/servers/{server_id}/provision/activate (header X-Provision-Token) and gets back: a fresh server API key (always), monitor agent key + URL (when the monitor service is reachable), frpc.toml + PKI bundle (when the server has FRP tunnels). Fields are null when the respective component is not configured — an unreachable monitor service does not block provisioning.
  3. Tokens are one-time: used_at is set, further activations fail.
  4. At runtime the agent syncs its frpc.toml via GET /api/frp/provision/{server_id}/config + config-hash (only when FRP is configured).

FRP — TOML Structure

modules/frp/config_generator.py renders the FRP configs straight from the DB models into TOML strings — no template engine, just string lists. Three generators:

FunctionTargetContent
generate_frps_toml(config)frps.toml (server)bindPort, optional vhostHTTPSPort/subDomainHost/maxPortsPerClient, optional dashboard block (webServer.* on 127.0.0.1), auth.method = "token" + token, plus a [transport.tls] server block.
generate_frpc_toml(config, tunnels, frpc_user)frpc.toml (agent)serverAddr/serverPort/user, token auth, client TLS block, then one [[proxies]] block per active tunnel.
generate_visitor_toml(config, tunnels, visitor_user)Visitor frpc.toml (admin PC)Aggregates all active STCP tunnels into one [[visitors]] block each (bindPort = visitor_port).

Only tunnels with enabled = true are rendered. Two tunnel types get special handling:

  • STCP (type = "stcp"): gets a secretKey and allowUsers (default ["ops-admin"], otherwise the passed allow_users). The matching [[visitors]] block sets serverName, serverUser (= agent user, because FRP registers proxies as {user}.{name}) and the same secretKey.
  • HTTPS (type = "https"): renders customDomains from the comma-separated custom_domains field.

extra_config (JSON) is appended per tunnel preserving types: strings quoted, bools as true/false, numbers raw.

frpc.toml (STCP example)
serverAddr = "frp.example.com"
serverPort = 7000
user = "k01-lnx1"

auth.method = "token"
auth.token = "<token>"

[transport.tls]
enable = true
trustedCaFile = "/etc/frp/pki/ca.crt"
certFile = "/etc/frp/pki/k01-lnx1.crt"
keyFile = "/etc/frp/pki/k01-lnx1.key"

[[proxies]]
name = "ssh"
type = "stcp"
localIP = "127.0.0.1"
localPort = 22
secretKey = "<secret>"
allowUsers = ["ops-admin"]

FRP PKI & TLS

modules/frp/pki.py manages the mTLS PKI for FRP using the cryptography library. Keys are RSA-4096, signatures SHA-256.

Master/published split (least privilege): the full PKI — CA private key and all client keys — lives server-private under PKI_DIR = FRP_PKI_DIR (default DATA_DIR/frp-pki, compose volume frp-pki, mounted on the server only). Only the subset frps needs is written into the volume shared with the internet-facing frps (FRP_CONFIG_DIR/pki) via _publish_frps_materials(): ca.crt, frps.crt, frps.key. frps verifies clients against ca.crt (public) and never needs the CA key. Deployments created before the split (where the whole PKI used to sit in the shared volume) are relocated once at startup by migrate_master_pki_out_of_shared() — the CA key and client keys leave the exposed volume without re-issuing the CA.

FunctionProducesProperties
generate_ca(common_name)ca.crt + ca.keySelf-signed, BasicConstraints(ca=True), KeyUsage with keyCertSign/crlSign. Valid 10 years (VALIDITY_DAYS_CA = 3650). Overwrites an existing CA.
generate_server_cert(server_addr)frps.crt + frps.keyCA-signed, SubjectAlternativeName = IP SAN for an IP address, otherwise DNS SAN; EKU SERVER_AUTH. Valid 1 year (VALIDITY_DAYS_CERT = 365).
generate_client_cert(client_name){client}.crt + {client}.keyCA-signed, EKU CLIENT_AUTH, 1 year. client_name is validated against ^[a-zA-Z0-9._-]+$ (max 64 chars, no ..) — path-traversal guard.

get_pki_status() reports CA/server-cert existence with expiry dates and lists all client certs (every *.crt except ca.crt/frps.crt). In the main.py lifespan, _ensure_pki() auto-generates the CA and server cert as soon as a FrpServerConfig exists; the server cert is re-issued when server_addr is no longer in the SANs.

The TLS wiring in frps.toml/frpc.toml is produced by the config_generator: the server block sets force = true with certFile/keyFile/trustedCaFile, the client block enable = true with trustedCaFile and — when a client user is set — the matching client certFile/keyFile. This yields mutual TLS against the same CA on both sides.

SQLAlchemy Models

Tables live in their respective modules (e.g. modules/users/models.py) and are created at startup via Base.metadata.create_all(). Idempotent column migrations run through _migrate_add_columns() with PRAGMA table_info.

  • User ↔ Server (n:m via user_servers), TokenBlacklist
  • Server (UUID PK, JSON tags, os_type) ↔ Connection (n:1, server_id nullable for standalone)
  • ApiKey with permission ∈ {read, read_write}
  • Hook (hook_type ∈ {webhook, event, schedule}, JSON event_triggers, Python script)
  • FrpServerConfig ↔ FrpTunnel (n:1) ↔ ProvisionToken (n:1, 24 h TTL, one-time)
  • Playbook (Ansible) with disk-backed YAML content

Event Bus & Scheduler

APScheduler runs in the same event loop as FastAPI. At startup load_all_scheduled_hooks() registers every active schedule hook; trigger types: predefined intervals (5m, 10m, 15m, 30m, 1h, 6h, 12h, 24h) or cron (5 fields).

The event bus in core/events.py distributes events through a ThreadPoolExecutor (4 workers, fire-and-forget). Active event types:

  • server.created, server.updated, server.deleted
  • connection.created, connection.updated, connection.deleted, connections.imported
  • user.created, user.deleted
  • api_key.created, api_key.deleted
  • frp.config.created, frp.config.updated, frp.config.deleted
  • frp.tunnel.created, frp.tunnel.updated, frp.tunnel.deleted
  • playbook.created, playbook.updated, playbook.deleted

Persistence

State splits across two locations:

  • PostgreSQL (compose volume postgres-data): two DBs adminhelper (server) and adminhelper_monitor (monitoring).
  • File system under DATA_DIR (default /app/data): bootstrap token, JWT secret, Ansible playbook files.
  • Master PKI (volume frp-pki, server-only): CA key + all client keys.
  • Shared volume frp-config (frps mounts it :ro at /etc/frp): frps.toml (0600) + the published cert subset.

The container starts as root only briefly to chown the mounted paths, then drops via gosu to a non-root user (uid 10001) — uvicorn, Alembic, cert generation and hook subprocesses all run unprivileged.

tree
/app/data/                         # DATA_DIR (server-only)
├── .bootstrap_token            # one-time setup-token hash (P0-6)
├── .secret_key                 # auto-gen JWT secret
└── ansible/playbooks/{id}/{file}.yaml

/app/frp-pki/                      # master PKI, server-only (volume frp-pki)
├── ca.crt, ca.key             # CA incl. PRIVATE key — never leaves the server
├── frps.crt, frps.key
└── {client}.crt, {client}.key

/app/frp-config/                   # shared with frps (volume frp-config, :ro for frps)
├── frps.toml
└── pki/
    └── ca.crt, frps.crt, frps.key   # published subset — NO ca.key, NO client keys

Tests

Pytest suite under server/tests/. Tests run against a real Postgres 17 via testcontainers (local) or services: postgres:17-alpine (CI). SAVEPOINT pattern in the db_session fixture: outer transaction + begin_nested + after_transaction_end listener — tests can call commit(), everything is rolled back at the end. Plus test_alembic_consistency.py as drift detector between Base.metadata and Alembic migrations.

bash
cd apps/server
uv run pytest -x                 # fail fast
uv run pytest --cov=app          # coverage
