Monitoring Stack

Monitoring is three layers cooperating: the AdminHelper server acts as a reverse proxy and forwards UI queries, the separate monitoring/ service holds checks/alerts/templates and runs scheduler-based pull checks, VictoriaMetrics stores the time-series data (90-day retention).

Data Flow

flow
[Agent]                                [Monitoring service]            [VictoriaMetrics]
   |                                          |                                |
   | POST /agent/{server_id}/report --------->|  agent_resources, service_     |
   |   (X-API-Key)                            |  process, docker_health,        |
   |                                          |  proxmox_backup, zfs_health,    |
   |                                          |  smart_health -> evaluate +     |
   |                                          |  format_line() ---------------->|
   |                                          |                                |
                                              |  APScheduler -> http / ping /  |
                                              |  tcp / agent_ping ------------>|
                                              |
[Web panel] -> /api/monitoring/* (admin) -> AdminHelper server -> /monitor/* (X-Internal-Key) -> Monitoring service

AdminHelper side: monitoring_proxy

There is no dedicated monitoring module on the AdminHelper server anymore; everything goes through server/app/modules/monitoring_proxy.py:

  • POST /api/monitoring/agent/{server_id}/report – public (X-API-Key), forwarded 1:1 to the service
  • GET/POST/PUT/DELETE /api/monitoring/{path:path} – admin-only, internally augmented with X-Internal-Key: $MONITOR_API_KEY
  • SSRF guard: allowed path prefixes are checks, alerts, log, metrics, status, templates, agent; path traversal (..) is blocked
  • Deleting a server in AdminHelper fires DELETE /servers/{id}/cleanup at the monitoring service – orphaned checks/alerts are pruned automatically

Push checks vs. pull checks

There are two classes of checks – what matters is who gathers the data:

ModeCheck typesTrigger
Push (agent)agent_resources, service_process, docker_health, proxmox_backup, zfs_health, smart_healthOn the next agent push (5-minute timer); APScheduler is not involved.
Pull (service)http, ping, tcp, agent_pingAPScheduler inside the monitoring service, by interval or cron.

Both paths converge in execute_check() → status update in MonitorState → format_line() → VictoriaMetrics write → process_alert() when fail_count ≥ consecutive_fails.

Push payload (agent → service)

The Go agent assembles the report in apps/agent/internal/monitor/report.go (BuildReport) and sends it as JSON via POST {url}/agent/{server_id}/report with the headers Content-Type: application/json and X-API-Key (PushReport). The server handler in routers/agent.py accepts the JSON as a free-form dict. Structure:

json
{
  "report_version": 2,
  "timestamp": "2026-06-04T10:00:00Z",
  "uptime_seconds": 864000,
  "resources": {
    "cpu_percent": 45.2,
    "memory_percent": 61.0,
    "memory_total_mb": 16000,
    "memory_used_mb": 9760,
    "load_1m": 0.8, "load_5m": 0.6, "load_15m": 0.5,
    "disks": [
      { "mount": "/", "fstype": "ext4", "percent": 72.0,
        "total_gb": 100, "used_gb": 72 }
    ],
    "temperatures": [
      { "sensor": "Package id 0", "temp_c": 54, "high": 80, "critical": 100 }
    ]
  },
  "systemd": { "all_services": [ /* unit, active_state, enabled_state */ ],
               "failed": [], "watched": [] },
  "services": [ { "name": "nginx", "running": true } ],
  "docker":  { /* plugin: docker_health */ },
  "proxmox": { /* plugin: proxmox_backup */ },
  "zfs":     { /* plugin: zfs_health */ },
  "smart":   [ { "device": "/dev/sda", "temp_c": 38,
                 "reallocated_sectors": 0, "pending_sectors": 0 } ]
}

Fields are optional: docker/proxmox/zfs/smart appear only when the agent detects the respective subsystem; VMs without sensors send no temperatures. services is a legacy key alongside systemd. The handler iterates over resources (CPU/RAM/load/disk/temp), uptime_seconds and smart, writes VictoriaMetrics lines from them (pseudo filesystems in EXCLUDED_FSTYPES are skipped), then evaluates the push checks configured for the server against the report. Response: {"status":"ok","metricsWritten":N,"checksUpdated":M}.

Checkers at a glance

Each checker returns a triple (status, message, metrics) with status ∈ {ok, warning, critical, unknown}. Pull checkers expose a run(config) method; agent/push checkers evaluate the push report directly in evaluate(config, report).

CheckerType(s)What it does
http.pyhttpHTTP/HTTPS request via httpx (method configurable, follows redirects). Compares the status code against expected_status, optionally searches for search_string in the body, measures response time. SSRF guard: URLs that resolve to private/reserved IP ranges are rejected.
ping.pypingICMP echo via ping -c 1 -W <timeout> as a subprocess. The target is validated against an allowlist regex (hostnames/IPs); parses the RTT from the output.
tcp.pytcpTCP connect via socket.create_connection((target, port)) with timeout. ok on a successful connect (with connect time), critical on timeout/refused.
agent.pyagent_ping, agent_resources, service_processAgentPingChecker checks (scheduler-driven) the age of the last push against stale_minutes. AgentResourcesChecker evaluates CPU/RAM/disk/temp against warn/crit thresholds. ServiceProcessChecker checks services – mode auto (failed + enabled-but-inactive systemd units, with an ignore list) or list (explicitly named services).
smart.pysmart_healthEvaluates SMART data from the push (ATA HDD/SSD and NVMe). Checks reallocated/pending sectors, NVMe critical_warning bits and temperatures against type-specific thresholds, among others. A missing smart key (VM or no smartctl) is ok, not an error.

The plugin checkers proxmox_backup, zfs_health and docker_health (in checkers/plugins.py) are likewise push-evaluated.

Auth modes

Both modes are implemented in core/auth.py as FastAPI dependencies:

  • require_agent (agent push): reads header X-API-Key. If the value matches the internal key exactly, it counts as __internal__ (access to all servers). Otherwise the key is SHA-256-hashed via MonitorAgentKey.hash_key() and compared against the hashed_key column in monitor_agent_keys – on a hit the dependency returns the associated server_id. A clear-text key is never stored; only the hash lives in the DB.
  • require_internal (AdminHelper ↔ service): reads header X-Internal-Key and compares it against the internal key. Its source is the env var MONITOR_API_KEY; if absent, a random key is generated at startup and persisted to $DATA_DIR/.api_key (core/config.py).

Agent key: issuance & server binding

Agent keys are issued per server (routers/admin.py):

  • POST /agent-keys/{server_id} creates a raw key with secrets.token_urlsafe(48), stores only its SHA-256 hash, and returns the raw key once in the response. If a key already exists for the server, it is deleted first (rotation = DELETE + POST). server_id is unique in the table, so exactly one key per server.
  • Both endpoints (POST/DELETE /agent-keys/{server_id}) are themselves guarded by require_internal – only the AdminHelper server may manage keys.
  • The push handler (routers/agent.py) additionally checks that the server_id derived from the key matches the server_id in the path; otherwise 403. A valid key for server A therefore cannot submit metrics for server B (__internal__ excepted).

Templates

MonitorTemplate bundles multiple CheckDef+AlertRuleDef entries under a stable template_def_id. POST /templates/{id}/assign/{server_id} rolls out the template; template_sync.py reconciles changes so server assignments stay drift-free.

VictoriaMetrics

External container (see docker-compose.yml), 90-day retention. The service writes in InfluxDB line protocol via core/victoria.py:

bash
monitor_agent_cpu_percent,server_id=srv123 value=45.2 1683000000

Reads use the standard Prometheus endpoints /api/v1/query and /api/v1/query_range. uPlot in the web panel renders the charts.

Ingest validation

The push endpoint enforces that the API key really belongs to the supplied server_id. A compromised agent key cannot forge metrics for other servers.

Local debugging

bash
# Run a single check on demand
docker compose exec monitoring \
    curl -s -H "X-Internal-Key: $MONITOR_API_KEY" \
    http://localhost:8080/checks/<id>/run

# Query VictoriaMetrics directly
docker compose exec monitoring curl -s \
    "http://victoria:8428/api/v1/query?query=monitor_agent_cpu_percent{server_id=\"srv123\"}"
