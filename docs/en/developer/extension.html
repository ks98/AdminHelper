Developer Guide

Browser Extension

The AdminHelper extension is intentionally minimal: Manifest V3, three script surfaces (popup, options, background service worker), no content scripts, no build pipeline. Plain vanilla JS, ~400 LoC.

Project Layout

tree
extension/
├── manifest.json        Manifest V3 declaration
├── background.js        Service worker (alarm + fetch + badge)
├── popup.html/.js/.css  Toolbar popup (connections dropdown)
├── options.html/.js/.css Settings page (server URL + API key)
└── icons/               32/64/128 PNG icons

Manifest V3

json
{
  "manifest_version": 3,
  "name": "AdminHelper",
  "version": "0.24.0",
  "permissions": ["storage", "tabs", "alarms"],
  "action":    { "default_popup": "popup.html" },
  "options_page": "options.html",
  "background": { "service_worker": "background.js" }
}

Only three permissions:

  • storage – chrome.storage.sync for server URL + API key, chrome.storage.local for the connections cache.
  • tabs – open new tabs from the popup.
  • alarms – periodic refresh every 5 minutes.

No host_permissions on purpose – the server is freely configurable and is implicitly allowed on first fetch.

Service-Worker Lifecycle

Manifest V3 forbids persistent background pages. The worker wakes on demand:

  • onInstalled / onStartup: create the alarm.
  • onAlarm: trigger the fetch.
  • Event ends → worker is suspended. All state must live in chrome.storage.local.
javascript
const ALARM_NAME = 'refresh';
const REFRESH_INTERVAL_MINUTES = 5;

chrome.runtime.onInstalled.addListener(() => {
  chrome.alarms.create(ALARM_NAME, { periodInMinutes: REFRESH_INTERVAL_MINUTES });
});

chrome.alarms.onAlarm.addListener(async (alarm) => {
  if (alarm.name !== ALARM_NAME) return;
  await fetchAndCache();
});

Fetch & Cache

Each alarm calls GET /api/connections?api_key=…, filters the list to kind === 'web', and stores the result in chrome.storage.local.cachedConnections:

javascript
const data = await res.json();
const conns = Array.isArray(data) ? data : (data.connections ?? []);
const webConns = conns.filter(c => c.kind === 'web');

await chrome.storage.local.set({
  cachedConnections: { connections: webConns, ts: Date.now() }
});
updateBadge(webConns.length);

The popup reads exclusively from cache – even offline, the list remains visible (with a stale ts).

Badge Counter

The web-connection count appears as a badge on the toolbar icon (chrome.action.setBadgeText). Empty at 0, cyan background (#38bdf8) for visibility on both light and dark Chrome themes.

Popup & Options

The popup has two states: with no config set it shows the setup view (server URL + API key inline). Otherwise the main view with search, a list/tag view toggle, and a footer (connection count + relative time of the last refresh). The chosen view is persisted in chrome.storage.local.viewMode; open/collapsed tag groups live in sessionStorage. Filtering runs live across name, URL, notes, and tags.

The options page offers a Test connection button next to „Save": it calls the same GET /api/connections and reports the total connection count plus the number of web connections, without saving anything. Changes to serverUrl/apiKey are picked up via chrome.storage.onChanged – an open popup then re-initialises automatically.

Security Model

API key stored in clear text

chrome.storage.sync is not a secret store. The API key is readable by every extension in the same Chrome profile. Always scope it to web_only – no connection CRUD, no hook access, no server-admin rights.

Web Filter

Both the background worker and the popup filter the server response to c.kind === 'web' – only web connections are cached and rendered. Connections of other kinds (RDP, SSH, VNC) carry a different kind and never reach the extension in the first place – no URI-handler hijacking surface. A click opens the connection via chrome.tabs.create({ url }) with the unmodified url from the server.

Local Testing

  1. Chrome → chrome://extensions → enable Developer mode.
  2. "Load unpacked" → pick the extension/ folder.
  3. Click the icon → open Options → set server + API key.
  4. Service-worker DevTools: chrome://extensions → Details → "Inspect service worker".

Firefox Compatibility

Firefox supports Manifest V3 from version 115 onwards. Only difference: browser.* vs. chrome.*. Currently only Chrome APIs are used – a Firefox port would require adding a browser-polyfill shim.
