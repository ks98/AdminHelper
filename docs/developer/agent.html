Go-Agent

Unified Go-Agent unter agent-go/. Ersetzt die alten Shell- und Python-Agents und vereint Monitoring, FRP-Client-Setup und Service-Installer in einer einzigen Binary, die gleichermaßen unter Linux (systemd) und Windows (Service-Manager) läuft.

Projekt-Layout

tree
agent-go/
├── cmd/adminhelper-agent/
│   ├── main.go             Cobra-Root mit fünf Subkommando-Gruppen
│   ├── handlers.go         Implementierung der Subcommands
│   └── run.go              Standalone-Modus ohne Systemd
├── internal/
│   ├── config/             paths_linux.go / paths_windows.go, Key=Value-Parser
│   ├── provision/          Activate über /api/servers/{id}/provision/activate, orchestriert frpc.Apply + monitor.Init
│   ├── frpc/               Apply (Datei + Service) + Sync, ZIP-Slip-sicher
│   ├── monitor/            Collector + Plugins (10 Dateien, OS-spezifische Splits)
│   └── service/            service_linux.go (systemd) / service_windows.go (SCM)
├── systemd/                .service + .timer Units
├── deb/                    Debian-Control-Dateien
├── rpm/                    Spec-File
├── Makefile                build-linux, build-windows, deb, rpm
├── build-deb.sh
├── build-rpm.sh
└── go.mod                  Go 1.24

Cobra-Subcommands

Sechs Top-Level-Befehle, davon drei mit Sub-Subcommands:

CommandZweck
versionVersion + Git-Commit ausgeben (gesetzt über -ldflags).
runStandalone-Modus: gesamten Zyklus einmal durchspielen (Monitoring-Push + FRP-Sync).
provisionErst-Setup: Token gegen /api/servers/{id}/provision/activate einlösen, Server-API-Key + optional Monitor-Key + optional FRP-Bundle in einem Aufruf installieren.
frpc syncPeriodischer Config-Sync per Hash-Vergleich; schreibt nur, wenn sich der Server-seitige Hash geändert hat.
monitor initMonitor-Config-Datei anlegen (URL, API-Key, Service-Liste).
monitor pushEinmaligen Metriken-Sammel- und -Push-Lauf ausführen.
service install / uninstallsystemd-Unit (Linux) oder Windows-Dienst (SCM) installieren bzw. entfernen.

Konfiguration

Zwei Konfigurationsdateien im Key=Value-Format (Anführungszeichen erlaubt). Pfade pro OS:

  • Linux: /etc/frp/frp.conf und /etc/adminhelper/monitor.conf
  • Windows: unter %ProgramData%\adminhelper\

FrpcConfig-Felder: ADMINHELPER_URL, API_KEY, SERVER_ID, CACERT, INSECURE. MonitorConfig ergänzt zusätzlich MONITOR_URL und SERVICES (kommagetrennt).

Monitor-Collector

Zentrale Sammlung in internal/monitor/collector.go. Alle Collector-Funktionen liefern map[string]any, das beim Push direkt JSON-serialisiert wird:

  • collectCPU() – Last und Per-Core-Utilization
  • collectMemory() – Total / Used / Free / Swap
  • collectLoad() – load_1m, load_5m, load_15m
  • collectDisks() – pro Mountpoint mit FS-Type, Belegung, Inodes
  • collectUptime()
  • collectTemperatures() – lm-sensors / Windows-WMI
  • collectServiceHealth() + collectWatchedServices(names)
  • collectDocker() – Container-Liste über Docker-Socket
  • collectZFS() – zpool status -p
  • collectProxmox() – pveversion + API (nur auf PVE-Hosts)
  • collectSmart() – smartctl --scan-open + --all --json=cv; ATA- und NVMe-Pfade getrennt geparst

Alle Daten landen unter dem JSON-Key resources in POST /agent/{server_id}/report am Monitoring-Service.

systemd-Timer mit Jitter

Statt cron nutzt der Agent systemd mit OnBootSec=60, OnUnitActiveSec=300 (5 min Intervall) und RandomizedDelaySec=30. Der Jitter verteilt die Last aller Agents auf das 5-Minuten-Fenster und verhindert Thundering-Herd-Requests am Server.

Build

bash
cd agent-go
make build-linux       # bin/adminhelper-agent       (ELF amd64)
make build-windows     # bin/adminhelper-agent.exe   (PE amd64)

VERSION=0.23.1 bash build-deb.sh
VERSION=0.23.1 bash build-rpm.sh

Das DEB-Paket installiert /usr/local/bin/adminhelper-agent sowie /usr/bin/frpc (in der CI heruntergeladen) und legt die Systemd-Unit + Timer an.

DEB/RPM Replaces

Die DEB/RPM-Pakete enthalten Replaces- und Conflicts-Einträge für die Legacy-Pakete srm-frpc-client, srm-monitor-agent und srm-agent. Ein apt install adminhelper-agent migriert saubere Installationen automatisch.
