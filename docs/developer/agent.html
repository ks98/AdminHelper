Go-Agent

Unified Go-Agent unter apps/agent/. Ersetzt die alten Shell- und Python-Agents und vereint Monitoring, FRP-Client-Setup und Service-Installer in einer einzigen Binary, die gleichermaßen unter Linux (systemd) und Windows (Service-Manager) läuft.

Projekt-Layout

tree
apps/agent/
├── cmd/adminhelper-agent/
│   ├── main.go             Cobra-Root mit fünf Subkommando-Gruppen
│   ├── handlers.go         Implementierung der Subcommands
│   └── run.go              Standalone-Modus ohne Systemd
├── internal/
│   ├── config/             paths_linux.go / paths_windows.go, Key=Value-Parser
│   ├── provision/          Activate über /api/servers/{id}/provision/activate, orchestriert frpc.Apply + monitor.Init
│   ├── frpc/               Apply (Datei + Service) + Sync, ZIP-Slip-sicher
│   ├── monitor/            Collector + Plugins (10 Dateien, OS-spezifische Splits)
│   └── service/            service_linux.go (systemd) / service_windows.go (SCM)
├── systemd/                .service + .timer Units
├── deb/                    Debian-Control-Dateien
├── rpm/                    Spec-File
├── Makefile                build-linux, build-windows, deb, rpm
├── build-deb.sh
├── build-rpm.sh
└── go.mod                  Go 1.24

Cobra-Subcommands

Sechs Top-Level-Befehle, davon drei mit Sub-Subcommands:

CommandZweck
versionVersion + Git-Commit ausgeben (gesetzt über -ldflags).
runStandalone-Modus: gesamten Zyklus einmal durchspielen (Monitoring-Push + FRP-Sync).
provisionErst-Setup: Token gegen /api/servers/{id}/provision/activate einlösen, Server-API-Key + optional Monitor-Key + optional FRP-Bundle in einem Aufruf installieren.
frpc syncPeriodischer Config-Sync per Hash-Vergleich; schreibt nur, wenn sich der Server-seitige Hash geändert hat.
monitor initMonitor-Config-Datei anlegen (URL, API-Key, Service-Liste).
monitor pushEinmaligen Metriken-Sammel- und -Push-Lauf ausführen.
service install / uninstallsystemd-Unit (Linux) oder Windows-Dienst (SCM) installieren bzw. entfernen.

Konfiguration

Zwei Konfigurationsdateien im Key=Value-Format (Anführungszeichen erlaubt). Pfade pro OS:

  • Linux: /etc/frp/adminhelper.conf und /etc/adminhelper/monitor.conf
  • Windows: unter %ProgramData%\AdminHelper\

FrpcConfig-Felder: ADMINHELPER_URL, API_KEY, SERVER_ID, CACERT, INSECURE. MonitorConfig ergänzt zusätzlich MONITOR_URL und SERVICES (kommagetrennt).

Monitor-Collector

Zentrale Sammlung in internal/monitor/collector.go. Alle Collector-Funktionen liefern map[string]any, das beim Push direkt JSON-serialisiert wird:

  • collectCPU() – Last und Per-Core-Utilization
  • collectMemory() – Total / Used / Free / Swap
  • collectLoad() – load_1m, load_5m, load_15m
  • collectDisks() – pro Mountpoint mit FS-Type, Belegung, Inodes
  • collectUptime()
  • collectTemperatures() – lm-sensors / Windows-WMI
  • collectServiceHealth() + collectWatchedServices(names)
  • collectDocker() – Container-Liste über Docker-Socket
  • collectZFS() – zpool status -p
  • collectProxmox() – pveversion + API (nur auf PVE-Hosts)
  • collectSmart() – smartctl --scan-open + --all --json=cv; ATA- und NVMe-Pfade getrennt geparst

Alle Daten landen unter dem JSON-Key resources in POST /agent/{server_id}/report am Monitoring-Service.

Push-Verhalten

  • Retry: Schlägt der Metrik-Push fehl, macht PushReport (internal/monitor/report.go) genau einen Retry nach 10 s – ein verlorener Push kostet sonst ein volles 5-Minuten-Fenster.
  • Inventar-Drosselung: Das all_services-Inventar (100–300 größtenteils statische Units) wird nur gesendet, wenn sich sein SHA-256-Hash ändert oder der letzte Voll-Send >1 h her ist; sonst fehlt der Key im Report. Der Drossel-Zustand liegt in .inventory-state.json neben monitor.conf (Linux /etc/adminhelper/, Windows %ProgramData%\AdminHelper\) und wird erst nach erfolgreichem Push persistiert. Watched Services sowie failed/enabled_inactive gehen weiterhin bei jedem Push mit.
  • Docker-Timeouts: Jeder Docker-CLI-Aufruf des docker_health-Plugins hat ein 10-s-Timeout – ein hängender Daemon blockiert den Push nicht.

Plattform-Splits: Linux vs. Windows

Collector, Service-Aktivierung und Pfade sind über Go-Build-Tags (//go:build linux / //go:build windows) getrennt. Die folgende Tabelle listet je Funktion das konkrete Plattform-Verhalten.

FunktionLinuxWindows
collectServiceHealth()
services_*.go		 systemctl list-units + list-unit-files + list-units --state=failed; füllt failed, enabled_inactive und all_services (mit active_state / enabled_state). sc query state= all; Windows-Status wird auf das systemd-Format gemappt (RUNNING→active, STOPPED/PAUSED→inactive, *_PENDING→activating). failed bleibt leer, enabled_state ist immer unknown.
collectWatchedServices(names)
services_*.go		 systemctl is-active; PID via systemctl show -p MainPID. sc query (Suche nach RUNNING); PID via sc queryex.
collectTemperatures()
temperatures.go (gemeinsam)		 Eine gemeinsame Datei ohne Build-Tag: liest gopsutil/v4/sensors.SensorsTemperatures(). Auf beiden Plattformen identischer Code; gopsutil greift unter Linux auf hwmon/lm-sensors, unter Windows auf WMI zu. Sensoren mit Temperatur ≤ 0 werden verworfen; ohne Sensoren wird nil zurückgegeben (kein Fehler).
collectLoad()
collector.go (gemeinsam)		 load.Avg() liefert load_1m/load_5m/load_15m. Liefert nil (kein Load-Average) — Guard runtime.GOOS == "windows".
collectDisks()
collector.go (gemeinsam)		 Gemeinsamer Code via gopsutil/disk. Die Filterlisten skipFSTypes (squashfs, tmpfs, overlay, proc…) und skipMountPrefixes (/sys, /proc, /dev/, /snap…) sind Linux-spezifisch; unter Windows treffen sie nicht zu und filtern nichts.
enableMonitorService()
platform_*.go		 systemctl enable --now adminhelper-agent.timer. No-op: gibt den Hinweis aus, adminhelper-agent service install auszuführen (SCM-Integration als TODO offen).
service.Install()
service_*.go		 Schreibt .service (Type=oneshot, ExecStart=<exe> run) + .timer (OnBootSec=60, OnUnitActiveSec=300, Persistent=true) nach /etc/systemd/system, dann daemon-reload + enable --now auf den Timer. Unit-Name adminhelper-agent. Kopiert die Binary nach %ProgramFiles%\AdminHelper\, registriert via sc create (binPath=<exe> run, start= auto, obj= LocalSystem), setzt Recovery via sc failure (Neustart nach 60 s) und startet mit sc start. Dienst-Name AdminHelper-Agent — ein dauerhaft laufender Dienst, kein Timer.
service.Uninstall()
service_*.go		 systemctl stop/disable für Timer und Service, löscht die Unit-Dateien, daemon-reload. sc stop + sc delete auf AdminHelper-Agent.
FrpDir()
paths_*.go		 /etc/frp %ProgramData%\AdminHelper\frp
MonitorDir()
paths_*.go		 /etc/adminhelper %ProgramData%\AdminHelper

systemd-Timer mit Jitter

Statt cron nutzt der Agent systemd mit OnBootSec=60, OnUnitActiveSec=300 (5 min Intervall) und RandomizedDelaySec=30. Der Jitter verteilt die Last aller Agents auf das 5-Minuten-Fenster und verhindert Thundering-Herd-Requests am Server.

Build

bash
cd apps/agent
make build-linux       # bin/adminhelper-agent       (ELF amd64)
make build-windows     # bin/adminhelper-agent.exe   (PE amd64)

VERSION=0.26.0 bash build-deb.sh
VERSION=0.26.0 bash build-rpm.sh

DEB- und RPM-Paket installieren /usr/bin/adminhelper-agent sowie /usr/bin/frpc (in der CI heruntergeladen) und legen die Systemd-Unit + Timer an.

DEB/RPM Replaces

Die DEB/RPM-Pakete enthalten Replaces- und Conflicts-Einträge für die Legacy-Pakete srm-frpc-client, srm-monitor-agent und srm-agent. Ein apt install adminhelper-agent migriert saubere Installationen automatisch.
