Entwickler-Handbuch

Übersicht

Das Entwickler-Handbuch dokumentiert Architektur, Code-Struktur und API von AdminHelper. Es richtet sich an alle, die am Code mitschreiben oder AdminHelper in ihr eigenes System integrieren wollen.

Tech-Stack auf einen Blick

Backend

Python 3.12, FastAPI, SQLAlchemy 2.0 + Alembic, APScheduler, PostgreSQL 17 via psycopg3. Strenge Pydantic-Schemas.

Web-Panel

Svelte 5 Runes, TypeScript strict, Vite, Hash-Router, uPlot, Playwright-E2E.

Desktop

Tauri 2 + Rust, Svelte 5 Frontend, keyring-Secrets, RDP-Race-Guard mit UUID.

Agent

Go 1.24, Cobra CLI, sqlx, native systemd + windowssvc-Integration.

Lokale Entwicklung

Jedes Unterprojekt ist eigenständig, kein Monorepo-Toolchain:

bash
# Server
cd apps/server && uv sync && uv run uvicorn app.main:app --reload --port 8080

# Web-Panel (mit Proxy zum laufenden Server)
cd apps/web && npm ci && npm run dev

# Desktop-Client (startet Tauri-Dev mit Auto-Reload)
cd apps/desktop/ui && npm ci
cd ../desktop && cargo tauri dev

# Go-Agent
cd apps/agent && go run ./cmd/adminhelper-agent version

Einheitliche PKI & mTLS

AdminHelper betreibt eine eigene interne PKI als Fundament für mTLS über alle Zugänge. Das vollständige, kanonische Modell (Entscheidungen D1–D11, Trust, Lifecycle, Backup) steht in ADR 0001; der Umsetzungsstand in ADR 0002. Kurzüberblick:

  • Vertrauenskette: eine Root → gescopte Intermediates tunnel (frps + Agenten), access (Mensch: Desktop/Browser), internal (Phase B). Clients pinnen die Root/CA, nicht ein Leaf — Server-Certs sind frei rotierbar (D2).
  • ca-issuer (eigener Container) ist die einzige Signier-Capability (D6): erzeugt die Hierarchie beim First-Boot, hält die Online-Intermediates, signiert per /enroll (Einmal-Token) und /renew (vorgelegtes Cert). Der Root-Key liegt kalt + passphrase-verschlüsselt (D7).
  • gateway (nginx) ist die einzige öffentliche TLS-Kante (`:443` Daten, `:8444` Enroll). Es terminiert TLS/mTLS und reicht die verifizierte Identität als X-Client-*-Header an die internen Dienste — uvicorn sieht das Cert nicht selbst (V1). server/ca-issuer haben keinen Host-Port, der Header ist daher unfälschbar.
  • Enrollment (ein Muster, drei Clients): Einmal-Token → On-Device-ECDSA-Key + CSR → Issuer signiert (Identität issuer-diktiert, CSR-Subject kosmetisch). Go-Agent (beim Provisioning), Desktop (nach Login), frps/Tunnel-Certs. Native Clients erneuern automatisch bei ~50 % Laufzeit; der Browser bekommt ein langlebiges Cert als PKCS12-Export aus dem Desktop (D5).
  • Per-Route-Scope: access = Mensch, tunnel = Agent (D8). Permissiv per Default (`MTLS_ENFORCE=false`): ein Scope-Mismatch wird geloggt, der Request läuft durch. Das Scharfschalten auf CERT_REQUIRED ist MTLS_ENFORCE=true — umgesetzt + end-to-end verifiziert (A8), eine bewusste Operator-Entscheidung.
  • Backup-Kronjuwel ist das ca-pki-Volume (Root + Intermediates) — scripts/backup.sh/restore.sh sichern es; die regenerierbaren Cert-Volumes bleiben bewusst draußen (ADR 0001 §5).
