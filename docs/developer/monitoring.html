Monitoring-Stack

Monitoring ist ein Zusammenspiel aus drei Ebenen: der AdminHelper-Server agiert als Reverse-Proxy und reicht UI-Queries weiter, der separate monitoring/-Service hält Checks/Alerts/Templates und führt Scheduler-Checks aus, VictoriaMetrics speichert die Zeitreihen (Retention 90 Tage).

Daten-Fluss

flow
[Agent]                                [Monitoring-Service]            [VictoriaMetrics]
   |                                          |                                |
   | POST /agent/{server_id}/report --------->|  agent_resources, service_     |
   |   (X-API-Key)                            |  process, docker_health,        |
   |                                          |  proxmox_backup, zfs_health,    |
   |                                          |  smart_health -> evaluate +     |
   |                                          |  format_line() ---------------->|
   |                                          |                                |
                                              |  APScheduler -> http / ping /  |
                                              |  tcp / agent_ping ------------>|
                                              |
[Web-Panel] -> /api/monitoring/* (admin) -> AdminHelper-Server -> /monitor/* (X-Internal-Key) -> Monitoring-Service

AdminHelper-seitig: monitoring_proxy

Im AdminHelper-Server gibt es kein eigenes Monitoring-Modul mehr; alles läuft durch server/app/modules/monitoring_proxy.py:

  • POST /api/monitoring/agent/{server_id}/report – öffentlich (X-API-Key), reicht 1:1 an den Service
  • GET/POST/PUT/DELETE /api/monitoring/{path:path} – admin-only, intern erweitert um X-Internal-Key: $MONITOR_API_KEY
  • SSRF-Schutz: erlaubte Pfad-Prefixe sind checks, alerts, log, metrics, status, templates, agent; Path-Traversal (..) wird blockiert
  • Beim Server-Delete im AdminHelper wird DELETE /servers/{id}/cleanup am Monitoring-Service gefeuert – verwaiste Checks/Alerts werden so automatisch aufgeräumt

Push-Checks vs. Pull-Checks

Es gibt zwei Klassen von Checks – entscheidend ist, wer die Daten beschafft:

ModusCheck-TypenTrigger
Push (Agent)agent_resources, service_process, docker_health, proxmox_backup, zfs_health, smart_healthBeim nächsten Agent-Push (5-Min-Timer); APScheduler ist nicht beteiligt.
Pull (Service)http, ping, tcp, agent_pingAPScheduler im Monitoring-Service nach interval oder Cron.

Beide Pfade landen am Ende in execute_check() → Status-Update in MonitorState → format_line() → VictoriaMetrics-Write → ggf. process_alert(), sobald fail_count ≥ consecutive_fails.

Push-Payload (Agent → Service)

Der Go-Agent baut den Report in agent-go/internal/monitor/report.go (BuildReport) und sendet ihn als JSON per POST {url}/agent/{server_id}/report mit den Headern Content-Type: application/json und X-API-Key (PushReport). Der Server-Handler in routers/agent.py nimmt das JSON als freies dict entgegen. Struktur:

json
{
  "report_version": 2,
  "timestamp": "2026-06-04T10:00:00Z",
  "uptime_seconds": 864000,
  "resources": {
    "cpu_percent": 45.2,
    "memory_percent": 61.0,
    "memory_total_mb": 16000,
    "memory_used_mb": 9760,
    "load_1m": 0.8, "load_5m": 0.6, "load_15m": 0.5,
    "disks": [
      { "mount": "/", "fstype": "ext4", "percent": 72.0,
        "total_gb": 100, "used_gb": 72 }
    ],
    "temperatures": [
      { "sensor": "Package id 0", "temp_c": 54, "high": 80, "critical": 100 }
    ]
  },
  "systemd": { "all_services": [ /* unit, active_state, enabled_state */ ],
               "failed": [], "watched": [] },
  "services": [ { "name": "nginx", "running": true } ],
  "docker":  { /* Plugin: docker_health */ },
  "proxmox": { /* Plugin: proxmox_backup */ },
  "zfs":     { /* Plugin: zfs_health */ },
  "smart":   [ { "device": "/dev/sda", "temp_c": 38,
                 "reallocated_sectors": 0, "pending_sectors": 0 } ]
}

Felder sind optional: docker/proxmox/zfs/smart erscheinen nur, wenn der Agent das jeweilige Subsystem erkennt; VMs ohne Sensoren liefern keine temperatures. services ist ein Legacy-Key parallel zu systemd. Der Handler iteriert über resources (CPU/RAM/Load/Disk/Temp), uptime_seconds und smart, schreibt daraus VictoriaMetrics-Zeilen (Pseudo-Dateisysteme aus EXCLUDED_FSTYPES werden übersprungen) und wertet anschließend die fuer den Server konfigurierten Push-Checks gegen den Report aus. Antwort: {"status":"ok","metricsWritten":N,"checksUpdated":M}.

Checker im Überblick

Jeder Checker liefert ein Tripel (status, message, metrics) mit status ∈ {ok, warning, critical, unknown}. Pull-Checker haben eine run(config)-Methode; Agent-/Push-Checker werten in evaluate(config, report) direkt den Push-Report aus.

CheckerTyp(en)Was er tut
http.pyhttpHTTP/HTTPS-Request via httpx (Methode konfigurierbar, folgt Redirects). Vergleicht Statuscode mit expected_status, optional Suche nach search_string im Body, misst Antwortzeit. SSRF-Schutz: URLs, die auf private/reservierte IP-Bereiche auflösen, werden abgewiesen.
ping.pypingICMP-Echo via ping -c 1 -W <timeout> als Subprozess. Ziel wird gegen ein Allowlist-Regex (Hostnamen/IPs) validiert; parst die RTT aus dem Output.
tcp.pytcpTCP-Connect via socket.create_connection((target, port)) mit Timeout. ok bei erfolgreichem Connect (mit Connect-Zeit), critical bei Timeout/Refused.
agent.pyagent_ping, agent_resources, service_processAgentPingChecker prüft (scheduler-getrieben) das Alter des letzten Push gegen stale_minutes. AgentResourcesChecker wertet CPU/RAM/Disk/Temp gegen Warn-/Crit-Thresholds aus. ServiceProcessChecker prüft Dienste – Modus auto (failed + enabled-but-inactive systemd-Units, mit Ignore-Liste) oder list (explizit benannte Dienste).
smart.pysmart_healthWertet SMART-Daten aus dem Push aus (ATA-HDD/SSD und NVMe). Prüft u. a. reallocated/pending Sectors, NVMe-critical_warning-Bits und Temperaturen gegen typ-spezifische Thresholds. Fehlt der smart-Key (VM oder kein smartctl), ist das ok, kein Fehler.

Die Plugin-Checker proxmox_backup, zfs_health und docker_health (in checkers/plugins.py) sind ebenfalls Push-ausgewertet.

Auth-Modi

Beide Modi sind in core/auth.py als FastAPI-Dependencies implementiert:

  • require_agent (Agent-Push): liest Header X-API-Key. Stimmt der Wert exakt mit dem internen Key überein, gilt er als __internal__ (Zugriff auf alle Server). Andernfalls wird der Key per MonitorAgentKey.hash_key() SHA-256-gehasht und gegen die Spalte hashed_key in monitor_agent_keys verglichen – bei Treffer liefert die Dependency die zugehörige server_id zurück. Es wird also nie ein Klartext-Key gespeichert; in der DB liegt nur der Hash.
  • require_internal (AdminHelper ↔ Service): liest Header X-Internal-Key und vergleicht ihn gegen den internen Key. Quelle ist die Env-Variable MONITOR_API_KEY; fehlt sie, wird beim Start ein zufälliger Key generiert und in $DATA_DIR/.api_key persistiert (core/config.py).

Agent-Key: Vergabe & Server-Bindung

Agent-Keys werden pro Server vergeben (routers/admin.py):

  • POST /agent-keys/{server_id} erzeugt mit secrets.token_urlsafe(48) einen Raw-Key, speichert nur dessen SHA-256-Hash und gibt den Raw-Key einmalig in der Antwort zurück. Existiert bereits ein Key für den Server, wird er vorher gelöscht (Rotation = DELETE + POST). server_id ist in der Tabelle unique, also genau ein Key pro Server.
  • Beide Endpunkte (POST/DELETE /agent-keys/{server_id}) sind selbst über require_internal geschützt – nur der AdminHelper-Server darf Keys verwalten.
  • Im Push-Handler (routers/agent.py) wird zusätzlich geprüft, dass die aus dem Key abgeleitete server_id der server_id im Pfad entspricht; sonst 403. Ein gültiger Key für Server A kann also keine Metriken für Server B einliefern (__internal__ ausgenommen).

Templates

MonitorTemplate bündelt mehrere CheckDef+AlertRuleDef-Einträge unter einer stabilen template_def_id. POST /templates/{id}/assign/{server_id} rollt das Template aus; template_sync.py reconciled bei Änderungen, sodass die Server-Zuweisungen drift-frei bleiben.

VictoriaMetrics

Externer Container (siehe docker-compose.yml), Retention 90 Tage. Der Service schreibt im InfluxDB-Line-Protocol über core/victoria.py:

bash
monitor_agent_cpu_percent,server_id=srv123 value=45.2 1683000000

Reads laufen über den Standard-Prometheus-Endpoint /api/v1/query bzw. /api/v1/query_range. Im Web-Panel zeichnet uPlot daraus die Charts.

Ingest-Validierung

Der Push-Endpunkt prüft, dass der API-Key wirklich zur angegebenen server_id gehört. Ein kompromittierter Agent-Key kann also keine fremden Server-Metriken fälschen.

Lokales Debugging

bash
# Einzelnen Check on-demand ausführen

# VictoriaMetrics direkt fragen
docker compose exec monitoring curl -s \
