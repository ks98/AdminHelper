CI/CD & Release

GitHub Actions baut und testet alle Komponenten, pusht die Server- und Monitoring-Images in die GitHub Container Registry (ghcr.io) und legt bei einem Versions-Tag ein Draft-Release mit Desktop- und Agent-Artefakten an.

Workflows

WorkflowAufgabe
ci.ymlLint & Tests aller Komponenten: Server-pytest (Postgres-Service, Python 3.12/3.13), Monitoring-pytest, Go-Agent (gofmt/vet/test), Desktop-UI & Web-Panel (svelte-check/eslint/vitest, Playwright-E2E), Extension (node --test), Rust/Tauri (fmt/clippy/test)
docker.ymlServer- + Monitoring-Image bauen & nach ghcr.io/ks98/adminhelper/{server,monitoring} pushen
release.ymlDesktop-DEB/RPM/Binary, Go-Agent-DEB/RPM + Linux/Windows-Binaries, Windows-MSI (best-effort), SHA256SUMS, Draft-Release

Trigger-Matrix

EventWas läuft
Pull Requestci.yml (muss grün sein)
Push auf mainci.yml + docker.yml (Image-Tags :main + :sha-…)
Tag v*docker.yml (Tags :0.25.0 + :latest + :sha-…) + release.yml (Draft-Release mit Artefakten)

Container-Images (ghcr.io)

Getaggt wird über docker/metadata-action. :latest entsteht nur auf einem (Nicht-Prerelease-)Semver-Tag – nicht bei jedem main-Push:

text
ghcr.io/ks98/adminhelper/server:0.24.0      # Release-Tag
ghcr.io/ks98/adminhelper/server:latest      # letzter Semver-Release
ghcr.io/ks98/adminhelper/server:main        # letzter main-Build
ghcr.io/ks98/adminhelper/server:sha-abc1234 # pro Commit
# analog: ghcr.io/ks98/adminhelper/monitoring:…
Package-Sichtbarkeit

Neu erzeugte ghcr-Pakete sind privat. Nach dem ersten erfolgreichen docker.yml-Lauf müssen server und monitoring in den GitHub-Package-Einstellungen einmalig auf public gestellt werden – sonst schlägt docker compose pull für externe Nutzer mit unauthorized/manifest-unknown fehl.

Release-Artefakte

release.yml läuft auf Tag v* und hängt die gebauten Artefakte an ein Draft-Release (nicht veröffentlicht, bis ein Mensch „Publish" drückt):

  • Desktop-Client: .deb + .rpm (Tauri) + ELF-Binary
  • Go-Agent: .deb + .rpm + linux/windows-Binaries
  • Desktop-Client Windows-.msi (best-effort – blockiert das Release nicht, wenn der Windows-Build fehlschlägt)
  • SHA256SUMS über alle Assets

Die Agent-Version wird aus dem Tag abgeleitet (${GITHUB_REF_NAME#v}) und per -ldflags "-X main.version=$VERSION" in die Binaries geschrieben.

Version-Stellen

Bei einem Release muss die neue Version an mehreren Stellen parallel gesetzt werden – der Tag allein reicht nicht:

  • apps/desktop/src-tauri/tauri.conf.json + Cargo.toml + Cargo.lock – Tauri-Bundle
  • apps/desktop/ui/package.json + apps/web/package.json (+ jeweils package-lock.json) – Vite-Builds
  • apps/desktop/ui/src/components/AppShell.svelte – Sidebar-Badge
  • extension/manifest.json – Browser-Extension
  • apps/agent/build-deb.sh + build-rpm.sh – Default-VERSION (CI überschreibt sie aus dem Tag)
  • docs/**/*.html – Footer-<span>v0.24.0</span> + Landing-„Was ist neu" + Code-Samples
  • README.md – Agent-DEB-Install-Befehl
  • CHANGELOG.md – neuer Abschnitt am Anfang + Link-Ref

Der Server selbst hält keine eigene version – das Backend wird über den Docker-Image-Tag identifiziert.

Release-Checkliste

  1. Versionen in den oben gelisteten Dateien auf den Zielwert hochziehen.
  2. Footer-Versionen in docs/**/*.html aktualisieren (Footer-Span + Landing-„Was ist neu" + Code-Samples).
  3. CHANGELOG.md pflegen (Keep-a-Changelog: neuer Abschnitt + Link-Ref).
  4. Commit & Tag: git commit -am "release: v0.24.0" && git tag v0.24.0.
  5. git push --follow-tags.
  6. docker.yml + release.yml beobachten; nach dem ersten Image-Push die ghcr-Pakete (einmalig) auf public stellen.
  7. Im GitHub-Draft-Release die Notes prüfen, Pflicht-Assets gegenchecken und „Publish" drücken.
Draft statt Auto-Publish

Das Release wird bewusst als Draft erzeugt und der Release-Job ist erfolgs-gated auf die Pflicht-Linux-Builds. So lässt sich vor der Veröffentlichung prüfen, ob alle Pflicht-Artefakte (Linux-Desktop + Agent) wirklich angehängt sind – der Windows-MSI-Build ist best-effort und darf fehlen.
