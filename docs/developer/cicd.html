CI/CD & Release

GitLab-CI-Pipeline baut Docker-Image, Desktop-DEBs und Go-Agent-Pakete. Release-Artefakte werden per Aptly-REST-API in ein signiertes Debian-Repo publiziert – nicht in GitLab-Packages.

Pipeline-Stages

StageJobs
checkpytest, svelte-check, cargo check, go vet
dockerServer-Image bauen & in docker.nevondo.com pushen
downloadfrpc-Binary vom offiziellen Release
buildDesktop-DEB (Trixie), Go-Agent-DEB/RPM für linux-amd64 + windows-amd64
publishDEBs via Aptly-API hochladen, Repo-Snapshots erzeugen & publizieren

Trigger-Matrix

EventWas läuft
Merge Requestcheck (muss grün sein)
Branch-Pushcheck + docker + build, aber kein publish
Tag v*.*.*Volle Pipeline inklusive publish ins Prod-Repo
Tag agent-v*.*.*Nur Agent-Build + Agent-Publish (Server & Desktop bleiben unberührt)

Version-Stellen

Bei einem Release muss die neue Version an mehreren Stellen parallel gesetzt werden – Tag allein reicht nicht. Stand v0.22.1:

  • frontend-src/package.json – Vite-Build des Web-Panels
  • desktop-src/package.json – Vite-Build des Desktop-Frontends
  • desktop/src-tauri/tauri.conf.json + desktop/src-tauri/Cargo.toml – Tauri-Bundle-Version
  • extension/manifest.json – Browser-Extension
  • .gitlab-ci.yml – AGENT_VERSION-Variable in den Agent-Build- und Publish-Jobs (mehrere Stellen)
  • docs/**/*.html – Footer-<span>v0.22.1</span> in 50 HTML-Dateien (DE + EN)
  • CHANGELOG.md – neuer Abschnitt am Anfang

Der Server selbst hält keine eigene version – das Backend wird über Docker-Image-Tag identifiziert. Im Go-Agent wird die Version per -ldflags "-X main.version=$VERSION" beim CI-Build reingeschrieben.

Aptly-Publish

Der CI-Job publish lädt DEBs in ein Aptly-Staging-Repo über die REST-API hoch (mutual TLS). Nach erfolgreichem Upload wird ein Snapshot erstellt und auf stable publiziert:

bash
# desktop/scripts/ci_publish_via_api.sh (gekürzt)
curl --cert "$APTLY_CLIENT_CERT" --key "$APTLY_CLIENT_KEY" \
     -F file=@"$DEB" \
     "$APTLY_URL/api/files/srm-staging"

curl --cert "$APTLY_CLIENT_CERT" --key "$APTLY_CLIENT_KEY" \
     -X POST "$APTLY_URL/api/repos/srm/file/srm-staging"

curl --cert "$APTLY_CLIENT_CERT" --key "$APTLY_CLIENT_KEY" \
     -X POST -H 'Content-Type: application/json' \
     -d "{\"Name\":\"srm-$VERSION\",\"SourceKind\":\"local\",\"SourceRepo\":\"srm\"}" \
     "$APTLY_URL/api/snapshots/srm"

curl --cert "$APTLY_CLIENT_CERT" --key "$APTLY_CLIENT_KEY" \
     -X PUT -H 'Content-Type: application/json' \
     -d "{\"Snapshots\":[{\"Component\":\"main\",\"Name\":\"srm-$VERSION\"}]}" \
     "$APTLY_URL/api/publish/stable/trixie"
Kein GitLab-Packages

DEBs werden ausschließlich über Aptly publiziert. GitLab-Packages wird nicht benutzt – nicht manuell hochladen, nicht als Fallback. Die öffentliche APT-Quelle bezieht ausschließlich aus dem Aptly-Repo.

Docker-Image

bash
# Multi-Stage: Frontend-Build → Python-Runtime
docker build \
  --target runtime \
  -t docker.nevondo.com/ks98/adminhelper/server:$CI_COMMIT_TAG \
  -t docker.nevondo.com/ks98/adminhelper/server:latest \
  .
docker push docker.nevondo.com/ks98/adminhelper/server:$CI_COMMIT_TAG
docker push docker.nevondo.com/ks98/adminhelper/server:latest

Tag latest wird nur bei Prod-Releases geschoben – Dev-Branches bekommen nur den Commit-SHA-Tag, um Deployment-Verwechslungen zu vermeiden.

Tauri-Build (Trixie)

Desktop-Client-Build läuft in debian:trixie Docker-Runner. Die Toolchain wird pro Job installiert, aber Cargo-Cache + target/ werden über GitLab-Cache wiederverwendet – spart ca. 6 min pro Build:

yaml
cache:
  key:
    files: [desktop/src-tauri/Cargo.lock]
  paths:
    - .cargo/bin
    - .cargo/registry/index
    - .cargo/registry/cache
    - .cargo/git/db
    - .rustup/toolchains
    - desktop/src-tauri/target

Go-Agent-Build

Cross-Compile für Linux + Windows in einem Job. Version wird per -ldflags reingeschrieben, damit adminhelper-agent version stimmt:

bash
VERSION=${CI_COMMIT_TAG#v}
GOOS=linux   GOARCH=amd64 go build \
  -ldflags "-X main.Version=$VERSION -X main.Commit=$CI_COMMIT_SHORT_SHA" \
  -o bin/adminhelper-agent ./cmd/adminhelper-agent

GOOS=windows GOARCH=amd64 go build \
  -ldflags "-X main.Version=$VERSION -X main.Commit=$CI_COMMIT_SHORT_SHA" \
  -o bin/adminhelper-agent.exe ./cmd/adminhelper-agent

VERSION=$VERSION bash build-deb.sh
VERSION=$VERSION bash build-rpm.sh

Release-Checkliste

  1. Versionen in den oben gelisteten Dateien auf den Zielwert hochziehen (z. B. mit sed über frontend-src/package.json, desktop-src/package.json, desktop/src-tauri/tauri.conf.json, desktop/src-tauri/Cargo.toml, extension/manifest.json, .gitlab-ci.yml).
  2. Footer-Versionen in docs/**/*.html aktualisieren: find docs -name '*.html' -exec sed -i 's/v0\.OLD/v0.NEW/g' {} \;
  3. CHANGELOG.md pflegen (manuell, nach Conventional-Commits-Vorgabe).
  4. Commit & Tag: git commit -am "release: v0.22.1" && git tag v0.22.1.
  5. git push --follow-tags.
  6. Pipeline beobachten – bei grünem publish ist die neue Version live auf deb.ks98.de.
  7. Release-Notes in GitLab-UI ergänzen (Markdown aus CHANGELOG.md).
Agent-only Release

Für Hotfixes am Go-Agent reicht ein Tag agent-v0.20.1. Die Pipeline überspringt Server- und Desktop-Jobs und publisht nur das Agent-DEB/-RPM. Spart 10+ min CI-Zeit.
