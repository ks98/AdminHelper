Entwickler-Handbuch

Browser-Extension

Die AdminHelper-Extension ist bewusst minimal: Manifest V3, drei Script-Surfaces (Popup, Options, Background-Service-Worker), keine Content-Scripts, keine Build-Pipeline. Reines Vanilla-JS, ~400 LoC.

Projekt-Layout

tree
extension/
├── manifest.json        Manifest V3 Deklaration
├── background.js        Service-Worker (Alarm + Fetch + Badge)
├── popup.html/.js/.css  Toolbar-Popup (Verbindungs-Dropdown)
├── options.html/.js/.css Einstellungs-Seite (Server-URL + API-Key)
└── icons/               32/64/128 PNG-Icons

Manifest V3

json
{
  "manifest_version": 3,
  "name": "AdminHelper",
  "version": "0.26.0",
  "permissions": ["storage", "tabs", "alarms"],
  "action":    { "default_popup": "popup.html" },
  "options_page": "options.html",
  "background": { "service_worker": "background.js" }
}

Nur drei Permissions:

  • storage – chrome.storage.sync für Server-URL + API-Key, chrome.storage.local für Connection-Cache.
  • tabs – neue Tabs aus dem Popup öffnen.
  • alarms – periodisches Refresh alle 5 Minuten.

Bewusst keine host_permissions – der Server ist frei konfigurierbar und wird beim ersten Fetch implizit erlaubt.

Service-Worker-Lifecycle

Manifest-V3 erlaubt keine Persistent-Background-Pages. Stattdessen startet der Worker on-demand:

  • onInstalled / onStartup: Alarm neu anlegen.
  • onAlarm: Fetch triggern.
  • Event-Ende → Worker wird suspendiert. Alle State-Daten müssen in chrome.storage.local persistiert werden.
javascript
const ALARM_NAME = 'refresh';
const REFRESH_INTERVAL_MINUTES = 5;

chrome.runtime.onInstalled.addListener(() => {
  chrome.alarms.create(ALARM_NAME, { periodInMinutes: REFRESH_INTERVAL_MINUTES });
});

chrome.alarms.onAlarm.addListener(async (alarm) => {
  if (alarm.name !== ALARM_NAME) return;
  await fetchAndCache();
});

Fetch & Cache

Pro Alarm wird GET /api/connections?api_key=… aufgerufen, die Liste auf kind === 'web' gefiltert und in chrome.storage.local.cachedConnections abgelegt:

javascript
const data = await res.json();
const conns = Array.isArray(data) ? data : (data.connections ?? []);
const webConns = conns.filter(c => c.kind === 'web');

await chrome.storage.local.set({
  cachedConnections: { connections: webConns, ts: Date.now() }
});
updateBadge(webConns.length);

Das Popup liest ausschließlich aus dem Cache – selbst bei Offline-Browser bleibt die Liste sichtbar (mit altem ts).

Badge-Counter

Die Anzahl der Web-Verbindungen erscheint als Badge am Toolbar-Icon (chrome.action.setBadgeText). Leer bei 0, Cyan-Background (#38bdf8) für Sichtbarkeit gegen helle und dunkle Chrome-Themes.

Popup & Options

Das Popup hat zwei Zustände: Ist keine Config gesetzt, erscheint die Setup-View (Server-URL + API-Key inline). Andernfalls die Main-View mit Suche, Listen-/Tag-Ansicht-Umschalter und Footer (Verbindungszahl + relative Zeit des letzten Refresh). Die gewählte Ansicht wird in chrome.storage.local.viewMode persistiert; offene/zugeklappte Tag-Gruppen leben in sessionStorage. Gefiltert wird live über Name, URL, Notes und Tags.

Die Options-Seite bietet neben „Speichern" einen Button Verbindung testen: er ruft denselben GET /api/connections auf und meldet die Gesamtzahl der Verbindungen sowie die Anzahl der Web-Verbindungen zurück, ohne etwas zu speichern. Änderungen an serverUrl/apiKey werden über chrome.storage.onChanged erkannt – ein offenes Popup re-initialisiert sich dann automatisch.

Security-Model

API-Key im Klartext

chrome.storage.sync ist kein Secret-Store. Der API-Key ist für alle Extensions im selben Chrome-Profil lesbar. Scope daher auf web_only beschränken – kein Connection-CRUD, kein Hook-Zugriff, keine Server-Admin-Rechte.

Web-Filter

Sowohl der Background-Worker als auch das Popup filtern die Server-Antwort auf c.kind === 'web' – nur Web-Verbindungen werden zwischengespeichert und gerendert. Verbindungen anderer Art (RDP, SSH, VNC) tragen ein abweichendes kind und erreichen die Extension gar nicht erst – keine Angriffsfläche für URI-Handler-Hijacking. Geklickt wird die Verbindung per chrome.tabs.create({ url }) mit der unveränderten url aus dem Server.

Lokales Testen

  1. Chrome → chrome://extensions → Entwicklermodus aktivieren.
  2. „Entpackte Erweiterung laden" → extension/-Verzeichnis wählen.
  3. Icon anklicken → Options öffnen → Server + API-Key setzen.
  4. DevTools des Service-Workers: chrome://extensions → Details → „Service-Worker inspizieren".

Firefox-Kompatibilität

Firefox unterstützt Manifest V3 seit Version 115. Einziger Unterschied: browser.* vs. chrome.*. Aktuell wird nur Chrome-API genutzt – für Firefox-Port müsste ein browser-polyfill-Shim ergänzt werden.
