API-Referenz

Alle HTTP-Endpoints des AdminHelper-Servers. Basis-URL: https://<host>. Das vollständige OpenAPI-Schema steht unter /api/docs (Swagger UI) bzw. /openapi.json.

Authentifizierung

Zwei Mechanismen, je nach Aufrufer:

  • JWT-Bearer (Web-Panel, Desktop): Authorization: Bearer <access_token>. Access- und Refresh-Token werden beide im JSON-Body geliefert – kein httpOnly-Cookie.
  • API-Key (Agents, Extension, Drittanbieter): Header X-API-Key. Permission ist read oder read_write.

Auth

POST/api/auth/login

Body: { "username", "password" }. Antwort: { "access_token", "refresh_token", "user" }. Login-Rate-Limit: 5 Fehlversuche / 60 s / IP.

POST/api/auth/refresh

Body: { "refresh_token" }. Rotiert beide Tokens, blacklisted den alten jti. Reuse eines bereits widerrufenen Refresh-Tokens invalidiert alle Sessions der betroffenen User-Kette.

POST/api/auth/logout

Blacklisted den aktuellen Access-Token (und optional einen mitgesendeten Refresh-Token).

GET/api/auth/me

Aktueller User; benötigt gültigen Access-Token.

Users

GET/api/users

Liste aller User (admin).

POST/api/users

Neuen User anlegen (admin). Passwort: SHA-256-Prehash → Bcrypt. Feuert user.created.

PUT/api/users/{id}

User ändern (admin).

DELETE/api/users/{id}

User löschen (admin). Feuert user.deleted.

GET/api/users/me/servers

Eigene zugewiesene Server.

Connections

GET/api/connections

Liste (read).

POST/api/connections

Anlegen (write). Feuert connection.created.

PUT/api/connections/{id}

Aktualisieren (write). Feuert connection.updated.

POST/api/connections/{id}/touch

Setzt last_used = now (read).

DELETE/api/connections/{id}

Löschen (admin). Feuert connection.deleted.

GET/api/connections/export

JSON-Export (admin).

POST/api/connections/import

?mode=replace|merge (admin). Feuert connections.imported.

Servers

GET/api/servers

Server-Inventar (admin).

POST/api/servers

Anlegen (admin). Feuert server.created.

GET/api/servers/{id}
PUT/api/servers/{id}

Aktualisieren (admin). Feuert server.updated.

DELETE/api/servers/{id}

Löschen (admin). Triggert zusätzlich Cleanup im Monitoring-Service.

API-Keys

GET/api/api-keys
POST/api/api-keys

Body: { name, permission: "read"|"read_write" }. Klartext-Key wird einmalig in der Response geliefert; gespeichert wird nur der SHA-256-Hash.

DELETE/api/api-keys/{id}

Hooks

GET/api/hooks
POST/api/hooks

Body: { name, hook_type: "webhook"|"event"|"schedule", event_triggers?: [...], schedule_interval?, script, enabled }. Bei webhook wird der Token serverseitig generiert (einmalig in der Response).

GET/api/hooks/{id}
PUT/api/hooks/{id}
DELETE/api/hooks/{id}

Entfernt automatisch aus dem Scheduler, falls Schedule-Hook.

POST/api/hooks/{id}/toggle

Enable/Disable; aktualisiert Scheduler-Registrierung.

POST/api/hooks/{id}/rotate

Webhook-Token erneuern.

POST/api/hooks/{id}/run

Manueller Test-Lauf mit leerem Context.

POST/api/hooks/trigger/{token}

Öffentlich (kein JWT). Rate-Limit: 20 Anfragen / 60 s / IP. Context: { payload, headers, params }.

Ansible-Playbooks

GET/api/ansible/playbooks
POST/api/ansible/playbooks

YAML-Validierung serverseitig; Speicherung auf Disk unter /data/ansible/playbooks/{id}/.

GET/api/ansible/playbooks/{id}
GET/api/ansible/playbooks/{id}/content

Liefert { content } (YAML als String).

PUT/api/ansible/playbooks/{id}
DELETE/api/ansible/playbooks/{id}

FRP – Server-Config

GET/api/frp/server-config
POST/api/frp/server-config

Schreibt zusätzlich frps.toml ins Shared-Volume.

GET/api/frp/server-config/{id}

Optional ?include_tunnels=true.

PUT/api/frp/server-config/{id}
DELETE/api/frp/server-config/{id}

FRP – Tunnels

GET/api/frp/tunnels

Query: ?server_id, ?frp_config_id.

POST/api/frp/tunnels

STCP (mit secret_key, visitor_port) oder HTTPS (custom_domains). auto_create_connection=true erzeugt direkt eine passende Connection.

GET/api/frp/tunnels/{id}
PUT/api/frp/tunnels/{id}
DELETE/api/frp/tunnels/{id}

FRP – Generate (TOML/ZIP)

GET/api/frp/generate/frps-toml?config_id=…
GET/api/frp/generate/frpc-toml/{server_id}
GET/api/frp/generate/visitor-toml?config_id=…&user_id=…
GET/api/frp/generate/visitor-bundle?config_id=…

Liefert { toml, pki }.

GET/api/frp/generate/bulk-zip?config_id=…

ZIP mit frps.toml + clients/*.toml + visitors/*.toml.

Server-Provisioning (FRP-unabhaengig, ab v0.23.1)

POST/api/servers/{server_id}/provision/token

Admin-only (JWT). Liefert { token, expiresAt, serverId, serverName } – one-time, 24 h TTL.

GET/api/servers/{server_id}/provision/tokens
POST/api/servers/{server_id}/provision/activate

Header X-Provision-Token. Antwort: { serverName, apiKey, monitorApiKey?, monitorUrl?, frp? }. monitorApiKey/monitorUrl sind null, wenn der Monitor-Service nicht erreichbar ist; frp ist null, wenn der Server keine FRP-Tunnel hat.

FRP – Sync (laufender Betrieb)

GET/api/frp/provision/{server_id}/config

Aktueller frpc.toml-Plaintext (JWT oder API-Key) – nur fuer den laufenden Sync.

GET/api/frp/provision/{server_id}/config-hash

SHA-256 zur drift-freien Synchronisation.

FRP – Status & PKI

GET/api/frp/status

Merged frps-Dashboard mit DB-Tunneln.

GET/api/frp/pki/status
POST/api/frp/pki/ca?common_name=…
POST/api/frp/pki/server-cert
POST/api/frp/pki/client-cert/{name}
GET/api/frp/pki/download/{filename}
GET/api/frp/pki/download-client-bundle/{client_name}

ZIP mit ca.crt, client.crt, client.key.

Monitoring (Reverse-Proxy)

POST/api/monitoring/agent/{server_id}/report

Öffentlich (X-API-Key). Reicht den Agent-Push 1:1 an den Monitoring-Microservice durch.

GET/api/monitoring/{path:path}

Admin-only Reverse-Proxy. Erlaubt sind nur die Pfad-Prefixe checks, alerts, log, metrics, status, templates, agent. Path-Traversal wird blockiert.

POST/api/monitoring/{path:path}
PUT/api/monitoring/{path:path}
DELETE/api/monitoring/{path:path}
Keine eigenen alerts-, audit- oder settings-Endpoints im AdminHelper

Alerts und Check-Status werden über den Monitoring-Proxy aus dem separaten Service geliefert. Audit-Trails laufen via Hook-Events über das Logging-System; ein dedizierter /api/audit-Endpoint existiert (Stand v0.23.1) nicht.

Fehler-Format

json
HTTP/1.1 422 Unprocessable Entity
Content-Type: application/json

{
  "detail": [
    { "loc": ["body", "port"], "msg": "ensure this value is less than 65536", "type": "value_error" }
  ]
}

Fehler folgen Pydantic-Validations-Konventionen.

Rate-Limits

  • Login: 5 Fehlversuche / 60 s / IP (Redis, In-Memory-Fallback).
  • Webhook-Trigger: 20 Anfragen / 60 s / IP.
  • Agent-Push: nicht limitiert; jede Anfrage muss aber per X-API-Key auf die eigene server_id matchen.
Swagger direkt nutzen

/api/docs rendert OpenAPI 3.1 interaktiv, inklusive „Try it out". Für CI-Clients reicht ein openapi-generator generate -g go -i https://srv/openapi.json.
