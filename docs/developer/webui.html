Web-Panel

Das Web-Panel ist eine eigenständige Svelte-5-SPA mit Hash-Router. Es liegt unter frontend-src/, wird via Vite gebaut und im Docker-Image per Multi-Stage in das Python-Runtime-Image kopiert.

Projekt-Struktur

tree

├── src/
│   ├── lib/
│   │   ├── api/           11 Module: client.ts + 9 Domain-Wrapper + types.ts
│   │   ├── components/    Button, Modal, TagChip, Tabs, EmptyState, Toast, ...

Routing

Hash-Router (kein SSR, kein Client-Router von SvelteKit): Die Pfade sind #/connections, #/servers usw. Das funktioniert zuverlässig hinter beliebigen Reverse-Proxies, ohne dass der Server SPA-Fallbacks verstehen muss. Implementierung in src/lib/router.ts – ein writable _path mit hashchange-Listener; exportiert navigate(), replace(), currentPath() und einen segments-Derived-Store.

Acht Produktiv-Routen: #/connections, #/servers, #/users, #/apikeys, #/hooks, #/frp, #/ansible, #/monitoring.

Stores

Zehn Stores in src/lib/stores/. Jeder Store ist eine TypeScript-Klasse mit $state-Feldern und Methoden, die direkt mit dem passenden API-Wrapper sprechen.

StoreZweck
authLogin, Logout, Refresh, aktueller User; Token in localStorage (adminhelper_token, adminhelper_refresh_token).
connectionsLive-Liste, Suche, Gruppenfilter, Tag-Hierarchie.
serversServer-Inventar, Provision-Token, User-Zuweisung.
usersBenutzer-CRUD inkl. Server-Mapping.
apikeysKey-CRUD, einmaliges Anzeigen des Klartext-Keys.
hooksHook-CRUD, Trigger-Test, Token-Rotation.
frpTunnel-CRUD, Provision, PKI-Status.
ansiblePlaybook-Verwaltung und Editor-State.
monitoringChecks, Alerts, Templates, VictoriaMetrics-Queries via Proxy.
notificationsGlobale Toasts und Bestätigungs-Dialoge.

API-Wrapper

Alle HTTP-Calls gehen durch src/lib/api/client.ts. Der Client verwaltet Access-Token, hält eine in-flight-deduplizierte tryRefresh()-Promise und triggert automatisch Refresh, wenn ein 401 zurückkommt. Schlägt das fehl, ruft registerAuthFailureHandler() den auth-Store, der zum Login leitet.

Domain-Wrapper liegen daneben: auth.ts, connections.ts, servers.ts, users.ts, apikeys.ts, hooks.ts, frp.ts, ansible.ts, monitoring.ts – jeweils typisiert über types.ts.

i18n

Sprachen: DE und EN. Aktive Sprache wird in localStorage.adminhelper_language persistiert; Browser-Detect dient nur als initialer Fallback. Die t()-Funktion ist als Store implementiert – das verhindert i18n-Leaks in reaktiven Komponenten.

uPlot-Charts

Monitoring-Charts nutzen uplot direkt – kein schwergewichtiges Chart-Framework. Wrapper in src/lib/components/. uPlot ist 30 kB gzipped und rendert schnell auch 100k Punkte.

Tests

Playwright-Smoketests decken alle 8 Produktiv-Seiten ab. Zwei Spec-Dateien: login.spec.ts und smoke.spec.ts; Visual-Diffs liegen in tests/e2e/*.spec.ts-snapshots/. Toleranz ist maxDiffPixelRatio: 0.02 (2 %). Locale ist de-DE, Viewport 1280×800. Reports landen als CI-Artifact (frontend_e2e-Job).

bash
cd frontend-src
npm run check           # svelte-check + tsc
npm run lint            # eslint + svelte-eslint
npx playwright test     # E2E

Build & Deployment

Der Vite-Build landet in frontend-src/dist/ (Sourcemaps an, Target ES2022). Im Repo-Root-Dockerfile wird dieser Output via Multi-Stage in /app/frontend/ kopiert, und die FastAPI-App mountet ihn als Static-Route. SPA-Fallback prüft vorher Datei-Existenz, bevor index.html zurückgegeben wird – das verhindert MIME-Fehler auf /assets/*.js.

Im Dev-Modus läuft Vite auf Port 5173 und proxied /auth, /api, /health auf localhost:8000.
