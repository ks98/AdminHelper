Web-Panel

Das Web-Admin-Panel ist eine eigenständige Svelte-5-SPA mit Hash-Router. Es liegt unter apps/web/, wird via Vite gebaut und im Docker-Image per Multi-Stage in das Python-Runtime-Image kopiert. Seit v0.32.0 deckt es nur noch die Instanz-Verwaltung ab (Benutzer, API-Keys, Hooks, FRP-Server-Konfiguration); die operative Server-/Verbindungs-/Tunnel-/Monitoring-/Ansible-Verwaltung ist in den Desktop-Client umgezogen (siehe Desktop-Client).

Routing

Hash-Router (kein SSR, kein Client-Router von SvelteKit): Die Pfade sind #/users, #/apikeys usw. Das funktioniert zuverlässig hinter beliebigen Reverse-Proxies, ohne dass der Server SPA-Fallbacks verstehen muss. Implementierung in src/lib/router.ts – ein writable _path mit hashchange-Listener; exportiert navigate(), replace(), currentPath() und einen segments-Derived-Store.

Produktiv-Routen seit v0.32.0: #/users (Standard-Landing), #/apikeys, #/hooks, #/frp (FRP-Server-Konfiguration). Die früheren operativen Routen (#/connections, #/servers, #/ansible, #/monitoring, Tunnel) sind in den Desktop-Client umgezogen.

Stores

Stores in src/lib/stores/. Jeder Store ist eine TypeScript-Klasse mit $state-Feldern und Methoden, die direkt mit dem passenden API-Wrapper sprechen. Seit v0.32.0 nur noch die Instanz-Verwaltung – die operativen Stores (connections, servers, ansible, monitoring, Tunnel) leben jetzt im Desktop-Client.

StoreZweck
authLogin, Logout, Refresh, aktueller User; nur der kurzlebige Access-Token liegt in localStorage (adminhelper_token), der langlebige Refresh-Token in einem HttpOnly-Cookie (vom Server gesetzt, für JS/XSS unlesbar).
usersBenutzer-CRUD inkl. User↔Server-Mapping (die Zuweisung bleibt im Web-Panel).
apikeysKey-CRUD, einmaliges Anzeigen des Klartext-Keys.
hooksHook-CRUD, Trigger-Test, Token-Rotation.
frpFRP-Server-Konfiguration (frps-Einstellungen), Config-Preview, frps-Status.
notificationsGlobale Toasts und Bestätigungs-Dialoge.

API-Wrapper

Alle HTTP-Calls gehen durch src/lib/api/client.ts. Der Client verwaltet Access-Token, hält eine in-flight-deduplizierte tryRefresh()-Promise und triggert automatisch Refresh, wenn ein 401 zurückkommt. Schlägt das fehl, ruft registerAuthFailureHandler() den auth-Store, der zum Login leitet.

Domain-Wrapper liegen daneben: auth.ts, users.ts, servers.ts (nur für die User↔Server-Zuweisung), apikeys.ts, hooks.ts, frp.ts (FRP-Server-Konfiguration) – jeweils typisiert über types.ts.

i18n

Sprachen: DE und EN. Aktive Sprache wird in localStorage.adminhelper_language persistiert; Browser-Detect dient nur als initialer Fallback. Die t()-Funktion ist als Store implementiert – das verhindert i18n-Leaks in reaktiven Komponenten.

Tests

Playwright-Smoketests decken die Produktiv-Seiten ab. Zwei Spec-Dateien: login.spec.ts und smoke.spec.ts; Visual-Diffs liegen in tests/e2e/*.spec.ts-snapshots/. Toleranz ist maxDiffPixelRatio: 0.02 (2 %). Locale ist de-DE, Viewport 1280×800. Reports landen als CI-Artifact (frontend_e2e-Job).

bash
cd apps/web
npm run check           # svelte-check + tsc
npm run lint            # eslint + svelte-eslint
npx playwright test     # E2E

Build & Deployment

Der Vite-Build landet in apps/web/dist/ (Sourcemaps an, Target ES2022). Im Repo-Root-Dockerfile wird dieser Output via Multi-Stage in /app/frontend/ kopiert, und die FastAPI-App mountet ihn als Static-Route. SPA-Fallback prüft vorher Datei-Existenz, bevor index.html zurückgegeben wird – das verhindert MIME-Fehler auf /assets/*.js.

Im Dev-Modus läuft Vite auf Port 5173 und proxied /auth, /api, /health auf localhost:8000.
