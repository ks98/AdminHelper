Server (Backend)

Der Server ist ein FastAPI-Monolith: HTTP-Router, SQLAlchemy-Models, APScheduler-Jobs und FRP-PKI laufen im selben Prozess. Separation by Module, nicht by Deployment.

Einstiegspunkte

DateiZweck
apps/server/app/main.pyFastAPI-App, Lifespan, Middleware-Stack, Static-Mounts für Vite-Output, Auto-Migrationen, SPA-Fallback.
apps/server/app/core/config.pyEnv-Var-Resolution, JWT-Settings, Secret-Key-Persistenz (.secret_key-Datei).
apps/server/app/core/auth.pyJWT-Encode/Decode, Bcrypt-Hashing mit SHA-256-Prehash, Refresh-Rotation, Token-Blacklist, API-Key-Auth.
apps/server/app/core/database.pySQLAlchemy-Engine + SessionLocal; PostgreSQL via psycopg3, Pool-Konfig (size=10, overflow=20, pre-ping, recycle=3600).
apps/server/app/core/rate_limit.pyRedis-basierter Fixed-Window-Limiter mit In-Memory-Fallback.
apps/server/app/core/middleware.pyIP-Filter (ALLOWED_IPS) und Proxy-Trust (TRUSTED_PROXIES).
apps/server/app/core/events.pyAsync Event-Bus für Hooks (ThreadPoolExecutor mit 4 Workern, Fire-and-Forget).

Alerts und Check-Status leben im Monitoring-Service und werden über den monitoring_proxy reingereicht. Ein dedizierter Audit-Log-Router existiert (Stand v0.23.2) nicht; Logging erfolgt über Python-Logging und Hook-Events.

Auth-Architektur

JWT-Access-Token (HS256, 8h) und Refresh-Token (7d) werden im JSON-Body ausgeliefert; der Refresh-Token wird zusätzlich als HttpOnly; Secure; SameSite=Strict-Cookie (Pfad /api/auth) gesetzt, das Browser-Clients statt localStorage nutzen (Desktop/CLI bleiben beim Body). Token-Claims: sub (Username), type (access/refresh), exp, iat (Ausstellzeit), jti (UUID).

  1. Login signiert beide Tokens und gibt sie im TokenResponse zurück.
  2. Bei POST /api/auth/refresh prüft der Server Signatur, Ablauf und Blacklist.
  3. Alter jti wird sofort in token_blacklist eingetragen, dann werden neue Tokens ausgestellt (Rotation).
  4. Wird ein bereits widerrufener Refresh erneut eingereicht → Reuse-Detection: alle Sessions der betroffenen User-Kette werden invalidiert (WARNING-Log mit IP).
  5. Passwort-Reset widerruft Sessions: ein Passwort-Wechsel setzt users.tokens_valid_after; jeder Token mit iat davor (oder ohne iat) wird abgelehnt — ohne dass einzelne jti bekannt sein müssen.

Passwort-Hashing nutzt SHA-256-Prehash → Bcrypt, weil Bcrypt nur 72 Bytes verarbeitet.

API-Keys werden über den Header X-API-Key akzeptiert; die Dependency ApiKeyOrUser erlaubt JWT oder API-Key am selben Endpoint.

Rate-Limit & IP-Filter

Login-Limit (5 pro 60s/IP) und Webhook-Limit (20 pro 60s/IP) sind Redis-basiert mit In-Memory-Fallback. IP-Filter via ALLOWED_IPS (CIDR-Liste); Proxy-Trust über TRUSTED_PROXIES (sicher) statt TRUST_PROXY_HEADERS (legacy).

Server-Provisioning

Neue Server werden über Einmal-Tokens mit 24-Stunden-TTL onboarded — seit v0.23.2 FRP-unabhaengig im Server-Modul:

  1. Admin ruft POST /api/servers/{server_id}/provision/token auf → erhält {token, expiresAt, serverId, serverName}.
  2. Der Agent löst den Token mit POST /api/servers/{server_id}/provision/activate ein (Header X-Provision-Token) und bekommt zurück: frischen Server-API-Key (immer), Monitor-Agent-Key + URL (wenn Monitor-Service erreichbar), frpc.toml (wenn der Server FRP-Tunnel hat) und einen einmaligen enrollment-Token. Felder sind null, wenn die jeweilige Komponente nicht konfiguriert ist — ein ausgefallener Monitor-Service blockiert das Provisioning nicht. Ein per-Client-PKI-Bundle wird nicht mehr mitgeliefert: der Agent enrollt sein mTLS-/Tunnel-Cert selbst am ca-issuer (A4/A7).
  3. Tokens sind one-time: used_at wird gesetzt, weitere Aktivierungen scheitern.
  4. Im laufenden Betrieb synchronisiert der Agent über GET /api/frp/provision/{server_id}/config + config-hash seinen frpc.toml-Stand (nur wenn FRP konfiguriert).

FRP — TOML-Struktur

modules/frp/config_generator.py rendert die FRP-Konfigurationen direkt aus den DB-Modellen zu TOML-Strings — kein Template-Engine, nur String-Listen. Drei Generatoren:

FunktionZielInhalt
generate_frps_toml(config)frps.toml (Server)bindPort, optional vhostHTTPSPort/subDomainHost/maxPortsPerClient, optionaler Dashboard-Block (webServer.* an 127.0.0.1), auth.method = "token" + Token, plus [transport.tls]-Server-Block.
generate_frpc_toml(config, tunnels, frpc_user)frpc.toml (Agent)serverAddr/serverPort/user, Token-Auth, Client-TLS-Block, dann je aktivem Tunnel ein [[proxies]]-Block.
generate_visitor_toml(config, tunnels, visitor_user)Visitor-frpc.toml (Admin-PC)Aggregiert alle aktiven STCP-Tunnel zu je einem [[visitors]]-Block (bindPort = visitor_port).

Nur Tunnel mit enabled = true werden gerendert. Zwei Tunnel-Typen sind speziell behandelt:

  • STCP (type = "stcp"): bekommt secretKey und allowUsers (Default ["ops-admin"], sonst die uebergebenen allow_users). Der zugehoerige [[visitors]]-Block setzt serverName, serverUser (= Agent-User, weil FRP Proxies als {user}.{name} registriert) und denselben secretKey.
  • HTTPS (type = "https"): rendert customDomains aus dem komma-getrennten Feld custom_domains.

extra_config (JSON) wird je Tunnel typ-erhaltend angehaengt: Strings gequotet, Bools als true/false, Zahlen roh.

frpc.toml (STCP-Beispiel)
serverAddr = "frp.example.com"
serverPort = 7000
user = "k01-lnx1"

auth.method = "token"
auth.token = "<token>"

[transport.tls]
enable = true
trustedCaFile = "/etc/adminhelper/identity/ca.crt"
certFile = "/etc/adminhelper/identity/agent.crt"
keyFile = "/etc/adminhelper/identity/agent.key"

[[proxies]]
name = "ssh"
type = "stcp"
localIP = "127.0.0.1"
localPort = 22
secretKey = "<secret>"
allowUsers = ["ops-admin"]

FRP-PKI & TLS

Seit A7 kommen das frps-Server-Cert und das Agent-Tunnel-Cert aus der tunnel-Intermediate des ca-issuer (ECDSA P-256, eine Vertrauenswurzel): frps mountet das vom Issuer provisionierte Material (frps.crt/frps.key/ca.crt) read-only unter /etc/frp-pki, der Agent nutzt sein bei der Enrollment (A4) erhaltenes Tunnel-Cert unter /etc/adminhelper/identity — ein Cert für Server-Pushes und den frp-Tunnel, auto-erneuert bei ~50 % Laufzeit.

Der Server signiert keine FRP-Zertifikate mehr (D6, F3). Die frühere server-eigene FRP-CA (modules/frp/pki.py + pki_router.py mit den /api/frp/pki/*-Endpunkten + die _ensure_pki()-CA-Erzeugung beim Start) ist mit der PKI-Migration vollständig entfernt, ebenso das frp-pki-Volume und der frühere Master-/Published-Split. _ensure_frps_config() schreibt beim Start nur noch das nicht-geheime frps.toml.

Der Desktop-STCP-Visitor präsentiert seit der Migration (F2) seine enrollte access-Identität als frpc-Client-Cert (der Desktop exportiert Key/Cert/CA aus dem Keyring in Dateien unter identity/). Damit frps diese akzeptiert, trägt der ca-issuer die access-Intermediate zusätzlich in frps' ca.crt ein (ensure_frps_cert(..., extra_trust)); die echte Per-Tunnel-Autorisierung bleibt der STCP-secretKey + die server-seitige Bundle-Filterung, nicht der Cert-Scope.

Die TLS-Verdrahtung in frps.toml/frpc.toml erzeugt der config_generator: Der Server-Block (force = true) zeigt auf das issuer-provisionierte Material unter /etc/frp-pki; der Agent-Block (enable = true) auf das enrollte Tunnel-Cert unter /etc/adminhelper/identity; der Visitor-Block auf die vom Desktop exportierte enrollte Identität (relatives identity/, zur Laufzeit auf absolute Pfade umgeschrieben).

SQLAlchemy-Models

Die Tabellen liegen jeweils im Modul (z. B. modules/users/models.py) und werden in main.py per Base.metadata.create_all() beim Startup angelegt. Idempotente Spalten-Migrationen erfolgen über _migrate_add_columns() mit PRAGMA table_info.

  • User ↔ Server (n:m über user_servers), TokenBlacklist
  • Server (UUID-PK, JSON-Tags, os_type) ↔ Connection (n:1, server_id nullable für Standalone)
  • ApiKey mit permission ∈ {read, read_write}
  • Hook (hook_type ∈ {webhook, event, schedule}, JSON-event_triggers, Python-script)
  • FrpServerConfig ↔ FrpTunnel (n:1) ↔ ProvisionToken (n:1, 24h-TTL, one-time)
  • Playbook (Ansible) mit Disk-Backed YAML-Content

Event-Bus & Scheduler

APScheduler läuft im selben Event-Loop wie FastAPI. Beim Startup lädt der load_all_scheduled_hooks()-Call alle aktiven Schedule-Hooks; Trigger-Typen: vordefinierte Intervalle (5m, 10m, 15m, 30m, 1h, 6h, 12h, 24h) oder Cron (5 Felder).

Der Event-Bus in core/events.py verteilt Events über einen ThreadPoolExecutor (4 Worker, fire-and-forget). Aktive Event-Typen:

  • server.created, server.updated, server.deleted
  • connection.created, connection.updated, connection.deleted, connections.imported
  • user.created, user.deleted
  • api_key.created, api_key.deleted
  • frp.config.created, frp.config.updated, frp.config.deleted
  • frp.tunnel.created, frp.tunnel.updated, frp.tunnel.deleted
  • playbook.created, playbook.updated, playbook.deleted

Details siehe Hooks & Events.

Persistenz

Stateful-Daten verteilen sich auf zwei Stellen:

  • PostgreSQL (Compose-Volume postgres-data): zwei DBs adminhelper (Server) und adminhelper_monitor (Monitoring).
  • File-System unter DATA_DIR (Default /app/data): Bootstrap-Token, JWT-Secret, Ansible-Playbook-Dateien.
  • Shared-Volume frp-config (frps mountet es :ro unter /etc/frp): nur das nicht-geheime frps.toml (0600). Die FRP-TLS-Materialien liefert der ca-issuer (Volume frps-certs), das PKI-Kronjuwel ist dessen ca-pki-Volume.

Der Container startet nur kurz als root, um die gemounteten Pfade zu chownen, und droppt dann via gosu auf einen Non-root-User (uid 10001) — uvicorn, Alembic und Hook-Subprozesse laufen unprivilegiert. uvicorn lauscht plain-HTTP auf 8080 (kein Host-Port); TLS terminiert das vorgelagerte gateway (nginx), nicht der Server.

tree
/app/data/                         # DATA_DIR (server-only)
├── .bootstrap_token            # Einmaliger Setup-Token-Hash (P0-6)
├── .secret_key                 # Auto-Gen JWT-Secret
└── ansible/playbooks/{id}/{file}.yaml

/app/frp-config/                   # Shared mit frps (Volume frp-config, :ro fuer frps)
├── frps.toml
└── pki/
    └── ca.crt, frps.crt, frps.key   # publizierte Teilmenge — KEIN ca.key, KEINE Client-Keys

Tests

Pytest-Suite unter server/tests/. Tests laufen gegen ein echtes Postgres 17 via testcontainers (lokal) bzw. services: postgres:17-alpine (CI). SAVEPOINT-Pattern in der db_session-Fixture: Outer-Transaction mit begin_nested + after_transaction_end-Listener — erlaubt commit() im Test-Code, am Ende sauberer Rollback. Plus test_alembic_consistency.py als Drift-Detector zwischen Base.metadata und Alembic-Migrations.

bash
cd apps/server
uv run pytest -x                 # schnell abbrechen bei Fehler
uv run pytest --cov=app          # Coverage
