Desktop-Client

Der Desktop-Client ist ein Tauri-2-Wrapper um eine Svelte-5-SPA. Das Backend in Rust (apps/desktop/src-tauri/) stellt 23 typisierte Commands bereit; das Frontend (apps/desktop/ui/) nutzt sie über eine 1:1-abgebildete Bridge.

Tauri-Commands

Die 23 Commands sind in src/main.rs als invoke_handler registriert und decken Auth, Connection-Persistenz, Connect-Flow, FRP-Tunnel, Keyring, API-Proxy und Ansible ab. Jeder Command ist typisiert – die Bridge in apps/desktop/ui/src/lib/bridge/ spiegelt sie 1:1.

RDP-Race-Guard

Wenn der Nutzer mehrmals kurz hintereinander auf Connect klickt, würden ohne Schutz mehrere RDP-Prozesse starten. Der Desktop-Client vergibt pro Klick eine monotone UUID. Läuft der FRP-Tunnel-Aufbau noch, während eine neue Connect-ID reinkommt, wird die alte verworfen. Implementierung in apps/desktop/src-tauri/src/rdp/race_guard.rs.

Keyring-Secrets

Passwörter landen nicht in der lokalen Connection-Datei, sondern im OS-Keyring:

  • Linux: secret-service (GNOME Keyring / KWallet)
  • macOS: Keychain
  • Windows: Credential Manager

Service-Prefix adminhelper:, Account = <connection-id>.

FRP-Sidecar

Tauri-Capabilities sind geschärft: frpc darf nur aus dem App-Binaries-Verzeichnis gestartet werden. Beim Build wird die passende Target-Binary (frpc-x86_64-unknown-linux-gnu, frpc-aarch64-apple-darwin, …) in apps/desktop/src-tauri/binaries/ erwartet.

Server-Mode-Flow

Im Server-Mode (Setting mode = 'server') holt der Client Verbindungen per JWT vom AdminHelper-Server und baut den FRP-Tunnel selbst auf. Die Schritte greifen über auth.rs, sync.rs, frpc.rs und tunnel.rs ineinander:

  1. Login – login postet an /api/auth/login, ruft danach /api/auth/me ab und legt Access-Token, Refresh-Token und Server-URL im OS-Keyring ab (Service com.adminhelper.app, Keys auth|jwt, auth|refresh, auth|server_url). Implementierung in auth.rs.
  2. Session-Restore – check_session validiert das gespeicherte Access-Token gegen /api/auth/me; bei 401 wird über /api/auth/refresh ein neues Token-Paar geholt und neu im Keyring gespeichert. Kein erneuter Login nach Neustart.
  3. Verbindungen laden – fetch_connections_jwt holt die Liste per authenticated_get von /api/connections (mit transparentem 401-Refresh-Retry) und schreibt sie lokal weg. Passwörter sind nie Teil der Sync-Daten.
  4. Tunnel starten – start_tunnel lädt das Visitor-Bundle (TOML + PKI-Map) von /api/frp/generate/visitor-bundle, schreibt visitor.toml und die PKI-Dateien in das App-Data-Verzeichnis (relative pki/-Pfade werden auf absolute umgeschrieben, .key-Dateien mit 0600) und startet anschließend frpc -c <config> als Sidecar.
  5. Status & Events – tunnel_status liefert running, visitor_name und connected_since. Der Sidecar-Prozess emittiert die Tauri-Events frpc-terminated und frpc-error; der Tunnel-Store (stores/tunnel.ts) hört darauf und speist die Statusanzeige (TunnelIndicator.svelte) mit den UI-Zuständen connecting / connected / disconnected.
  6. Tunnel-Auflösung – fetch_tunnels holt die Mappings von /api/frp/tunnels, resolve_connection rewritet passende Connections: stcp → host = 127.0.0.1, port = visitor_port (Web zusätzlich http://127.0.0.1:<port>), https → url = https://<custom-domain>.

Der Auto-Start des Sidecars passiert nur im Server-Mode: startIfServerMode() im Tunnel-Store bricht ab, wenn keine Session vorliegt oder settings.mode !== 'server'. Beim Beenden der App räumt RunEvent::Exit in main.rs den frpc-Prozess via stop_frpc auf.

Connect-Flow

  1. User klickt Connect → connectFlow-Store setzt state = 'connecting'
  2. Bridge ruft connect_rdp auf (mit Race-Guard-UUID)
  3. Rust-Backend resolved den FRP-Tunnel (lokaler Visitor-Port)
  4. Passwort wird aus dem Keyring geholt oder via PasswordPrompt-Modal abgefragt
  5. RDP/SSH/VNC-Client wird mit temporärer Config gestartet
  6. Nach Exit wird der Tunnel wieder abgebaut
