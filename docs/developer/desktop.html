Desktop-Client

Der Desktop-Client ist ein Tauri-2-Wrapper um eine Svelte-5-SPA. Das Backend in Rust (desktop/src-tauri/) stellt 23 typisierte Commands bereit; das Frontend (desktop-src/) nutzt sie über eine 1:1-abgebildete Bridge.

Architektur

tree

Tauri-Commands

Die 23 Commands sind in src/main.rs als invoke_handler registriert und decken Auth, Connection-Persistenz, Connect-Flow, FRP-Tunnel, Keyring, API-Proxy und Ansible ab. Jeder Command ist typisiert – die Bridge in desktop-src/src/lib/bridge/ spiegelt sie 1:1.

RDP-Race-Guard

Wenn der Nutzer mehrmals kurz hintereinander auf Connect klickt, würden ohne Schutz mehrere RDP-Prozesse starten. Der Desktop-Client vergibt pro Klick eine monotone UUID. Läuft der FRP-Tunnel-Aufbau noch, während eine neue Connect-ID reinkommt, wird die alte verworfen. Implementierung in desktop/src-tauri/src/rdp/race_guard.rs.

Keyring-Secrets

Passwörter landen nicht in der lokalen Connection-Datei, sondern im OS-Keyring:

  • Linux: secret-service (GNOME Keyring / KWallet)
  • macOS: Keychain
  • Windows: Credential Manager

Service-Prefix adminhelper:, Account = <connection-id>.

FRP-Sidecar

Tauri-Capabilities sind geschärft: frpc darf nur aus dem App-Binaries-Verzeichnis gestartet werden. Beim Build wird die passende Target-Binary (frpc-x86_64-unknown-linux-gnu, frpc-aarch64-apple-darwin, …) in desktop/src-tauri/binaries/ erwartet.

Connect-Flow

  1. User klickt Connect → connectFlow-Store setzt state = 'connecting'
  2. Bridge ruft connect_rdp auf (mit Race-Guard-UUID)
  3. Rust-Backend resolved den FRP-Tunnel (lokaler Visitor-Port)
  4. Passwort wird aus dem Keyring geholt oder via PasswordPrompt-Modal abgefragt
  5. RDP/SSH/VNC-Client wird mit temporärer Config gestartet
  6. Nach Exit wird der Tunnel wieder abgebaut
