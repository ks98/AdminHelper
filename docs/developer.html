Developer Guide

Entwicklung, Architektur und Erweiterung

Diese Seite dokumentiert den technischen Aufbau des Projekts inklusive lokaler Entwicklungsumgebung, API-Verhalten, Datenmodellen und Release-Pipeline.

Setup starten API-Referenz

1) Entwicklungs-Setup

Empfohlen: Linux oder Windows mit aktuellen Toolchains.

Bereich Pflicht Hinweis
Client (Tauri) Rust stable, cargo tauri Linux benötigt WebView-/GTK-Abhängigkeiten und für RDP xfreerdp3 oder xfreerdp.
Server Python 3.12+, pip FastAPI + SQLAlchemy + SQLite, optional Docker-Betrieb.
Extension Chrome/Chromium Manifest V3, lokal als entpackte Erweiterung ladbar.
Agent Go 1.24+, make Unified Go Agent für Linux + Windows (FRP-Sync + Monitoring).
Monitoring Python 3.12+, pip FastAPI Monitoring-Service, wird per Docker Compose mitgestartet.

Tauri CLI installieren

cargo install tauri-cli --locked --version '^2'

Server lokal (ohne Docker)

cd server
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8080

4) Client-Architektur (Tauri)

Der Client trennt Frontend-UI und native Funktionen über Tauri-Commands. Einstieg ist desktop/src-tauri/src/main.rs.

Modul Datei Aufgabe
Command Bridge commands.rs Expose von load/save/open/sync/password an das JS-Frontend.
Verbindung öffnen connection/* SSH, RDP und Web je OS-spezifisch starten.
Lokale Persistenz storage.rs connections.json + settings.json im OS-AppData-Verzeichnis, auf Unix mit 0600-Rechten.
Sync sync.rs Lädt Connection-JSON nur über https://.
Passwortspeicher password.rs Nur für RDP; unter Linux Keyring, unter Windows Credential Manager.
Ansible ansible.rs INI-Inventory aus Servern generieren, Playbook speichern, ansible-playbook im Terminal starten.
Wichtig: Im Sync-Modus sind lokale CRUD-Operationen im Frontend gesperrt.

5) Server-Architektur (FastAPI)

Einstieg: server/app/main.py. Beim Start werden DB-Schema erzeugt und ein initialer Admin angelegt, falls noch kein Benutzer existiert.

Modulare Struktur unter app/core/ (Config, Auth, DB, Middleware, Events) und app/modules/:

Modul Pfad Details
Auth core/auth.py, modules/users/auth_router.py JWT Bearer, bcrypt mit SHA-256-Prehash für lange Passwörter.
Users modules/users/ User-Management, Login/Logout, Token-Refresh.
Connections modules/connections/ Connection CRUD (SQLite). Automatische Migration von JSON zu SQL beim Start.
Servers modules/servers/ Server-Registry mit Tags, Host, Port.
FRP modules/frp/ Hierarchisch: Config, Tunnel, Generate, Status, PKI, Provision (6 Sub-Router).
Hooks modules/hooks/ Webhook, Event, Schedule (APScheduler). Script-Runner mit vollem Python-Zugriff (Admin-only).
API-Keys modules/api_keys/ API-Key Management, SHA-256-gehasht in DB.
Ansible modules/ansible/ Playbook CRUD mit YAML-Validierung. Dateiablage unter DATA_DIR/ansible/playbooks/.
Monitoring Proxy modules/monitoring_proxy.py Reverse-Proxy zum Monitoring-Service für das Web-UI.
IP Filter core/middleware.py Blockt Requests anhand ALLOWED_IPS, Proxy-aware mit Trust-Optionen.

6) Extension-Architektur

Manifest V3 mit Service Worker. Ziel ist schnelle Anzeige von kind: web-Verbindungen aus dem Server.

  • background.js: Alarm alle 5 Minuten, Fetch + Cache + Badge-Update.
  • popup.js: Setup-Flow, Suche, Listen-/Tag-Ansicht, Öffnen in neuem Tab.
  • options.js: Speicherung und Verbindungstest (Server-URL + API-Key).
  • Konfiguration in chrome.storage.sync, Cache in chrome.storage.local.

7) API und Auth-Verhalten

API nutzt zwei Auth-Wege: JWT Bearer (Benutzer) oder API-Key über Header X-API-Key bzw. Query-Parameter api_key.

Endpoint Zugriff Kommentar
POST /api/auth/login öffentlich Login, liefert JWT.
GET /api/connections JWT oder API-Key (read/read_write) Liste aller Connections.
POST/PUT /api/connections Admin-JWT oder API-Key read_write Create/Update mit UUID-Vergabe bei Create.
DELETE /api/connections/{id} nur Admin-JWT API-Key reicht hier nicht.
/api/users, /api/api-keys, /api/hooks nur Admin-JWT Betriebs-/Sicherheitsverwaltung.

8) Hook-Scripting

Alle Hooks laufen im gleichen Script-Runner (server/app/script_runner.py) mit vollständigen Python-Builtins inkl. import (Admin-only). Der Kontext im Script ist je nach Typ unterschiedlich.

Gemeinsame Variablen (alle Typen)

VariableTypBeschreibung
load_connections() Funktion Lädt alle Connections aus connections.json.
save_connections(list) Funktion Speichert die komplette Connection-Liste.
uuid4() Funktion Erzeugt eine neue UUID als String.
result, logs, log(msg) dict/list/Funktion Antwort und Logging.
import ... Statement Alle Python-Module nutzbar: requests, json, re, …

Typ-spezifischer Kontext

TypZusätzliche VariablenAuslöser
webhook payload, headers, params POST /api/hooks/trigger/<token>
event event_type (str), event_data (dict) Intern bei connection.*/user.*/server.startup
schedule triggered_at (ISO-str), last_run (ISO-str | None) APScheduler Hintergrund-Thread

Beispiel: Webhook – Verbindung anlegen/updaten

Datei im Repository: docs/examples/webhook_add_connection.py 

connections = load_connections()
incoming = payload.get("connection")

name = str(incoming.get("name", "")).strip()
kind = str(incoming.get("kind", "")).strip().lower()
if kind not in ("ssh", "rdp", "web"):
    raise ValueError("connection.kind muss ssh, rdp oder web sein")

# ... Normalisierung + ID-Suche ...
save_connections(connections)
result["mode"] = mode
result["connectionId"] = connection_id

Beispiel: Event-Hook – Benachrichtigung bei Änderungen

import requests

# Läuft bei connection.created, connection.updated, connection.deleted
msg = f"[SRM] {event_type}: {event_data.get('name', '?')} ({event_data.get('kind', '?')})"
requests.post("https://chat.example.com/webhooks/notify", json={"text": msg}, timeout=5)
result["notified"] = True

Beispiel: Scheduled Hook – tägliche Bereinigung

# Intervall: 24h
connections = load_connections()
seen, unique = set(), []
for c in connections:
    key = (c.get("kind"), c.get("host"), c.get("port"))
    if key not in seen:
        seen.add(key)
        unique.append(c)
removed = len(connections) - len(unique)
if removed:
    save_connections(unique)
result["removed"] = removed
log(f"Bereinigt am {triggered_at}: {removed} Duplikat(e)")
Ausführungsmodell: Webhook-Scripts laufen synchron im Request-Thread (Timeouts setzen!). Event-Hooks und Scheduled Hooks laufen in Daemon-Threads im Hintergrund — Exceptions brechen keine Server-Operationen ab.

9) Datenmodelle

Connection-Objekt (Sync + API)

{
  "id": "uuid",
  "name": "Prod DB",
  "kind": "ssh | rdp | web",
  "host": "example.local",
  "port": 22,
  "username": "admin",
  "domain": "",
  "keyPath": "~/.ssh/id_ed25519",
  "url": "",
  "notes": "",
  "tags": ["prod", "db"],
  "trustCert": false,
  "lastUsed": "2026-01-27T10:39:21.574Z"
}

Settings (Client)

  • mode: local oder sync
  • url: Sync-URL (HTTPS)
  • intervalMinutes: 1 bis 1440
  • storePasswords: lokaler RDP-Keychain-Speicher
  • allowSelfSignedCerts: akzeptiert self-signed TLS im Sync-Request

10) Security-Mechanismen im Code

  • Passwörter werden nicht im Connection-JSON gespeichert; optional nur lokal für RDP.
  • Server-API-Keys werden gehasht (SHA-256) in der DB gespeichert.
  • IP-Filter unterstützt CIDR und optionale Proxy-Ketten-Auswertung.
  • Sync akzeptiert clientseitig ausschließlich https:// URLs.
  • Hook-Scripts laufen mit vollständigen Builtins (inkl. import) in einem isolierten Namespace; Zugriff nur über explizit injizierte Funktionen möglich.
Empfehlung: Für Reverse-Proxy-Betrieb immer TRUSTED_PROXIES setzen und nicht global TRUST_PROXY_HEADERS=true verwenden.

11) CI/CD Pipeline (.gitlab-ci.yml)

  • docker: baut/pusht Server-Image in Registry.
  • download: lädt frpc Binary (v0.61.1) herunter.
  • build: erstellt parallel:
    • Desktop: DEB + RPM + Linux-Binary (Debian Trixie)
    • Go Agent: Linux + Windows Binaries, DEB + RPM Pakete
  • publish: DEB-Publishing via Aptly-API-Script (bei Tags).

Das Script desktop/scripts/ci_publish_via_api.sh erwartet mTLS- und Aptly-Variablen (URL, Zertifikate, Repo, Distribution, GPG-Key).

12) Updaten

Updates sollten komponentenweise und reproduzierbar erfolgen. Für den Server ist der Standardpfad Docker-Image-Update, für den Client ein neuer Build.

Server updaten

docker compose pull
docker compose up -d

Vor dem Update Backup von data/, certs/ und .env erstellen.

Client updaten

cd desktop/src-tauri
cargo tauri build

Neue Version verteilen und Plattform-Abhängigkeiten (z. B. FreeRDP auf Linux) mitprüfen.

Empfehlung: Versionssprünge immer gegen Sync, Auth und Webhook-Funktionalität testen.

13) Typischer Erweiterungs-Workflow

  1. Frontend-Datenmodell in desktop/src/connectionModel.js oder settingsModel.js erweitern.
  2. Rust-Modell in desktop/src-tauri/src/models.rs anpassen.
  3. Tauri-Command ergänzen (commands.rs, main.rs).
  4. Server-Schema/Router anpassen, falls API-Auswirkung vorhanden.
  5. In Sync- und Admin-UI-Verhalten gegenprüfen.

14) Troubleshooting

Sync schlägt fehl

Client verlangt HTTPS. Für self-signed Zertifikate Option allowSelfSignedCerts aktivieren.

RDP auf Linux startet nicht

Prüfen, ob xfreerdp3 oder xfreerdp im PATH verfügbar ist.

Server Login funktioniert nicht

Initiales ADMIN_PASSWORD gilt nur bei Erstinitialisierung der DB.

403 trotz gültigem Token

IP-Filter prüfen: ALLOWED_IPS, TRUSTED_PROXIES.
