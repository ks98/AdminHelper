Agent-Deployment

Der Go-Agent läuft auf jedem überwachten Server. Er übernimmt zwei Aufgaben: Metrik-Collector-Läufe und FRP-Client-Auto-Provisioning. Ausgeliefert wird er als DEB, RPM (Linux) oder MSI/ZIP (Windows).

Was macht der Agent?

  • Monitoring: läuft alle 5 Minuten (systemd.timer mit Jitter), sammelt CPU, RAM, Disk, SMART, Docker-Container-Status, ZFS-Pools, sendet die Metriken tunnelfrei an den AdminHelper-Server (Port 443) unter POST /api/monitoring/agent/{server_id}/report, der sie intern an den Monitoring-Dienst weiterreicht.
  • FRP-Client: lädt beim Erst-Start mit einem Provision-Token sein Client-Zertifikat und startet frpc.service.
  • Updates: neues DEB/RPM von der GitHub-Releases-Seite installieren; Konfig und Service werden bei Upgrades nicht überschrieben. Hinweis: Beim Umstieg auf das interne-only-Monitoring (Server-Proxy statt Host-Port 8480) müssen bestehende Agents einmal neu provisioniert werden, damit ihre gespeicherte MONITOR_URL auf /api/monitoring umgestellt wird.

Paket installieren (Debian/Ubuntu)

Die DEB- und RPM-Pakete hängen als Assets an jedem GitHub-Release. Datei herunterladen und lokal installieren:

bash
VERSION=0.26.0
curl -fsSLO "https://github.com/ks98/AdminHelper/releases/download/v${VERSION}/adminhelper-agent_${VERSION}_amd64.deb"

# Installiert inkl. Abhaengigkeiten; verdraengt alte srm-* Pakete (Replaces/Conflicts):
sudo apt install "./adminhelper-agent_${VERSION}_amd64.deb"

Paket installieren (Rocky/Alma/RHEL)

Das passende .rpm-Asset von der Releases-Seite laden, dann lokal installieren:

bash
sudo dnf install ./adminhelper-agent-*.x86_64.rpm

Erst-Provisionierung

Nach Installation läuft der Agent noch nicht produktiv – er hat weder einen Server-URL noch ein FRP-Client-Zertifikat. Provisionierung in zwei Schritten:

  1. Im Web-Panel → Server → Neuen Server registrieren klicken. Das System legt einen Server-Eintrag an (mit einer Server-ID) und generiert einen einmaligen Provision-Token (24h gültig). Beide brauchst du im nächsten Schritt.
  2. Auf dem Ziel-Server:
    bash
    sudo adminhelper-agent provision \
    --url       https://srm.example.com \
    --token     <provision-token> \
    --server-id <server-id>

Der Agent ruft /api/servers/{id}/provision/activate auf, bekommt je nach Konfiguration Server-API-Key, Monitor-Key, frpc.toml und einen Enrollment-Token, aktiviert adminhelper-agent.timer und (falls FRP) frpc.service. Der Token wird nach Einlösung ungültig.

mTLS-Enrollment (automatisch): Beim Provisioning holt der Agent zusätzlich ein eigenes Client-Zertifikat vom ca-issuer — ECDSA-Key on-device (verlässt den Host nie), CSR über die Enroll-Plane des Gateways auf Port 8444. Cert + Key landen unter /etc/adminhelper/identity/ (Key 0600), die interne Root-CA wird gepinnt. Danach weist sich der Agent bei allen Server-Pushes mit diesem Cert aus und erneuert es automatisch bei ~50 % Laufzeit. Best-effort: schlägt das Enrollment fehl (z. B. Port 8444 nicht erreichbar), läuft der Agent vorerst mit dem API-Key weiter.

Ansible-One-Liner

Für Rollouts auf viele Hosts: ein Playbook mit apt/yum-Modul plus command: adminhelper-agent provision .... Beispiel-Role im Repo unter apps/agent/ansible/.

Subcommand-Referenz

Alle Subcommands der Agent-Binary. Erforderliche Flags sind mit * markiert.

BefehlFunktionWichtige Flags
versionZeigt die Version an—
runFRPC-Sync + Monitor-Push ausführen (ohne Flag Dauerbetrieb, alle 5 Minuten)--once
provisionServer gegen AdminHelper provisionieren (Server-API-Key + optional Monitor + FRP)--url*, --token*, --server-id*, --cacert, --insecure
frpc syncEinmaliger Config-Sync—
monitor initErsteinrichtung des Monitor-Agents--url*, --api-key*, --server-id*, --services, --cacert, --insecure
monitor pushEinmaliger Metriken-Push—
service installOS-Service registrieren und starten—
service uninstallOS-Service deregistrieren—

--insecure gilt nur fuer den einmaligen provision-Aufruf: der Agent erfasst dabei das vom Server praesentierte Zertifikat und pinnt es (Trust-on-first-use), sodass die wiederkehrende Sync-/Push-Schleife wieder verifiziert — statt die TLS-Pruefung dauerhaft abzuschalten. Mit --cacert wird gar nichts uebersprungen.

Windows (MSI)

Die Windows-Binary adminhelper-agent.exe wird als signiertes MSI verteilt. Nach Installation:

powershell
adminhelper-agent.exe provision `
    --url       https://srm.example.com `
    --token     <provision-token> `
    --server-id <server-id>

Der Agent registriert sich als Windows-Service (adminhelper-agent) und startet automatisch beim Boot.

Prüfen, ob der Agent läuft

bash
systemctl status adminhelper-agent.timer   # Linux – nächste Aktivierung
systemctl status frpc.service              # FRP-Client-Verbindung
journalctl -u adminhelper-agent.service --since "1 hour ago"

Upgrade

Bei gesetztem Apt-Repo reicht apt upgrade adminhelper-agent auf allen Servern. Die systemd-Units und die Konfig (/etc/adminhelper/agent.toml) bleiben erhalten.
