Installation

AdminHelper wird als Docker-Compose-Stack betrieben: Server, Postgres, Redis, FRP-Server (frps), Monitoring-Service und VictoriaMetrics. Diese Seite führt durch eine frische Installation inklusive Reverse-Proxy-Setup und Erstanmeldung.

Voraussetzungen

  • Linux-Host mit Docker 24+ und Docker-Compose v2
  • Öffentlich erreichbarer Port 443 (Web/API) und 8444 (Enrollment); 7000 und 7443 für FRP
  • DNS-Eintrag, der auf den Host zeigt (srm.example.com)
  • Kein externer Reverse-Proxy nötig: der Stack bringt ein eigenes TLS-Gateway (nginx) mit, das auf 443 terminiert (siehe unten)

Stack starten

Die offizielle docker-compose.yml liegt im Repository-Root. Für einen schnellen Test reicht:

bash
git clone https://github.com/ks98/AdminHelper.git
cd AdminHelper

cp .env.example .env

# Sichere Zufalls-Secrets generieren (SECRET_KEY, MONITOR_API_KEY,
# POSTGRES_PASSWORD, CA_ROOT_PASSPHRASE).
# Idempotent: laesst bereits gesetzte Werte in Ruhe.
./scripts/init-secrets.sh

# Anschliessend in .env die nicht-Secret-Variablen anpassen:
#   DOMAIN, ADMIN_PASSWORD, TZ, TRUSTED_PROXIES, ALLOWED_IPS

docker compose pull
docker compose up -d
MONITOR_API_KEY ohne Init-Script = Monitoring-UI tot

Server- und Monitoring-Container teilen sich MONITOR_API_KEY als internes Shared Secret. Bleibt der Wert in .env leer, generiert sich der Monitoring-Container einen eigenen Random — der Server schickt aber einen leeren Key, jeder /api/monitoring/*-Aufruf endet mit 401. ./scripts/init-secrets.sh verhindert das.

Genauso wichtig: SECRET_KEY. Bleibt er auf change-me-in-production oder leer, generiert der Server beim ersten Start einen Random in ./data/.secret_key — funktioniert, ist aber nicht portabel zwischen Hosts. Init-Script schreibt einen festen Wert in die .env, damit der Stack reproduzierbar migrierbar bleibt.

TLS-Gateway (eingebaut)

Der Stack enthält ein eigenes Gateway (nginx), das als einzige öffentliche TLS-Kante auf 443 terminiert und intern auf den server (plain-HTTP, kein Host-Port) weiterleitet. server und ca-issuer sind nur im Compose-Netz erreichbar — dadurch ist die vom Gateway gesetzte Client-Identität (mTLS-Header) nicht von außen fälschbar.

Das TLS-Zertifikat des Gateways stellt der interne ca-issuer beim ersten Start aus (signiert von der access-Intermediate der internen CA, SAN = DOMAIN). Da diese CA nicht öffentlich vertraut ist, zeigt der Browser zunächst eine Zertifikatswarnung — erwartet; die Root-CA wird im Vertrauensmodell der Clients gepinnt (siehe Troubleshooting). Ein externer Reverse-Proxy ist nicht erforderlich; wer trotzdem einen davorsetzen will, terminiert dessen eigenes TLS und leitet an das Gateway auf 443 weiter (und trägt dessen IP in TRUSTED_PROXIES ein).

Client-Zertifikate (mTLS). Agent und Desktop holen ihr Client-Zertifikat automatisch von der internen CA (Enrollment, siehe Entwickler-Übersicht „PKI & mTLS"). Für den Browser exportiert der Desktop-Client ein PKCS12 (.p12, passwortgeschützt), das in den Zertifikatsspeicher des Browsers importiert wird (Schritt-für-Schritt unter Benutzer & Zugriff → Browser-Zugriff). Aktuell läuft die Datenebene permissiv — ein Client-Cert ist noch nicht erzwungen; die mTLS-Pflicht (CERT_REQUIRED) wird separat scharfgeschaltet.

Erstanmeldung – Bootstrap-Token

Es gibt keinen fest verdrahteten Default-Login mehr. Beim ersten Start mit leerem ADMIN_PASSWORD generiert der Server einen einmaligen Setup-Token nach data/.bootstrap_token und zeigt ihn im WARNING-Log:

bash
docker compose logs server | grep -A2 'Setup-Token'

Mit dem Token den ersten Admin anlegen:

bash
curl -k -X POST https://srm.example.com/api/auth/bootstrap \
  -H 'Content-Type: application/json' \
  -d '{"token":"<TOKEN>","username":"admin","password":"<eigenes-passwort>"}'

Antwort enthält sofort access_token und refresh_token. Die Token-Datei wird nach erfolgreichem Bootstrap geloescht. Der Endpoint kann nicht erneut aufgerufen werden, sobald ein User existiert (409-Antwort).

CI / Power-User: ADMIN_PASSWORD direkt setzen

Wenn du in .env einen anderen Wert als admin oder leer setzt (z. B. ADMIN_PASSWORD=initialpw), wird beim ersten Start direkt ein User admin mit diesem Passwort angelegt – kein Bootstrap-Schritt nötig. Praktisch für CI-Pipelines.

Health-Check

Der Server liefert unter /api/health einen kurzen JSON-Status. Nutze den Endpunkt für Monitoring-Systeme oder Docker-Healthchecks.

Updates

Der Server läuft aus der GitHub Container Registry ghcr.io/ks98/adminhelper/server. Standard-Tags:

  • :latest – letzter veröffentlichter Release (Semver-Tag)
  • :0.26.0 – konkreter Release-Tag
  • :main – letzter Build der Default-Branch
  • :sha-<commit> – deterministisches Image pro Commit
bash
docker compose pull server
docker compose up -d server

Datenbank-Migrationen laufen automatisch beim Container-Start via Alembic (siehe server/alembic/versions/). Der Entrypoint wartet erst auf pg_isready, führt dann alembic upgrade head aus, dann erst startet uvicorn. Der Agent hat einen eigenen Release-Zyklus – siehe Agent-Deployment.
