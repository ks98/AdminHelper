Benutzer & Zugriff

AdminHelper unterscheidet drei Zugriffsarten: interaktive Benutzer (Web- und Desktop-Login), API-Keys (Agenten und Skripte) sowie FRP-Provision-Tokens (Einmalverwendung). Diese Seite erklärt, wie du alle drei verwaltest.

Benutzerrollen

RolleDarf
adminAlles: Benutzer verwalten, API-Keys erstellen, Server registrieren, Hooks definieren, globale Einstellungen.
userEigene Verbindungen pflegen, Server und Monitoring lesen. Keine Admin-Aktionen.

Benutzer anlegen

  1. Im Web-Panel Benutzer → Neu öffnen
  2. Username und Passwort vergeben; Passwort wird mit bcrypt gehasht
  3. Rolle wählen – Admins sehen und bearbeiten alles

Admins können jederzeit fremde Passwörter zurücksetzen. Normale Benutzer können ihr eigenes Passwort in den Einstellungen ändern.

Login-Flow

Der Login-Flow nutzt JWT mit Refresh-Token-Rotation:

  • Access-Token: 8h gültig, wird bei jedem Request im Authorization-Header mitgeschickt.
  • Refresh-Token: 7 Tage gültig, rotiert bei jeder Verwendung (der alte wird sofort blacklisted).
  • Reuse-Detection: Wird ein geblacklisteter Refresh-Token erneut verwendet, werden alle Sessions des Users sofort invalidiert – ein Hinweis auf einen gestohlenen Token.
Login-Rate-Limit

5 fehlgeschlagene Login-Versuche pro IP innerhalb von 60 Sekunden → weitere Versuche werden mit 429 abgewiesen. Ein erfolgreicher Login setzt den Zähler zurück. Implementierung via Redis (fixed-window), mit In-Memory-Fallback für Single-Worker-Setups.

Browser-Zugriff (Client-Zertifikat)

Sobald die Datenebene auf CERT_REQUIRED scharfgeschaltet ist, braucht jeder Browser, der das Web-Panel öffnet, ein gültiges Client-Zertifikat aus der internen CA. Anders als Agent und Desktop kann sich der Browser kein Zertifikat selbst holen – du exportierst es einmalig aus dem Desktop-Client und importierst es in den Zertifikatsspeicher des Browsers.

  1. Exportieren: Desktop-Client → Einstellungen (im Server-Modus, angemeldet) → Browser-Zertifikat exportieren. Ein Passwort (mind. 8 Zeichen) vergeben und Exportieren klicken. Der Client enrollt ein langlebiges access-Zertifikat, verpackt es als passwortgeschützte .p12 (adminhelper-browser.p12) im App-Datenverzeichnis und zeigt den vollständigen Pfad an.
  2. Importieren – Chrome/Edge: Einstellungen → Datenschutz und Sicherheit → Zertifikate verwalten → Eigene Zertifikate → Importieren, die .p12 wählen, Passwort eingeben. (Unter Windows/macOS landet das Zertifikat im Betriebssystem-Speicher; die genaue Menüführung variiert je nach Version.)
  3. Importieren – Firefox: Einstellungen → Datenschutz & Sicherheit → Zertifikate → Zertifikate anzeigen → Ihre Zertifikate → Importieren, die .p12 wählen, Passwort eingeben.
  4. Verbinden: Web-Panel neu laden. Beim ersten Aufruf fragt der Browser, welches Client-Zertifikat er vorzeigen soll – das importierte auswählen.
Browser-Cert unter erzwungenem mTLS

Seit 0.29.0 läuft die Datenebene per Default scharf (CERT_REQUIRED) – ein Browser ohne importiertes Zertifikat wird schon am TLS-Handshake abgewiesen, das Web-Panel lädt dann nicht. Verteile die .p12 also an jeden, der das Panel im Browser nutzt. Das Browser-Zertifikat ist langlebig und erneuert sich nicht automatisch – vor Ablauf einfach neu exportieren und importieren.

Neuen Nutzer unter erzwungenem mTLS onboarden

Ist die Datenebene scharf (CERT_REQUIRED, siehe Betrieb & Konfiguration), kommt ein brandneuer Client ohne Zertifikat nicht durch den :443-Handshake – auch nicht zum Login. Das Onboarding läuft daher über die entkoppelte Enrollment-Tür (ADR 0003): erst Zertifikat per Token, dann Login.

  1. Benutzer anlegen (Admin) – wie oben unter „Benutzer anlegen". Der spätere Zertifikats-CN ist genau dieser Username.
  2. Enrollment-Token ausstellen (Admin, selbst bereits mit Cert): POST /api/enrollment/token/for mit { "username": "<neuer-user>" }. Die Antwort enthält ein einmaliges, 10 Minuten gültiges token.
  3. Token out-of-band übergeben – per sicherem Kanal an den neuen Nutzer (Copy-Paste). Es ist single-use und kurzlebig.
  4. Gerät enrollen (neuer Nutzer): im Desktop-Client „Erstes Mal? Gerät mit Token einrichten" wählen, Server-URL + Token eingeben → Gerät enrollen. Der Client erzeugt on-device einen Schlüssel, holt sein Zertifikat an :8444 und legt es im OS-Keyring ab. Kein Login nötig.
  5. Anmelden – danach normaler Login (Username/Passwort); der Client präsentiert jetzt sein Zertifikat am Handshake.
  6. Optional Browser – für das Web-Panel exportiert der Nutzer aus dem Desktop ein .p12 und importiert es (siehe „Browser-Zugriff" oben).
Ausnahme: der allererste Admin

Für den ersten Admin gibt es noch niemanden, der ein Token ausstellen könnte. Am einfachsten erledigt das scripts/install.sh (legt Admin + Erst-Token out-of-band an und schaltet scharf). Manuell ohne Script: ein einmaliges permissives Fenster (MTLS_ENFORCE=false, ersten Admin anlegen + Desktop enrollen, dann MTLS_ENFORCE=true). Ab da läuft jedes weitere Onboarding über die Token-Tür – ohne erneutes Lockern.

API-Keys

API-Keys authentifizieren nicht-interaktive Clients (Go-Agent, Skripte, CI-Jobs). Unterschiede zu User-Credentials:

  • Kein Ablauf, aber widerrufbar per Klick
  • Eine von zwei Berechtigungen: read (nur lesen) oder read_write (lesen und schreiben)
  • Server speichert nur den SHA-256-Hash, der Klartext ist nach dem Anzeigen weg

API-Keys erstellen:

  1. Web-Panel → API-Keys → Neu
  2. Name vergeben und Berechtigung (read oder read_write) wählen
  3. Klartext-Key sofort kopieren – wird nie wieder angezeigt
  4. Als HTTP-Header X-API-Key: <key> senden (Fallback: Query-Parameter ?api_key=… für Sync-URLs)

IP-Restriktion

Mit ALLOWED_IPS lassen sich alle Zugriffe auf eine Whitelist begrenzen. Beispiel:

.env
ALLOWED_IPS=10.0.0.0/8,192.168.1.0/24,203.0.113.42

Wird die Liste leer gelassen, ist der Zugriff offen (für Reverse-Proxy- oder VPN-geschützte Setups).
