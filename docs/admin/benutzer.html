Benutzer & Zugriff

AdminHelper unterscheidet drei Zugriffsarten: interaktive Benutzer (Web- und Desktop-Login), API-Keys (Agenten und Skripte) sowie FRP-Provision-Tokens (Einmalverwendung). Diese Seite erklärt, wie du alle drei verwaltest.

Benutzerrollen

RolleDarf
adminAlles: Benutzer verwalten, API-Keys erstellen, Server registrieren, Hooks definieren, globale Einstellungen.
userEigene Verbindungen pflegen, Server und Monitoring lesen. Keine Admin-Aktionen.

Benutzer anlegen

  1. Im Web-Panel Benutzer → Neu öffnen
  2. Username und Passwort vergeben; Passwort wird mit bcrypt gehasht
  3. Rolle wählen – Admins sehen und bearbeiten alles

Admins können jederzeit fremde Passwörter zurücksetzen. Normale Benutzer können ihr eigenes Passwort in den Einstellungen ändern.

Login-Flow

Der Login-Flow nutzt JWT mit Refresh-Token-Rotation:

  • Access-Token: 8h gültig, wird bei jedem Request im Authorization-Header mitgeschickt.
  • Refresh-Token: 7 Tage gültig, rotiert bei jeder Verwendung (der alte wird sofort blacklisted).
  • Reuse-Detection: Wird ein geblacklisteter Refresh-Token erneut verwendet, werden alle Sessions des Users sofort invalidiert – ein Hinweis auf einen gestohlenen Token.
Login-Rate-Limit

5 fehlgeschlagene Login-Versuche pro IP in 15 Minuten → 5-Minuten-Sperre. Implementierung via Redis, mit In-Memory-Fallback für Single-Worker.

API-Keys

API-Keys authentifizieren nicht-interaktive Clients (Go-Agent, Skripte, CI-Jobs). Unterschiede zu User-Credentials:

  • Kein Ablauf, aber widerrufbar per Klick
  • Eigener Scope: z. B. nur Monitoring-Ingest oder nur Server-Read
  • Server speichert nur den SHA-256-Hash, der Klartext ist nach dem Anzeigen weg

API-Keys erstellen:

  1. Web-Panel → API-Keys → Neu
  2. Label + Scope wählen
  3. Klartext-Key sofort kopieren – wird nie wieder angezeigt
  4. Als Authorization: Bearer <key> verwenden

IP-Restriktion

Mit ALLOWED_IPS lassen sich alle Zugriffe auf eine Whitelist begrenzen. Beispiel:

.env
ALLOWED_IPS=10.0.0.0/8,192.168.1.0/24,203.0.113.42

Wird die Liste leer gelassen, ist der Zugriff offen (für Reverse-Proxy- oder VPN-geschützte Setups).

Audit-Log

Admin-Aktionen werden in die DB-Tabelle audit_log geschrieben: Benutzer-Änderungen, API-Key-Erzeugung/-Widerruf, Server-Registrierungen, Hook-Ausführungen. Im Web-Panel unter Admin → Audit einsehbar.
