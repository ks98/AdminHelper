FRP-Tunnel

FRP (Fast Reverse Proxy) baut verschlüsselte, bidirektionale Tunnel zwischen Agent und Server auf. So erreichst du RDP/SSH/HTTP auf Servern hinter NAT oder Firewalls ohne Port-Forwarding.

Architektur

Drei Komponenten:

  • frps – läuft im Server-Stack, exponiert Port 7000 (Kontrollkanal) und 7443 (Tunnel).
  • frpc – läuft auf jedem Agent-Host, verbindet sich zu frps und meldet Services an.
  • PKI – die interne tunnel-Intermediate des ca-issuer signiert das frps-Server-Cert und die Agent-Tunnel-Certs; jeder frpc authentifiziert sich per mTLS (seit A7 eine einheitliche Vertrauenswurzel statt einer frps-eigenen CA).

PKI-Setup

Die Tunnel-PKI ist Teil der einheitlichen internen CA (ADR 0001):

  • frps-Server-Cert: vom ca-issuer unter tunnel ausgestellt (ECDSA P-256, SAN = öffentliche frps-Adresse), beim ersten Start ins frps-Volume provisioniert. Der CA-Key erreicht das internet-zugewandte frps nie.
  • Agent-Tunnel-Cert: der Agent enrollt es beim Provisioning on-device (A4) — dasselbe tunnel-Cert dient Server-Pushes und dem frp-Tunnel.
  • trustedCaFile: beidseitig die tunnel-Kette (Intermediate + Root).
CA-Backup nicht vergessen

Das Security-Asset ist jetzt das ca-pki-Volume des ca-issuer (Root + Intermediates). Verlust bedeutet, dass alle Clients neu enrollen müssen. Backup-Details unter Datenverzeichnis-Backup.

Client-Zertifikate rotieren

Das Agent-Tunnel-Cert wird vom Agent automatisch erneuert, sobald es ~50 % seiner Laufzeit erreicht (über /ca/renew, mit dem aktuellen Cert als Nachweis) — kein manueller Rotate-Trigger nötig. Der Desktop-Visitor präsentiert seit der PKI-Migration seine enrollte access-Identität (dieselbe wie für den Web-/API-Zugriff) und erneuert sie automatisch — kein server-gemüntztes Visitor-Cert mehr.

Provision-Token

Jeder neue Agent bekommt beim Registrieren einen Einmal-Token. Token-Eigenschaften:

  • 24h Gültigkeit
  • Einmal einlösbar (server-seitig nach Verwendung sofort invalidiert)
  • Bindet an eine Server-ID, d. h. der Token passt nur zu einem einzigen Agent-Eintrag

Tunneltypen

TypUse-Case
tcpSSH, RDP, VNC, beliebige TCP-Services
http/httpsWeb-UIs mit Virtual-Host-Routing (idrac.srv1.example.com)
stcpSecure-TCP, braucht Visitor auf Client-Seite – Desktop-Client löst das automatisch.

Visitor-Ports

STCP-Tunnel brauchen lokal einen freien Port für den Visitor. Der Desktop-Client wählt zufällig aus dem Bereich VISITOR_PORT_START–VISITOR_PORT_END (Default 6000–6999). Bei Firewall-Restriktionen den Bereich über die .env verkleinern.

Troubleshooting

  • Agent verbindet sich nicht: Zertifikats-Ablauf prüfen (openssl x509 -in /etc/adminhelper/client.crt -noout -enddate), Server-Erreichbarkeit via nc -vz srm.example.com 7000.
  • "Tunnel not found": Service im Panel aktiv? FRP-Logs: docker compose logs -f frps.
  • STCP klappt nicht: Visitor-Port bereits belegt? Desktop-Client-Log prüfen.
