Monitoring

Das Monitoring-System besteht aus drei Komponenten: Checks (Konfigurationen, was wie oft geprüft wird), Collectors im Agent (führen Messungen durch) und VictoriaMetrics (speichert die Zeitreihen). Charts im Web-Panel lesen direkt von VictoriaMetrics via uPlot.

Check-Typen

Der Typ-Bezeichner (check_type) ist exakt der Wert, den das Web-Panel im Dropdown anzeigt. Server-seitige Probes (ping, tcp, http) laufen direkt im Monitoring-Container, alle agent_*- und Plugin-Checks werten Push-Daten des Agents aus.

TypWas wird gemessenWo läuft er
pingICMP-Erreichbarkeit + Round-Trip-TimeMonitoring-Service (zentral)
tcpPort-Erreichbarkeit + Connect-ZeitMonitoring-Service
httpHTTP-Status-Code, Latenz, optional Text-Suche im BodyMonitoring-Service
agent_pingHat sich der Agent innerhalb von stale_minutes gemeldet?Agent (Push)
agent_resourcesCPU, RAM, Disk-Mountpoints und Temperatur-Sensoren gegen ThresholdsAgent (Push)
service_processsystemd-Units: Modus auto (failed + enabled-but-inactive) oder list (explizite Service-Namen)Agent (Push)
proxmox_backupBackup-Alter aller VMs/CTs gegen max_backup_age_hoursAgent (Push)
zfs_healthPool-Health (DEGRADED/FAULTED/…) + Kapazität gegen capacity_warn/critAgent (Push)
docker_healthContainer-State + unhealthy-Erkennung, optional ignore_containersAgent (Push)
smart_healthSMART-Werte + Temperatur für ATA (HDD/SSD) und NVMeAgent (Push)
Agent-Plugins werden automatisch erkannt

Die Plugin-Checks proxmox_backup, zfs_health und docker_health werten nur dann Daten aus, wenn der Agent das jeweilige Subsystem (Proxmox, ZFS, Docker) auf dem Host findet. Fehlt das Subsystem, bleibt der Check ohne Daten – kein Fehler.

Check anlegen

  1. Web-Panel → Monitoring → Neu
  2. Typ wählen, Ziel (Server oder URL) auswählen
  3. Parameter ausfüllen – typabhängig, z. B. bei http: URL, erwartete Codes, Timeout
  4. Intervall und Alert-Thresholds setzen

Alert-Regeln

Jeder Check hat bis zu drei Thresholds:

  • ok bis warn_threshold
  • warn zwischen warn_threshold und crit_threshold
  • crit oberhalb crit_threshold

Bei crit-Übergängen wird ein Hook gefeuert (alert.triggered). Details zur Hook-Konfiguration: Hooks & Events.

SMART-Monitoring

Der Agent erkennt automatisch SATA-, SAS- und NVMe-Festplatten (smartctl --scan-open). Für NVMe werden die Critical-Warning-Bits einzeln dekodiert:

BitBedeutung
0Spare-Capacity kritisch niedrig
1Temperatur außerhalb der Grenzwerte
2Device Reliability degradiert
3Read-Only-Mode
4Volatile Memory Backup fehlgeschlagen
5Persistent Memory Read-Only

VictoriaMetrics

VictoriaMetrics speichert Zeitreihen 90 Tage lang (-retentionPeriod=90d in der Compose-Datei). Pro Metrik werden vier Labels geschrieben: server, check, type, plus check-spezifische Labels wie mountpoint oder device. Direkter Zugriff (Debug): http://victoria:8428/api/v1/query?query=up.

Ingest-Authentifizierung

Der Agent sendet seine Metriken per POST /api/monitoring/agent/{server_id}/report an den AdminHelper-Server (Port 443, tunnelfrei) und authentifiziert mit seinem server-spezifischen X-API-Key (beim Provisioning vergeben). Der geteilte MONITOR_API_KEY ist davon getrennt: er sichert nur die interne Server→Monitoring-Proxy-Strecke (X-Internal-Key) und muss zwischen Server- und Monitoring-Container identisch sein.

Charts im Web-Panel

Die Monitoring-Seite kombiniert uPlot-Liniencharts mit einer Alert-Liste. Zeitfenster wählbar: 1h, 6h, 24h, 7d, 30d. Klick auf eine Linie öffnet Detail-Ansicht mit Rohdaten.

Templates

Ein Monitoring-Template bündelt mehrere Check- und Alert-Definitionen und wird einem Server zugewiesen, statt jeden Check einzeln anzulegen. Beim Zuweisen werden aus den Definitionen echte Checks und Alert-Regeln für den Server erzeugt.

  1. Web-Panel → Monitoring → Templates → Template anlegen, Check- und Alert-Definitionen hinzufügen
  2. Template einem Server zuweisen – AdminHelper erstellt alle enthaltenen Checks und Alert-Regeln
  3. Wird das Template später geändert, läuft ein Live-Sync: alle zugewiesenen Server werden automatisch aktualisiert (Checks erstellt, geändert oder gelöscht). Jede Definition trägt eine stabile def_id, über die der Sync zuordnet.

In Definitionen lassen sich Platzhalter verwenden, die beim Zuweisen pro Server ersetzt werden:

PlatzhalterErsetzt durch
{{hostname}}Hostname des Servers
{{server_name}}Anzeigename des Servers
{{server_id}}Interne Server-ID

Eine Zuweisung entfernen löscht die daraus erzeugten Checks und Alerts wieder. Ein Template zu löschen entfernt diese auf allen zugewiesenen Servern.

Benachrichtigungen & Alerting

Alert-Regeln feuern, wenn ein Check seinen Status wechselt (z. B. ok → critical). Jede Regel filtert optional nach severity und Server und versendet über einen Kanal.

KanalKonfiguration
webhookurl – AdminHelper sendet ein JSON-POST mit Check-Name, Typ, Server, Severity, altem/neuem Status und Klartext-Nachricht (Timeout 10 s, Erfolg bei HTTP < 300)
emailrecipients (oder to, auch als CSV). SMTP standardmäßig aus den Umgebungsvariablen SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD, SMTP_FROM; pro Regel überschreibbar. Bei Port 587 wird STARTTLS genutzt.

Cooldown: Nach einem erfolgreich versendeten Alert unterdrückt cooldown_minutes (Default 30) weitere Alerts derselben Regel für denselben Check.

Recovery: Geht ein Check zurück auf ok, wird eine Recovery-Benachrichtigung versendet – diese umgeht den Cooldown bewusst, damit die Entwarnung immer ankommt. Jeder Versand wird im Alert-Log mit Erfolg/Fehler protokolliert.
