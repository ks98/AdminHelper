Ansible

Die Ansible-Verwaltung trennt zentrale Pflege von lokaler Ausführung: Playbooks werden im Web-Panel gespeichert und versioniert, ausgeführt werden sie über den Desktop-Client mit lokal installiertem ansible-playbook. Der Server führt selbst keine Playbooks aus.

Playbook-CRUD im Web-Panel

Playbooks werden über /api/ansible/playbooks verwaltet – alle Schreib-Operationen sind Admin-only. Jedes Playbook hat Metadaten in der Datenbank (Tabelle ansible_playbooks) und eine YAML-Datei auf Disk unter DATA_DIR/ansible/playbooks/<id>/<filename>.

FeldBedeutung
nameAnzeigename im Panel und Desktop-Client
filenameDateiname der YAML-Datei – muss auf .yml oder .yaml enden, keine Pfad-Separatoren
descriptionOptionale Beschreibung
tagsListe von Tags zur Filterung/Suche
contentDer YAML-Inhalt – wird auf Disk geschrieben, nicht in der DB

Beim Erstellen (POST) und beim Bearbeiten (PUT, falls content mitgesendet wird) validiert der Server das YAML mit yaml.safe_load; ungültiges YAML liefert 422. Wird der Dateiname geändert, benennt der Server die Datei auf Disk um bzw. löscht die alte. Beim Löschen (DELETE) wird das gesamte Verzeichnis <id>/ entfernt.

HTTP
GET    /api/ansible/playbooks              # Alle Playbooks auflisten
POST   /api/ansible/playbooks              # Playbook erstellen (Admin)
GET    /api/ansible/playbooks/{id}         # Playbook-Metadaten
GET    /api/ansible/playbooks/{id}/content # YAML-Inhalt abrufen
PUT    /api/ansible/playbooks/{id}         # Playbook aktualisieren (Admin)
DELETE /api/ansible/playbooks/{id}         # Playbook löschen (Admin)

CRUD-Operationen feuern Hooks: playbook.created, playbook.updated und playbook.deleted (Payload: id + name). Details zur Hook-Konfiguration: Hooks & Events.

3-Schritt-Workflow (Desktop)

Der Desktop-Client führt durch einen dreistufigen Assistenten. Schritt 2 wird erst aktiv, wenn ein Playbook gewählt ist; Schritt 3 erst, wenn zusätzlich mindestens ein Ziel-Server ausgewählt wurde.

  1. Playbook auswählen – Liste aller im Server gepflegten Playbooks mit Name, Beschreibung, Dateiname und Tags.
  2. Ziel-Server auswählen – zwei Modi: Server (Einzelauswahl per Checkbox) oder Tags (Chip pro Tag wählt alle Server mit diesem Tag).
  3. Ausführen – Zusammenfassung (Playbook + Anzahl Server), dann Start via ansible-playbook in einem nativen Terminal.

Beim Ausführen lädt der Client den YAML-Inhalt frisch vom Server (/{id}/content), schreibt ihn in eine temporäre Datei, generiert das Inventory und startet den Lauf. Server und Playbooks bezieht der Client aus dem Server-Web-Interface – lokal wird nichts gepflegt.

Inventory-Generierung

Der Desktop-Client erzeugt aus den ausgewählten Servern automatisch eine Inventory-Datei im INI-Format in einem temporären Verzeichnis. Alle gewählten Hosts landen unter [all]; zusätzlich entsteht pro Server-Tag eine Gruppe (alphabetisch sortiert) mit den zugehörigen Hosts.

INI
[all]
web01.example.com
db01.example.com

[database]
db01.example.com

[web]
web01.example.com

Die Gruppen entstehen direkt aus den Tags der Server – dieselben Tags, die im Web-Panel beim Server gesetzt sind und im Tag-Modus die Auswahl steuern. Hostnamen sind die hostname-Felder der Server.

Voraussetzungen

  • ansible-playbook muss auf dem Desktop-Rechner installiert und im PATH sein. Fehlt es, bricht der Lauf mit einer Fehlermeldung ab.
  • Die Ziel-Server müssen im Server-Web-Interface unter Server angelegt sein – nur dort gepflegte Hosts erscheinen im Desktop-Assistenten.
  • Playbook-Pflege (Erstellen/Bearbeiten/Löschen) erfordert eine Admin-Session.
SSH-Erreichbarkeit

Die Ausführung läuft auf dem Desktop-Rechner gegen die Hostnamen aus dem Inventory. SSH-Zugang, Keys und ggf. ansible_user regelst du wie gewohnt über deine Ansible-/SSH-Konfiguration – AdminHelper generiert nur Inventory und Playbook-Datei.
