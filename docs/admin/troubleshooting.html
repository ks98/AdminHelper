Troubleshooting

Diagnose-Rezepte für die häufigsten Probleme – gegliedert nach Symptom. Wenn du nichts Passendes findest: docker compose logs ist fast immer der richtige erste Schritt.

Installation bricht ab

password authentication failed for user "adminhelper"

Der Install wartet 240 s und bricht ab; im Server-Log steht FATAL: password authentication failed. Ursache: ein altes postgres-data-Volume aus einem früheren (abgebrochenen) Versuch. Postgres setzt das Passwort nur beim ersten Init des Volumes — ein neu generiertes .env-Passwort (z. B. nachdem der Ordner gelöscht wurde) passt also nicht mehr. Lösung — Volumes wegwerfen (gefahrlos, solange kein echter Datenbestand drin ist):

cd adminhelper
docker compose down -v
# oder direkt beim Install:
./scripts/install.sh … --reset

Stolperfalle: Der Compose-Projektname ist der Verzeichnisname. Zwei Installs in verschiedenen Pfaden mit gleichem Ordnernamen (adminhelper) teilen sich dasselbe Volume adminhelper_postgres-data — daher kann ein „frischer" Ordner trotzdem auf ein altes Volume treffen.

Login funktioniert nicht

„Zugriff verweigert" trotz korrektem Passwort

Zu viele Fehlversuche → Rate-Limit. 5 Min warten oder den Redis-Key manuell löschen:

docker compose exec redis redis-cli keys "ratelimit:*"

Admin-Passwort vergessen

Neues ADMIN_PASSWORD in die .env schreiben, DB-Entry aktualisieren:

docker compose exec server python -m app.cli reset-admin

„Session abgelaufen" direkt nach Login

Uhrzeit-Drift zwischen Host und Client > 5 Min? NTP auf dem Server aktivieren. JWT-iat/exp-Validierung ist zeitabhängig.

TLS / Zertifikats-Probleme

  • Browser warnt vor unbekanntem Zertifikat: Normal. Das Gateway präsentiert auf 443 ein Leaf der internen CA (vom ca-issuer ausgestellt), die nicht öffentlich vertraut ist. Die interne Root-CA im Browser/OS als vertrauenswürdig importieren, oder die Warnung akzeptieren – siehe TLS-Gateway.
  • Desktop-Client meldet certificate verify failed: Eigene CA in die System-Trust-Store des Clients importieren oder in den Desktop-Einstellungen Self-Signed akzeptieren aktivieren.
  • Desktop-Client, „Selbstsignierte Zertifikate erlauben": Beim ersten Verbinden wird der SHA-256-Fingerprint des Server-Zertifikats gepinnt (Trust-On-First-Use, wie SSH known_hosts) und im OS-Keyring abgelegt — danach wird nur noch genau dieses Zertifikat akzeptiert; ein Wechsel wird als mögliche MITM-Attacke abgelehnt. Erscheint nach einer legitimen Zertifikats-Rotation die Meldung AdminHelper TOFU: … hat sich geändert, in den Desktop-Einstellungen Gepinntes Zertifikat zurücksetzen klicken und neu verbinden (pinnt dann neu).
  • Desktop-Client, bereits registriertes Gerät (mTLS): Ist das Gerät registriert, prüft der Client das Server-Zertifikat gegen die bei der Registrierung gepinnte CA (nicht mehr den Leaf-Pin). Nach einer Server-Neuinstallation bzw. neu erzeugten PKI passt die alte CA nicht mehr und die Verbindung scheitert mit AdminHelper: Das Server-Zertifikat wird nicht mehr von der … gepinnten CA gedeckt (mögliche MITM-Attacke). Ist der Wechsel erwartet: in den Desktop-Einstellungen (Modus Server) Geräte-Identität zurücksetzen klicken — das löscht das mTLS-Geräte-Zertifikat und den Zertifikat-Pin —, danach neu anmelden und das Gerät neu registrieren.

Agent meldet sich nicht mehr

  1. systemctl status adminhelper-agent.timer – läuft der Timer?
  2. journalctl -u adminhelper-agent.service --since "1 hour ago"
  3. API-Key widerrufen? Im Desktop-Client (Server-Modus) → Infrastruktur → Server → Tab Provisionierung einen neuen Token erzeugen und am Host einlösen.
  4. Zertifikat abgelaufen? openssl x509 -in /etc/adminhelper/client.crt -noout -enddate

Monitoring zeigt keine Daten

  • VictoriaMetrics erreichbar? docker compose exec monitoring curl -s http://victoria:8428/health
  • Server → Monitoring: MONITOR_API_KEY identisch in beiden Containern?
  • Agent-Debug: adminhelper-agent monitor run --dry-run auf dem Ziel-Host

FRP-Tunnel bricht ab

  • docker compose logs -f frps → Client-Auth-Fehler?
  • Cert-Rotation hängt? Bei Clients mit < 30 Tagen Restlaufzeit triggert der Agent die Rotation – Serverseitig blockieren CRL-Einträge eine neue Ausstellung, CRL-Content prüfen.
  • Proxy / Firewall blockiert Port 7000 oder 7443? Reiner TCP-Check: nc -vz srm.example.com 7000.

Monitoring-Scheduler stoppt nach Upgrade

APScheduler verwendet einen Job-Store in der DB. Bei Schema-Changes werden veraltete Job-Einträge migriert; bei wirklich festgefahrenen Situationen einmalig:

bash
docker compose exec server python -m app.cli rebuild-scheduler

Performance

  • Langsame Charts: VictoriaMetrics-Retention prüfen (-retentionPeriod=90d). Falls DB mehrere GB groß: Downsampling per vmagent aktivieren.
  • FastAPI-Worker überlastet: REDIS_URL setzen und Worker hochdrehen – siehe Betrieb.

Diagnose-Bundle erzeugen

Für einen Bug-Report ein redaktiertes Diagnose-Bundle auf dem Server-Host erzeugen:

bash
./scripts/diagnostics.sh               # erzeugt adminhelper-diagnostics-<ts>.tar.gz
./scripts/diagnostics.sh --tail 1000   # mehr Logzeilen pro Service

Das Bundle enthält report.txt (Versionen, OS, Docker, Service-Status), compose.yml, eine env.sanitized (Secret-Werte maskiert) und die letzten Logzeilen pro Service. Anschließend an ein GitHub-Issue anhängen.

Für den Desktop-Client: Einstellungen → Diagnose → „Diagnose-Bericht erstellen". Für einen Agent auf einem verwalteten Server: adminhelper-agent diagnostics.

Vor dem Senden durchsehen

Secrets (SECRET_KEY, Passwörter, API-Keys, JWT-/Bearer-Tokens) werden automatisch maskiert — die Redaction ist jedoch Best-effort. Bitte das Bundle vor dem Hochladen kurz überfliegen.
