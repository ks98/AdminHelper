Betrieb & Konfiguration

Alltagsaufgaben: Datenverzeichnis verstehen, Backups anlegen, Logs finden, Konfiguration anpassen, Server neu starten.

Datenverzeichnis & Postgres-Volume

Zustandsbehaftete Daten verteilen sich auf zwei Stellen: DATA_DIR für File-basierte Artefakte (Bootstrap-Token, PKI, Zertifikate) und das postgres-data-Volume für die eigentlichen Datenbanken.

PfadInhalt
postgres-data (Docker-Volume)PostgreSQL-Cluster mit zwei DBs: adminhelper (Benutzer, Verbindungen, Server, API-Keys, Hooks) und adminhelper_monitor (Checks, Alerts, Templates, Agent-Keys).
data/.bootstrap_tokenEinmaliger Setup-Token-Hash für den ersten Admin (siehe Erstanmeldung in der Installations-Doku).
data/.secret_keyAuto-generierter JWT-Key (sofern SECRET_KEY nicht gesetzt).
frp-config (Docker-Volume)Nur die generierte frps.toml (kein Cert-Material mehr). Die FRP-TLS-Materialien liefert der ca-issuer (Volume frps-certs, wiederherstellbar). Die alte server-eigene FRP-Master-PKI (Volume frp-pki) ist entfernt — siehe ca-pki unten.
ca-pki (Docker-Volume)PKI-Hierarchie des ca-issuer: Root (passphrase-verschlüsselt) + Intermediates (tunnel/access/internal). Das zentrale Backup-Ziel der internen CA — Verlust bedeutet, dass alle Clients neu enrollen müssen. Issuer-privat, nie ins Gateway/frps gemountet.
gateway-certs (Docker-Volume)TLS-Material des Gateways (access-signiertes Leaf + Key + CA-Trust-Bundle), vom ca-issuer bereitgestellt. Wiederherstellbar — der Issuer provisioniert es neu.
victoria-data (Docker-Volume)VictoriaMetrics-Speicher – Retention 90 Tage (eigenes Volume, nicht Teil dieses Datenverzeichnisses).

Backup

scripts/backup.sh sichert den kompletten Stand inklusive des CA-Kronjuwels in ein Tarball — bei laufendem Stack:

bash
./scripts/backup.sh                 # -> ./backups/adminhelper-backup-<ts>.tar.gz
./scripts/backup.sh --with-victoria # zusätzlich die (große) Metrik-Historie

Enthalten: ca-pki (Root + Intermediates), pg_dump beider DBs, monitoring-data, optional victoria-data und die .env. Nicht enthalten — und auch nicht nötig — sind gateway-certs/frps-certs: der ca-issuer regeneriert sie beim Start aus ca-pki.

CA_ROOT_PASSPHRASE getrennt aufbewahren

Die Passphrase wird bewusst nicht ins Backup geschrieben — sie verschlüsselt den mit-gesicherten Root-Key, beides zusammen wäre wertlos (ADR 0001 §5/D7). Im Passwortmanager getrennt halten. Der restaurierte Stack läuft auch ohne sie; gebraucht wird sie nur zum Rotieren der Intermediates.

Für automatische tägliche DB-Backups: den auskommentierten pg-backup-Service in docker-compose.yml einkommentieren (nur DBs, Retention 7 Tage) — ersetzt nicht das vollständige backup.sh inkl. CA.

Restore

bash
./scripts/restore.sh ./backups/adminhelper-backup-<ts>.tar.gz

Stoppt den Stack, stellt Volumes + DBs wieder her und fährt hoch. Die restaurierte Root ist identisch — bereits enrollte Agenten bleiben vertraut, ohne Re-Enrollment. Die .env aus dem Backup wird als .env.restored abgelegt (überschreibt eine vorhandene .env nicht); auf einem frischen Host die getrennt aufbewahrte CA_ROOT_PASSPHRASE ergänzen und nach .env übernehmen.

Logs

Die Container schreiben Plain-Text-Zeilen mit Zeitstempel auf stderr (stdout bleibt für Subprozess-IPC frei); docker compose logs -f <service> ist der schnellste Weg. Format: 2026-06-15 11:25:53 INFO adminhelper.auth … (Datum, Uhrzeit, Level, Logger, Meldung).

bash
docker compose logs -f server            # FastAPI + Scheduler
docker compose logs -f monitoring        # Check-Ausführungen
docker compose logs -f frps              # FRP-Server
docker compose logs -f --tail=200        # alle Services, letzte 200 Zeilen

Der Server-Container nutzt benannte Logger mit Präfix adminhelper.*; einige Module loggen unter ihrem Modulpfad app.*. Interessante Log-Namen:

  • adminhelper.auth / adminhelper.auth_router – Logins, Refresh-Rotationen, Reuse-Detection
  • adminhelper.rate_limit – Blockierte Requests
  • adminhelper.provisioning – Server-Provisionierung
  • adminhelper.servers / adminhelper.ansible – Server- und Ansible-Aktionen
  • app.modules.frp.pki / app.modules.frp.docker_manager – PKI-Aktionen und FRP-Container
  • app.modules.hooks.scheduler – Hook-Ausführungen inkl. Subprozess-Exitcodes

Der Monitoring-Container loggt unter dem Präfix monitor.* (z. B. monitor.engine, monitor.scheduler, monitor.alerter, monitor.victoria). Beide Container schreiben Plain-Text-Zeilen nach stderr, kein JSON.

Log-Level: Server und Monitoring lesen die Ausführlichkeit aus der Env-Variable LOG_LEVEL (DEBUG/INFO/WARNING/ERROR, Default INFO). Für eine tiefere Diagnose LOG_LEVEL=DEBUG in der .env setzen und docker compose up -d.

Rotation: Die Container-Logs sind über den Docker-json-file-Treiber begrenzt (max-size: 10m, max-file: 5 → max. 50 MB pro Service), damit sie nicht unbegrenzt wachsen. Konfiguriert über den x-logging-Anker in der docker-compose.yml.

Audit-Log

Sicherheits- und änderungsrelevante Aktionen werden in einem append-only Audit-Trail protokolliert — mit Actor (Benutzer oder API-Key), Quell-IP und Zeitstempel. Erfasst werden u. a. das Anlegen/Ändern/Löschen und die Nutzung von Verbindungen, Änderungen an Servern/Benutzern/API-Keys/FRP/Ansible sowie Logins (Erfolg/Fehlschlag), Logout und der Erst-Admin-Bootstrap.

Einsehbar im Web-UI unter Audit-Log (nur Admins; filterbar nach Actor, Objekt und Aktion) oder per API: GET /api/audit (paginiert; Query-Parameter action, actor_type, object_type, object_id, status, q).

Aufbewahrung

Ein täglicher Job löscht Einträge, die älter als AUDIT_RETENTION_DAYS (Default 365) sind — der einzige Löschpfad der ansonsten append-only Tabelle. 0 behält alles dauerhaft.

Local-Mode-Lücke

Verbindungen, die im Desktop-Client ohne Server (Local-Mode) genutzt werden, erreichen den Server nicht und erscheinen daher nicht im Audit-Trail.

Konfiguration ändern

Alle Config-Werte kommen aus Environment-Variablen. Änderungen erfordern ein docker compose up -d, damit die Container neu erzeugt werden:

bash
# .env anpassen, danach:
docker compose up -d server

Multi-Worker-Betrieb

Standardmäßig läuft ein einzelner Uvicorn-Worker. Für höhere Last:

  1. REDIS_URL setzen (sonst wird das In-Memory-Rate-Limit inkonsistent)
  2. Im docker-compose.override.yml command überschreiben: uvicorn app.main:app --host 0.0.0.0 --port 8080 --workers 4 (plain-HTTP intern; TLS terminiert das Gateway)

Zeitzone

TZ wird an alle Container weitergegeben. Voreinstellung Europe/Berlin. Das beeinflusst Audit-Log-Zeitstempel und Cron-ähnliche Hook-Jobs.

Zertifikate erneuern

Das TLS-Zertifikat des Gateways stellt der ca-issuer aus (access-signiert, 365 Tage). Es wird beim ersten Start ins gateway-certs-Volume provisioniert; ein Redeploy/Neustart des ca-issuer erneuert es bei Bedarf. Die FRP-Client-Zertifikate laufen nach 12 Monaten ab – siehe FRP-Tunnel für Rotation.

mTLS scharfschalten (CERT_REQUIRED)

Die Datenebene (:443) läuft seit 0.29.0 per Default scharf (MTLS_ENFORCE=true): das Gateway verlangt am TLS-Handshake ein gültiges Client-Zertifikat (CERT_REQUIRED) und der Server weist geschützte Routen ohne gültigen Scope mit 403 ab. Ein frischer Install ist via scripts/install.sh sofort enforced nutzbar (es besorgt dem Erst-Admin sein Cert out-of-band). Permissiv (MTLS_ENFORCE=false) lässt certlose Clients durch und loggt nur — nötig nur für einen manuellen Erst-Bootstrap. Beides steuert eine Variable: MTLS_ENFORCE.

Erst alle Clients enrollen, dann umlegen

Schalte erst scharf, wenn jeder Client – Agent, Desktop und dein eigener Browser – ein Zertifikat hat, sonst sperrst du dich selbst aus. Agent und Desktop enrollen automatisch; den Browser bereitest du über den Desktop-P12-Export vor (Benutzer & Zugriff → Browser-Zugriff). Vorher auf echter Hardware testen: Windows-Desktop-Enrollment und Browser-Import.

Scharfschalten:

bash
# in der .env setzen:
MTLS_ENFORCE=true

# Gateway + Server neu starten (lesen die Variable beim Start):
docker compose up -d gateway server

Der Gateway-Log bestätigt den Modus beim Start: mTLS ENFORCED (CERT_REQUIRED) bzw. mTLS permissive.

Rollback (jederzeit, verlustfrei): MTLS_ENFORCE=false setzen und docker compose up -d gateway server – die Datenebene ist sofort wieder permissiv.

Neue Clients nach dem Scharfschalten

Die Enrollment-Plane (:8444) bleibt immer certless und token-gegated – neue Agenten und Desktops holen ihr erstes Zertifikat dort, unabhängig vom Enforcement. Neuer Nutzer: ein Admin mintet ihm ein Enrollment-Token (POST /api/enrollment/token/for) und reicht es out-of-band weiter; im Desktop-Client wählt der Neue „Erstes Mal? Gerät mit Token einrichten", gibt Server-URL + Token ein und enrollt – ohne vorigen Login. Danach meldet er sich normal an (jetzt mit Zertifikat). Ein permissives Fenster ist dafür nicht mehr nötig (entkoppelte Enrollment-Tür, ADR 0003).

Nur der allererste Admin (es gibt noch keinen Admin, der ein Token ausstellen könnte) ist der Sonderfall — am einfachsten über scripts/install.sh, das den Admin + einen Erst-Token out-of-band anlegt und dann scharfschaltet. Manuell ohne Script: ein einmaliges permissives Setup-Fenster (MTLS_ENFORCE=false, ersten Admin + Desktop enrollen, wieder true).
