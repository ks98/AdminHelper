Betrieb & Konfiguration

Alltagsaufgaben: Datenverzeichnis verstehen, Backups anlegen, Logs finden, Konfiguration anpassen, Server neu starten.

Datenverzeichnis & Postgres-Volume

Zustandsbehaftete Daten verteilen sich auf zwei Stellen: DATA_DIR für File-basierte Artefakte (Bootstrap-Token, PKI, Zertifikate) und das postgres-data-Volume für die eigentlichen Datenbanken.

PfadInhalt
postgres-data (Docker-Volume)PostgreSQL-Cluster mit zwei DBs: adminhelper (Benutzer, Verbindungen, Server, API-Keys, Hooks) und adminhelper_monitor (Checks, Alerts, Templates, Agent-Keys).
data/.bootstrap_tokenEinmaliger Setup-Token-Hash für den ersten Admin (siehe Erstanmeldung in der Installations-Doku).
data/.secret_keyAuto-generierter JWT-Key (sofern SECRET_KEY nicht gesetzt).
frp-pki (Docker-Volume)FRP-Master-PKI: Root-CA inkl. Private-Key (10 J.) + alle Client-Keys. Das wichtigste Backup-Ziel — Verlust bedeutet Neuausstellung der gesamten mTLS-Trust-Domain.
frp-config (Docker-Volume)Generierte frps.toml + die publizierte Cert-Teilmenge, die frps liest (ca.crt, frps.crt, frps.key — kein CA-Private-Key, keine Client-Keys).
data/certs/Self-Signed-TLS für den FastAPI-Port 8443.
victoria-data (Docker-Volume)VictoriaMetrics-Speicher – Retention 90 Tage (eigenes Volume, nicht Teil dieses Datenverzeichnisses).

Backup

Zwei Backup-Pfade — beide laufen kombinieren für ein vollständiges Restore:

1. PostgreSQL-Dump (DBs)

bash
# Manueller Dump beider DBs (lokal, ohne Stop):
docker compose exec postgres pg_dump -U adminhelper -d adminhelper -Fc -f /tmp/adminhelper.dump
docker compose exec postgres pg_dump -U adminhelper -d adminhelper_monitor -Fc -f /tmp/adminhelper_monitor.dump
docker compose cp postgres:/tmp/adminhelper.dump ./backups/
docker compose cp postgres:/tmp/adminhelper_monitor.dump ./backups/

# Oder via mitgeliefertem Script:
docker compose exec postgres /usr/local/bin/pg-backup.sh    # falls scripts/ gemountet

Für automatische tägliche Backups: den auskommentierten pg-backup-Service in docker-compose.yml einkommentieren — schreibt nach ./backups/, Retention 7 Tage.

2. File-Daten (PKI, Bootstrap-Token, Zertifikate)

bash
tar czf adminhelper-files-$(date +%F).tar.gz data/

Restore

bash
# DB zurueckspielen:
docker compose cp ./backups/adminhelper-2026-05-02.dump postgres:/tmp/restore.dump
docker compose exec postgres pg_restore -U adminhelper -d adminhelper --clean --if-exists --no-owner /tmp/restore.dump

# Files:
tar xzf adminhelper-files-2026-05-02.tar.gz

Logs

Die Container schreiben auf stdout; docker compose logs -f <service> ist der schnellste Weg.

bash
docker compose logs -f server            # FastAPI + Scheduler
docker compose logs -f monitoring        # Check-Ausführungen
docker compose logs -f frps              # FRP-Server
docker compose logs -f --tail=200        # alle Services, letzte 200 Zeilen

Der Server-Container nutzt benannte Logger mit Präfix adminhelper.*; einige Module loggen unter ihrem Modulpfad app.*. Interessante Log-Namen:

  • adminhelper.auth / adminhelper.auth_router – Logins, Refresh-Rotationen, Reuse-Detection
  • adminhelper.rate_limit – Blockierte Requests
  • adminhelper.provisioning – Server-Provisionierung
  • adminhelper.servers / adminhelper.ansible – Server- und Ansible-Aktionen
  • app.modules.frp.pki / app.modules.frp.docker_manager – PKI-Aktionen und FRP-Container
  • app.modules.hooks.scheduler – Hook-Ausführungen inkl. Subprozess-Exitcodes

Der Monitoring-Container loggt unter dem Präfix monitor.* (z. B. monitor.engine, monitor.scheduler, monitor.alerter, monitor.victoria). Beide Container schreiben Plain-Text-Zeilen nach stdout, kein JSON.

Konfiguration ändern

Alle Config-Werte kommen aus Environment-Variablen. Änderungen erfordern ein docker compose up -d, damit die Container neu erzeugt werden:

bash
# .env anpassen, danach:
docker compose up -d server

Multi-Worker-Betrieb

Standardmäßig läuft ein einzelner Uvicorn-Worker. Für höhere Last:

  1. REDIS_URL setzen (sonst wird das In-Memory-Rate-Limit inkonsistent)
  2. Im docker-compose.override.yml command überschreiben: uvicorn app.main:app --host 0.0.0.0 --port 8443 --workers 4 --ssl-keyfile ... --ssl-certfile ...

Zeitzone

TZ wird an alle Container weitergegeben. Voreinstellung Europe/Berlin. Das beeinflusst Audit-Log-Zeitstempel und Cron-ähnliche Hook-Jobs.

Zertifikate erneuern

Die internen TLS-Zertifikate des FastAPI-Ports haben 10 Jahre Gültigkeit. Die FRP-Client-Zertifikate laufen nach 12 Monaten ab – siehe FRP-Tunnel für Rotation.
